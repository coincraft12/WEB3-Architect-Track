# 14강. 권한 통제 취약점 유형: 어떤 구조가 어떻게 문제가 되는가

## 1) 이 강의의 한 문장
**권한 통제 취약점은 단일 관리자 집중, 무한 Approve, 업그레이드 권한 집중, 역할 미분리의 네 가지 핵심 유형으로 나뉘며, 각 유형이 어떤 공격 경로 또는 운영 위험으로 이어지는지를 이해해야 통제안을 설계할 수 있다.**

## 2) 강의 목표 (수강생이 끝나고 할 수 있어야 하는 것)
이 강의를 마치면 수강생은:

1. 권한 통제 취약점의 4가지 핵심 유형을 정의할 수 있다.
2. 각 유형이 실제 서비스에서 어떻게 나타나는지 예시를 들어 설명할 수 있다.
3. 각 유형이 어떤 리스크(공격 경로, 운영 위험)로 이어지는지 설명할 수 있다.
4. 서비스를 분석할 때 해당 유형이 존재하는지 확인하는 방법을 말할 수 있다.

## 3) Module 2의 시작: 왜 권한 통제 취약점부터인가

Module 1(ch9~ch13)에서 서비스 구조를 3축으로 분석했다.
그 중 권한·책임 축(ch12)에서 도출된 취약점들을 Module 2에서 깊이 다룬다.

**왜 권한 통제가 핵심인가:**
```
WEB3 서비스의 자산은 컨트랙트에 보관된다.
컨트랙트를 통제하는 것은 권한이다.
권한이 잘못 설계되면 → 자산 전체가 위험해진다.

온체인에서 "신뢰"는 코드로 구현된다.
코드의 신뢰는 권한 구조로 보장된다.
```

## 4) 유형 1: 단일 관리자 집중 (Single Admin Concentration)

### 정의
서비스의 핵심 권한이 단일 주소(EOA 또는 소수의 Multisig)에 집중되어 있어, 그 주소가 침해되면 서비스 전체가 위험에 노출되는 구조.

### 나타나는 방식

**케이스 A: 단일 EOA Admin**
```solidity
address public owner;

modifier onlyOwner() {
    require(msg.sender == owner, "Not owner");
    _;
}

function withdraw(address to, uint256 amount) external onlyOwner {
    // Admin이 모든 자산을 임의 인출 가능
}

function pause() external onlyOwner { ... }
function upgrade() external onlyOwner { ... }
```

**케이스 B: 1-of-N Multisig (사실상 단일)**
```
3명의 서명자 중 1명의 서명으로 실행 가능한 Multisig.
서명 임계값이 낮아 단일 서명자 탈취로 모든 권한 행사 가능.
```

**케이스 C: 역할은 분리되어 있지만 Admin이 모두 부여**
```
Admin이 Operator, Risk Manager, Fee Collector 역할을 모두 스스로에게 부여.
역할 분리처럼 보이지만 실질적으로 단일 통제.
```

### 발생 리스크

| 리스크 | 내용 |
|---|---|
| 키 탈취 | Admin 개인키 탈취 시 서비스 전체 통제권 즉시 이전 |
| 내부자 위협 | Admin 역할 보유자가 악의적으로 행동할 경우 견제 불가 |
| 단일 실패점 | Admin 키 분실 시 핵심 기능 영구 불가 (패닉 상황) |
| 투명성 부재 | Admin 행동을 사전에 알 수 없어 사용자 대응 불가 |

### 확인 방법
- 컨트랙트에서 `onlyOwner` 함수 목록 확인
- `owner` 주소가 EOA인지 Multisig인지 확인 (Etherscan 등)
- Multisig라면 임계값(M-of-N) 확인
- 감사 보고서에서 Admin 권한 집중 언급 여부 확인

## 5) 유형 2: 무한 Approve (Unlimited Approval)

### 정의
사용자가 컨트랙트에 토큰 사용 한도를 `type(uint256).max` (무한)으로 승인하는 구조. 해당 컨트랙트가 침해되면 사용자의 지갑에 있는 해당 토큰 전체를 인출할 수 있다.

### 나타나는 방식

**온보딩 시 무한 Approve 요청:**
```
사용자 지갑에서 Approve 트랜잭션:
- 대상 컨트랙트: 0xABCD...
- 승인 금액: 115792089237316195423570985008687907853269984665640564039457584007913129639935
  (= type(uint256).max)
```

**프론트엔드에서의 표시:**
```
"Approve USDC to [Service Name]"
→ 실제로는 무한 금액 승인이지만 UI에서 명시하지 않는 경우
```

**취약점 경로:**
```
1. 사용자가 A 서비스에 무한 Approve
2. A 서비스 컨트랙트가 해킹됨
3. 공격자가 A 서비스를 통해 사용자 지갑의 토큰 전액 인출
   → 사용자가 A 서비스에서 자산을 모두 출금했어도 위험 지속
```

### 발생 리스크

| 리스크 | 내용 |
|---|---|
| 잔여 Approve 위험 | 서비스 사용 종료 후에도 Approve가 남아 있어 지속 위험 |
| 연쇄 피해 | 서비스 해킹 → Approve한 모든 사용자 피해 |
| 범위 초과 피해 | 서비스에 예치한 금액보다 지갑 전체 잔액이 위험 |
| 사용자 인지 부재 | UI에서 무한 Approve 여부를 명시하지 않는 경우 多 |

### 확인 방법
- 서비스의 Approve 요청 금액 확인 (Etherscan Token Approvals 탭)
- 프론트엔드 코드에서 `approve(address, type(uint256).max)` 패턴 확인
- 서비스 문서에서 Approve 정책 명시 여부 확인
- Revoke 기능 제공 여부 확인

## 6) 유형 3: 업그레이드 권한 집중 (Upgrade Authority Concentration)

### 정의
Proxy 구조에서 구현 컨트랙트를 교체할 수 있는 권한(Upgrader)이 적절한 통제 없이 단일 주체에 집중되어 있는 구조. 업그레이드 권한은 "컨트랙트의 모든 로직을 바꿀 수 있는 권한"이므로 가장 강력한 권한 중 하나다.

### 나타나는 방식

**케이스 A: Admin과 Upgrader가 동일**
```
Admin (3-of-5 Multisig):
  - pause() / unpause()
  - setOracle()
  - upgrade() ← 컨트랙트 로직 전체 교체 가능
```

**케이스 B: Upgrader가 별도이지만 단일 EOA**
```
ProxyAdmin 컨트랙트의 owner = 0xAAAA (단일 EOA)
→ 이 주소가 탈취되면 컨트랙트 전체 로직 교체 가능
```

**케이스 C: Timelock 없는 즉시 업그레이드**
```
upgrade() 실행 → 즉시 구현 주소 변경
→ 사용자가 대응할 시간 없음
```

**업그레이드 권한 남용 시나리오:**
```
1. Upgrader 주소가 침해됨
2. 새 구현 컨트랙트 배포 (악성 로직 포함)
3. upgrade() 실행 → Proxy가 악성 컨트랙트를 가리킴
4. 악성 컨트랙트의 withdraw() 실행 → 모든 자산 인출
   → 기존 사용자 자산이 새 로직에 의해 이동됨
```

### 발생 리스크

| 리스크 | 내용 |
|---|---|
| 로직 교체 | 자산 인출 로직을 새로 삽입하여 즉시 실행 가능 |
| 투명성 부재 | Timelock 없으면 사용자가 변경을 사전에 알 수 없음 |
| 권한 집중 효과 | 업그레이드 = 모든 권한을 우회하는 권한 |
| 거버넌스 우회 | 거버넌스 결정 없이 핵심 파라미터 변경 가능 |

### 확인 방법
- Proxy 구조 여부 확인 (EIP-1967 슬롯, OpenZeppelin Proxy 패턴)
- ProxyAdmin 또는 업그레이드 권한 주소 확인
- 업그레이드 권한 주소가 EOA인지 Multisig인지 확인
- 업그레이드 시 Timelock 적용 여부 확인
- 과거 업그레이드 이력 확인 (정기적으로 발생하는지)

## 7) 유형 4: 역할 미분리 (Role Conflation)

### 정의
서비스 내 다양한 기능에 대한 접근 권한이 하나의 역할로 통합되어 있어, 특정 기능을 위해 부여된 권한이 의도치 않게 다른 핵심 기능에도 접근할 수 있는 구조.

### 나타나는 방식

**케이스 A: onlyOwner 남용**
```solidity
function setFee(uint256 newFee) external onlyOwner { ... }
function pause() external onlyOwner { ... }
function withdraw(address to, uint256 amount) external onlyOwner { ... }
function upgrade() external onlyOwner { ... }
```
→ 수수료 설정 권한자가 출금, 정지, 업그레이드를 모두 할 수 있음

**케이스 B: RBAC 형식만 존재**
```
OPERATOR_ROLE: 파라미터 조정 + 수수료 수취 + 긴급 정지
→ 세 가지 성격이 다른 권한이 하나의 역할에 묶여 있음
```

**케이스 C: 역할 분리는 되어 있지만 위임 구조가 단일**
```
Admin → Operator에게 grantRole()
Admin → Risk Manager에게 grantRole()
Admin → Fee Collector에게 grantRole()
→ 역할은 분리되어 있지만 Admin이 모두 통제
   Admin 탈취 = 전체 역할 구조 무력화
```

**케이스 D: 운영자에게 과도한 권한**
```
Operator:
  - 파라미터 조정 (정상)
  - 사용자 자산 직접 이동 (비정상: Admin 수준 권한)
```

### 발생 리스크

| 리스크 | 내용 |
|---|---|
| 권한 남용 | 낮은 권한의 역할이 높은 영향의 행동을 실행 가능 |
| 공격 표면 확대 | 탈취 대상이 늘어남 (Admin 외 Operator도 위험) |
| 감사 복잡성 | 어떤 역할이 무엇을 할 수 있는지 파악이 어려워짐 |
| 책임 경계 불명확 | 사고 발생 시 어떤 역할이 문제였는지 추적 어려움 |

### 확인 방법
- 컨트랙트 코드에서 `onlyOwner`, `onlyRole` 함수 목록 추출
- 각 역할이 접근 가능한 함수 목록을 매핑 (역할 × 함수 행렬)
- 단일 역할에 과도하게 많은 함수가 연결되어 있는지 확인
- 역할별 기능의 성격(일상 운영 / 비상 통제 / 자산 이동)이 혼재하는지 확인

## 8) 4가지 유형 요약 비교

| 유형 | 핵심 문제 | 주요 리스크 | 탐지 포인트 |
|---|---|---|---|
| 단일 관리자 집중 | 권한이 한 곳에 집중 | 키 탈취 = 서비스 전체 위험 | Admin 주소 유형, Multisig 임계값 |
| 무한 Approve | 사용자 지갑 전체 노출 | 서비스 해킹 = 사용자 잔액 전체 위험 | Approve 금액, Revoke 기능 |
| 업그레이드 권한 집중 | 로직 교체 권한 집중 | 악성 로직 삽입 → 자산 전체 이동 | Upgrader 주소 유형, Timelock 여부 |
| 역할 미분리 | 역할별 권한 범위 과도 | 낮은 역할 탈취로 높은 피해 | 역할 × 함수 매핑, onlyOwner 목록 |

## 9) 서비스 분석 시 유형 확인 체크리스트

```
□ 단일 관리자 집중
  □ Admin이 EOA인가? Multisig인가?
  □ Multisig라면 임계값이 적절한가? (최소 M > N/2 권장)
  □ Admin 교체 이력이 있는가?

□ 무한 Approve
  □ 서비스 진입 시 Approve 요청 금액을 확인했는가?
  □ 특정 금액 Approve 옵션이 있는가?
  □ Revoke 기능 또는 안내가 있는가?

□ 업그레이드 권한 집중
  □ Proxy 구조 여부를 확인했는가?
  □ Upgrader가 누구인가? Admin과 동일한가?
  □ 업그레이드 시 Timelock이 있는가?
  □ 과거 업그레이드 이력에서 이상 패턴이 있는가?

□ 역할 미분리
  □ 역할별 함수 목록을 추출했는가?
  □ 단일 역할에 성격이 다른 권한이 묶여 있는가?
  □ 운영자 역할이 자산 이동 권한을 갖는가?
```

## 용어 보충 (Quick Reference)
- **단일 관리자 집중 (Single Admin Concentration)**: 서비스의 핵심 권한이 하나의 주소에 집중된 구조. 단일 실패점(Single Point of Failure)을 형성한다.
- **무한 Approve (Unlimited Approval)**: `type(uint256).max` 금액으로 토큰 사용 권한을 부여하는 것. 컨트랙트가 침해되면 사용자 지갑의 해당 토큰 전체가 위험해진다.
- **업그레이드 권한 집중 (Upgrade Authority Concentration)**: Proxy 패턴에서 구현 컨트랙트를 교체할 수 있는 권한이 적절한 통제 없이 집중된 구조.
- **역할 미분리 (Role Conflation)**: 서로 다른 성격의 권한이 하나의 역할로 묶여 있어 의도하지 않은 권한 남용이 가능한 구조.
- **공격 표면 (Attack Surface)**: 공격자가 시스템을 침해하기 위해 접근할 수 있는 모든 경로와 취약점의 집합.

## 헷갈리는 개념 보충
- 무한 Approve는 서비스 자체의 취약점이 아니라 Approve를 요청하는 방식의 문제다. 서비스 컨트랙트가 안전해도, 미래에 해킹되면 Approve가 피해 경로가 된다.
- 업그레이드 권한은 그 자체로 취약점이 아니다. 권한이 적절히 통제(Multisig + Timelock)되면 안전하다. 통제 부재가 문제다.
- 역할 미분리는 형식적인 RBAC 구조가 있어도 발생한다. 역할의 이름이 아니라 역할이 접근할 수 있는 함수의 성격을 분석해야 한다.
