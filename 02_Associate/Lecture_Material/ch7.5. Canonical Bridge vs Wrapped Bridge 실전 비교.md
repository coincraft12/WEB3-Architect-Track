# 7.5강. Canonical Bridge vs Wrapped Bridge 실전 비교

## 1) 이 강의의 한 문장
**브릿지 분석의 핵심은 "자산이 어디에 잠기고, 누가 어떤 방식으로 정합성을 보장하는가"를 구분하는 것이며, Canonical Bridge와 Wrapped Bridge의 차이를 정확히 읽는 데서 리스크 판정이 시작된다.**

## 2) 강의 목표 (수강생이 끝나고 할 수 있어야 하는 것)
이 강의를 마치면 수강생은:

1. Canonical Bridge와 Wrapped Bridge의 구조 차이를 설명할 수 있다.
2. 동일 자산 심볼이라도 신뢰 가정이 다를 수 있음을 판정할 수 있다.
3. Challenge Period, Finality, Liquidity Route를 구분하여 말할 수 있다.
4. 서비스가 어떤 브릿지 경로를 사용하는지 온체인 근거로 확인할 수 있다.
5. 브릿지 선택에 따른 리스크를 FFRS 항목으로 정리할 수 있다.

---

## 3) 왜 이 구분이 중요한가

실무에서 가장 자주 발생하는 오해:

- "ETH면 다 같은 ETH다."
- "출금이 빠르면 더 좋은 브릿지다."
- "TVL이 크니 안전하다."

이 셋은 모두 불완전한 판단이다.
같은 자산 심볼이라도 브릿지 구조가 다르면 보안 가정이 다르다.

핵심 문장:
> **자산 심볼이 아니라, 정합성을 보장하는 메커니즘을 봐야 한다.**

---

## 4) 두 구조의 정의

### A. Canonical Bridge

- 특정 체인/롤업의 "공식 경로"로 간주되는 브릿지
- 보통 해당 체인의 정식 검증 메커니즘(예: 롤업 증명/이의 제기)에 연동
- 빠른 UX보다 정합성 보장을 우선하는 경우가 많음

예시 관점:
- Ethereum <-> Optimism 공식 브릿지
- Ethereum <-> Arbitrum 공식 브릿지

### B. Wrapped Bridge (3rd-party Bridge 포함)

- 별도 운영/검증 네트워크가 원본 자산을 락하고 표현 자산을 발행
- 빠른 이동성과 멀티체인 확장성에 강점
- 브릿지 운영 주체/검증자/유동성 풀의 신뢰에 의존

예시 관점:
- 외부 브릿지가 발행한 wrapped ETH 계열 자산
- 특정 브릿지 전용 표기 토큰

---

## 5) 구조 비교표 (실전 판독 기준)

| 항목 | Canonical Bridge | Wrapped Bridge |
|---|---|---|
| 신뢰 가정 | 체인/롤업 공식 검증 메커니즘 | 브릿지 운영 주체/검증자/유동성 네트워크 |
| 정산 속도 | 느릴 수 있음 (Challenge Period 등) | 빠른 경우 많음 |
| 자산 표현 | 체인 공식 표현 자산 맥락 | 브릿지별 표현 토큰 맥락 |
| 주요 리스크 | 지연/운영 복잡성, 체인 자체 이슈 | 검증자 탈취, 과발행, 운영자 리스크 |
| 분석 포인트 | 최종성, 이의 구조, 공식 경로 여부 | 담보 증명, 발행-락 정합성, 운영 권한 집중 |

---

## 6) Challenge Period, Timelock, Finality 구분

세 개념은 비슷해 보이지만 목적이 다르다.

- **Challenge Period**: 이미 제출된 상태 전이에 대해 이의를 제기하는 검증 창
- **Timelock**: 민감 권한 실행을 지연시켜 사전 검토 시간을 주는 통제 장치
- **Finality**: 거래가 되돌릴 수 없다고 보는 합의 확정 상태

정리:
> Challenge Period는 검증 절차, Timelock은 권한 통제 절차다.

---

## 7) 실전 판독 절차 (Explorer 기반)

### Step 1: 자산 출처 확인

1. 사용 중인 토큰 컨트랙트 주소 확인
2. 토큰 발행자/브릿지 연계 문서 확인
3. 공식 브릿지 경로인지, 외부 브릿지 발행 토큰인지 분류

### Step 2: 브릿지 컨트랙트/이벤트 확인

1. 락/민트/번/언락 이벤트 확인
2. 발행량과 락된 원본 자산의 정합성 확인
3. 관리자 권한(업그레이드/긴급정지/동결) 확인

### Step 3: 출금 경로 확인

1. 공식 경로의 정산 시간(지연) 확인
2. 빠른 출금 경로가 있다면 유동성 제공자 의존 여부 확인
3. 수수료와 슬리피지, 실패 시 롤백 조건 확인

## 8) 비교 시나리오 (실습)

가정:
- 사용자 A는 Ethereum에서 L2로 100 ETH 이동
- 경로 1: Canonical Bridge
- 경로 2: Wrapped Bridge 기반 Fast Route

### 경로 1: Canonical Bridge

- 장점: 공식 검증 메커니즘 기반 정합성
- 단점: 정산 대기 시간(Challenge Period)으로 UX 저하 가능

### 경로 2: Wrapped Bridge/Fast Route

- 장점: 빠른 정산, 즉시 사용성
- 단점: 브릿지 운영/유동성/검증자 신뢰 리스크 추가

판정 규칙:
- 핵심 담보 이동 자산: 보수적으로 Canonical 우선 검토
- 소액/빈번 이동: Wrapped 경로의 운영 통제를 별도 점검

## 9) FFRS 입력 예시

| 흐름구간 | 리스크유형 | 심각도 | 개선방향 |
|---|---|---|---|
| Ethereum -> L2 입금 (Wrapped Route) | 브릿지 검증자/운영 주체 리스크 | High | 공식 경로 대안 제공, 경로별 리스크 고지 |
| L2 -> Ethereum 출금 (Fast Route) | 유동성 경로 고갈/지연 리스크 | Medium | 출금 모드 선택 UI, 예상 지연시간 명시 |
| 브릿지 토큰 보관 구간 | 발행-락 정합성 확인 부족 | High | 정합성 모니터링 대시보드, 이상치 알림 |

## 용어 보충 (Quick Reference)

- **Canonical Bridge**: 특정 체인 생태계에서 공식 경로로 간주되는 브릿지.
- **Wrapped Bridge**: 원본 자산 락과 표현 자산 발행을 외부 브릿지 계층이 담당하는 구조.
- **Challenge Period**: 상태 전이에 대한 이의 검증 창.
- **Fast Route**: 유동성 제공자를 활용해 체감 정산 시간을 줄이는 경로.
- **Finality**: 거래 확정 불가역성 상태.

## 헷갈리는 개념 보충

- Canonical이 항상 "절대 안전"을 뜻하지는 않는다. 신뢰 가정의 종류가 다를 뿐이다.
- Wrapped가 항상 "위험"한 것도 아니다. 운영 통제와 검증 투명성이 높으면 안정적으로 운영될 수 있다.
- 빠른 출금은 보통 별도 신뢰 가정을 추가한다. 속도와 신뢰는 교환 관계다.
