# 11강. 자산흐름 경로 분석: 진입에서 출구까지 자산이 어떻게 이동하는가

## 1) 이 강의의 한 문장
**자산흐름 분석은 서비스에 들어온 자산이 어떤 경로로 변환되고 이동하며 나가는지를 추적하는 작업이다. 진입점·내부 경로·출구를 완전히 그릴 수 있을 때, 숨겨진 리스크가 보이기 시작한다.**

## 2) 강의 목표 (수강생이 끝나고 할 수 있어야 하는 것)
이 강의를 마치면 수강생은:

1. 자산흐름 분석의 3요소(진입점 / 내부 경로 / 출구)를 설명할 수 있다.
2. 자산 상태 변환(코인 → 토큰 → LP 토큰 등)을 추적하는 방법을 말할 수 있다.
3. 텍스트 기반 자산흐름 다이어그램을 작성할 수 있다.
4. 숨겨진 자산 경로(백도어, 수수료 흐름, 외부 의존성)를 식별하는 방법을 말할 수 있다.
5. 자산흐름에서 드러나는 주요 취약점 유형을 말할 수 있다.

---

## 3) 왜 자산흐름을 추적해야 하는가

ch10(온보딩)에서 사용자가 서비스에 권한을 주고 자산을 넣었다.
이제 그 자산은 어디에 있는가?

대부분의 사용자는 이 질문에 정확히 답하지 못한다.
"내 잔액이 화면에 보이면 됐지."

하지만 그 잔액이 의미하는 것이:
- 컨트랙트 안에 실제로 잠겨 있는 자산인가?
- 다른 프로토콜에 재예치된 자산의 청구권인가?
- 브릿지를 통해 다른 체인에 있는 래핑 토큰인가?
- 이미 다른 곳에 담보로 잠긴 자산인가?

에 따라 리스크 구조가 완전히 달라진다.

핵심 문장:
> **자산흐름을 추적하지 않으면, "내 잔액이 있다"는 사실 말고는 아무것도 모르는 것이다.**

## 4) 자산흐름 분석의 3요소

### 1. 진입점 (Entry Point)

자산이 서비스 안으로 처음 들어오는 경계.

**유형:**
- 입금 주소로의 직접 전송
- 컨트랙트의 deposit() 함수 호출
- 스왑 후 풀로 유입
- 브릿지를 통한 다른 체인에서의 유입

**진입점 분석 시 확인 사항:**
- 진입점이 몇 개인가? (하나인가, 여러 경로가 있는가)
- 진입 가능한 자산 유형은 무엇인가? (네이티브 코인 / 특정 토큰 / 모든 토큰)
- 진입 시 검증 절차가 있는가? (화이트리스트 주소, KYC, 한도 등)

### 2. 내부 경로 (Internal Flow)

서비스 안에서 자산이 이동하고 변환되는 경로.

**세 가지 내부 흐름 유형:**

**A. 보관 (Storage)**
자산이 특정 컨트랙트에 잠겨 있는 상태.
```
사용자 예치 → Vault 컨트랙트 (자산 잠김)
```

**B. 변환 (Transformation)**
자산이 다른 형태로 바뀌는 과정.
```
ETH 예치 → stETH 발행 (이자 누적 형태로 변환)
USDC + ETH → LP 토큰 발행 (풀 지분으로 변환)
담보 예치 → 대출 가능 포인트 생성
```

**C. 라우팅 (Routing)**
여러 컨트랙트를 거쳐 이동하는 경로.
```
사용자 → Router → Pool A → Pool B → 사용자
(최적 경로로 여러 풀을 거치는 멀티홉 스왑)
```

### 3. 출구 (Exit Point)

자산이 서비스 밖으로 나가는 경계.

**유형:**
- withdraw() 함수 호출
- LP 토큰 반환 후 원금 회수
- 대출 상환 후 담보 반환
- 브릿지를 통한 다른 체인으로 이동

**출구 분석 시 확인 사항:**
- 출구가 언제나 열려 있는가? 조건이 있는가?
- 출금 지연이 있는가? (락업 기간, Timelock, 출금 큐)
- 출금 한도나 일일 제한이 있는가?
- 긴급 상황에서 출금 경로가 막히는 구조가 있는가?

## 5) 자산 상태 변환 추적

자산흐름에서 가장 놓치기 쉬운 것이 **자산 상태 변환**이다.

사용자가 "내 ETH"라고 생각하는 것이 실제로는 여러 단계를 거쳐 다른 형태가 되어 있을 수 있다.

**변환 체인 예시 (DeFi 레버리지 파밍):**
```
ETH (원본)
  ↓ Liquid Staking
stETH (이자 누적 증표)
  ↓ AMM에 유동성 공급
stETH-ETH LP 토큰 (풀 지분 증표)
  ↓ Lending에 담보로 예치
대출 가능 금액 생성
  ↓ 추가 ETH 대출
ETH (대출된 자산)
  ↓ 다시 Liquid Staking
stETH (반복)
```

이 경우:
- 원본 ETH가 어디에 있는가? → Liquid Staking 프로토콜
- stETH의 가치는 무엇에 의존하는가? → Liquid Staking 프로토콜의 신뢰
- LP 토큰의 가치는? → stETH 가격 + ETH 가격 + 풀 유동성
- 담보 가치 하락 시? → 청산 연쇄 (ch6 연결)

설계자 포인트:
- 변환이 많을수록 리스크가 **레이어링(Layering)**된다.
- 최종 자산의 가치는 가장 아래 레이어의 신뢰에 달려 있다.
- 변환 체인이 길수록 어느 한 지점이 무너졌을 때 전체가 연쇄적으로 영향받는다.

## 6) 자산흐름 다이어그램 작성 방법

복잡한 서비스를 분석할 때 텍스트 기반 다이어그램으로 흐름을 시각화한다.

**작성 원칙:**
1. 참여자(사용자, 컨트랙트, 외부 프로토콜)를 박스로 표시
2. 자산 이동 방향을 화살표로 표시
3. 화살표 위에 이동하는 자산 유형과 조건을 표기
4. 변환이 일어나는 지점을 명시

**AMM 예치 + Lending 담보 사용 흐름 예시:**
```
[사용자 지갑]
    │ ETH + USDC (approve + deposit)
    ▼
[AMM Pool 컨트랙트]
    │ LP 토큰 발행
    ▼
[사용자 지갑] LP 토큰 보유
    │ LP 토큰 (approve + deposit)
    ▼
[Lending 프로토콜]
    │ 대출 가능 금액 생성
    ▼
[사용자] USDC 대출 수령
    │
    ▼
[외부 사용 또는 재투자]

※ 역방향 (출금):
[사용자] USDC + 이자 상환
    ▼
[Lending 프로토콜] LP 토큰 반환
    ▼
[사용자 지갑] LP 토큰 회수
    ▼
[AMM Pool 컨트랙트] LP 소각 후 ETH + USDC 반환
    ▼
[사용자 지갑] 원금 ± 손익
```

## 7) 숨겨진 자산 경로 식별

공식 문서에 나오지 않는 자산 흐름이 있다.
이것이 구조 분석에서 가장 중요한 발견 중 하나다.

### A. 수수료 흐름

프로토콜이 수수료를 어디로 보내는지가 권한 구조와 연결된다.

```
스왑 수수료 → 프로토콜 재무(Treasury) 주소
                ↓
        Treasury 주소의 통제권은 누가 갖는가?
```

- Treasury가 단일 키 또는 소규모 Multisig라면 핵심 자금 흐름의 단일 실패점이다.
- 수수료 분배 비율 변경 권한이 누구에게 있는가? (ch3 RBAC 연결)

### B. 외부 프로토콜 의존성

서비스가 내부에서만 자산을 처리하지 않고 외부 프로토콜을 호출하는 경우.

```
사용자 → A 서비스 예치
            ↓
A 서비스 → B 프로토콜에 재예치 (사용자 모르게)
            ↓
C 프로토콜에 담보 → D 프로토콜에서 대출
```

- 사용자가 A 서비스에만 예치했다고 생각하지만, 실제 자산은 B, C, D 프로토콜에 분산되어 있다.
- B, C, D 중 하나가 해킹되면 A 서비스 전체가 영향받는다.

### C. 관리자 인출 경로

Admin이 사용자 자산에 직접 접근할 수 있는 숨겨진 함수.

```
withdraw(address to, uint256 amount) onlyOwner
```

- 문서에 언급되지 않을 수 있다.
- 컨트랙트 코드나 감사 보고서에서 확인해야 한다.
- 이 함수가 존재하더라도, **사용자 예치자산을 관리자 권한만으로 임의 인출할 수 있고(추가 승인·제약 없음), 그 통제가 충분히 제한되지 않은 경우** 실질적으로 Custodial 성격이 강해진다. (ch8 연결)

### D. 긴급 기능 (Emergency Function)

일부 프로토콜은 비상 시 자산을 강제 이동시키는 함수를 갖는다.

```
emergencyWithdraw() → 사용자 자산을 강제 반환
pause() → 일체의 출금 차단
```

- 긴급 기능 자체가 나쁜 것은 아니다. 누가 이 권한을 갖는지가 중요하다.
- pause() 권한이 악용되면 사용자 자산이 무기한 동결될 수 있다.

## 8) 자산흐름에서 드러나는 취약점 유형

**유형 1: 출구 없는 자산 (Asset Lockup)**
```
자산이 들어가는 경로는 있지만, 나오는 경로가 조건부이거나 막혀 있다.
예: "30일 락업", "DAO 투표로만 출금 가능", "긴급 정지 후 출금 불가"
```

**유형 2: 단방향 변환 (Irreversible Transformation)**
```
변환 후 원래 자산으로 돌아올 수 없는 경우.
예: 토큰이 소각되어 포인트로 전환 → 포인트를 다시 토큰으로 바꿀 수 없음
```

**유형 3: 외부 의존성 숨김 (Hidden External Dependency)**
```
서비스가 사용자 자산을 외부 프로토콜에 재예치하지만 이를 명시하지 않는 경우.
사용자는 리스크를 모른 채 간접적으로 외부 프로토콜 리스크를 지게 된다.
```

**유형 4: 수수료 흐름 집중 (Fee Flow Centralization)**
```
모든 수수료가 단일 주체에게 흐르고, 그 주체가 강한 통제를 받지 않는 경우.
```

**유형 5: 관리자 우선 인출 경로 (Admin Backdoor)**
```
Admin이 사용자 자산에 직접 접근할 수 있는 경로가 존재하는 경우.
```

**유형 6: 흐름 경로의 중앙화 지점 (Centralized Chokepoint)**
```
모든 자산이 단일 컨트랙트나 주소를 반드시 거쳐야 하는 구조.
그 지점이 공격받거나 멈추면 전체 흐름이 차단된다.
```

## 9) 자산흐름을 Architecture Read Sheet에 기록하는 방법

**섹션 3: 자산흐름 구조 작성 형식:**

```
[진입점]
- 진입 가능 자산: ETH, USDC, WBTC
- 진입 함수: deposit(address token, uint256 amount)
- 진입 조건: 없음 (누구나 가능)
- 진입 컨트랙트: 0x... (Proxy 여부: Yes/No)

[내부 경로]
- 단계 1: deposit → Vault 컨트랙트에 보관
- 단계 2: Vault → 외부 Yield 프로토콜에 재예치 (자동)
  → 외부 의존성: [프로토콜 이름, 주소, 감사 여부]
- 단계 3: 수익 발생 → Vault 잔액 증가
- 자산 변환: deposit 토큰 → Vault share 토큰 (증표)

[출구]
- 출금 함수: withdraw(uint256 shares)
- 출금 조건: 없음 / 락업 기간: 없음 / 출금 한도: 없음
- 긴급 출금 기능: 있음 (권한 보유자: 0x... Multisig)

[수수료 흐름]
- 수익의 10% → Treasury 주소 (0x...)
- Treasury 통제: 3-of-5 Multisig

[숨겨진 경로 확인]
- 관리자 직접 인출 함수: 없음 (감사 보고서 확인)
- 긴급 pause 기능: 있음 (권한: 0x... Multisig)

[자산흐름 취약점 메모]
- 외부 Yield 프로토콜 의존성: 해당 프로토콜 리스크가 전이됨
- pause 권한이 출금을 막을 수 있음 → 권한 보유자 신뢰 필요
```

## 10) 서비스 유형별 자산흐름 핵심 확인 포인트

ch5~ch8에서 배운 서비스 유형별로 흐름 분석의 핵심이 달라진다.

| 서비스 유형 | 핵심 내부 경로 | 주요 숨겨진 흐름 | 출구 취약점 |
|---|---|---|---|
| AMM | 풀 ↔ 스왑 경로 | 수수료 → Treasury | 유동성 고갈 시 인출 불가 |
| Lending | 담보 → 대출 → 청산 | 오라클 가격 경로 | 청산 후 담보 부족 |
| Bridge | Lock → 검증 → Mint | 검증자 서명 경로 | 원본 체인 자산 동결 |
| Custody | 입금 → 집금 → 보관 | 관리자 인출 경로 | pause로 출금 차단 |

## 용어 보충 (Quick Reference)
- **진입점 (Entry Point)**: 자산이 서비스 컨트랙트로 처음 들어오는 지점 또는 함수.
- **출구 (Exit Point)**: 자산이 서비스 컨트랙트에서 사용자에게 반환되는 지점 또는 함수.
- **자산 상태 변환 (Asset State Transformation)**: 자산이 다른 형태의 토큰으로 바뀌는 과정. 예: ETH → stETH → LP 토큰.
- **레이어링 (Layering)**: 자산 변환이 여러 단계로 쌓이는 것. 각 레이어가 리스크를 추가한다.
- **숨겨진 경로 (Hidden Flow)**: 공식 문서에 명시되지 않은 자산 이동 경로. 수수료 흐름, 관리자 인출, 외부 재예치 등.
- **Treasury**: 프로토콜의 수익이 모이는 재무 주소. 거버넌스 또는 팀이 관리한다.
- **락업 (Lockup)**: 일정 기간 동안 자산을 출금할 수 없도록 제한하는 구조.
- **Chokepoint**: 모든 자산이 반드시 통과해야 하는 단일 경로 지점.

## 헷갈리는 개념 보충
- 자산이 "서비스에 있다"는 것과 "컨트랙트에 잠겨 있다"는 것은 다를 수 있다. 서비스가 그 자산을 외부 프로토콜에 재예치했다면, 실제 자산은 외부 프로토콜에 있다.
- 출구가 존재한다고 해서 항상 열려 있는 것은 아니다. pause 기능, 출금 큐, 유동성 고갈 등으로 막힐 수 있다.
- 수수료 흐름은 자산 흐름의 일부다. 수수료가 어디로 가는지 모르면 전체 흐름을 분석한 것이 아니다.
