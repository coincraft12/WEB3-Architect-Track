# 3강. 역할 기반 권한 구조(RBAC): 누가 무엇을 할 수 있는가를 분리하는 설계

## 1) 이 강의의 한 문장
**RBAC(Role-Based Access Control)는 "누가 무엇을 할 수 있는가"를 역할 단위로 분리하는 구조다. onlyOwner 하나가 모든 권한을 독점하는 대신, 역할마다 다른 권한을 부여해 권한 범위를 최소화한다.**

## 2) 강의 목표 (수강생이 끝나고 할 수 있어야 하는 것)
이 강의를 마치면 수강생은:

1. onlyOwner 단일 권한 구조의 문제를 구조적으로 설명할 수 있다.
2. RBAC의 핵심 원리(역할 분리, 최소 권한)를 말할 수 있다.
3. WEB3 서비스에서 자주 등장하는 역할 유형(Admin/Operator/Pauser 등)을 구분할 수 있다.
4. 역할 부여·취소 구조가 왜 중요한지 설명할 수 있다.
5. DAO 거버넌스 기반 권한 구조가 RBAC와 어떻게 연결되는지 개요 수준에서 말할 수 있다.

---

## 3) 문제에서 시작: onlyOwner의 한계

WEB3 스마트컨트랙트에서 가장 기본적인 권한 구조는 `onlyOwner`다.

`onlyOwner`는 단순하다.
- Owner 주소가 있다.
- Owner만 특정 함수를 실행할 수 있다.
- Owner가 아니면 모두 차단된다.

초기 프로토타입이나 소규모 서비스에는 충분하다.
하지만 서비스가 커지면 문제가 드러난다.

**onlyOwner가 커버하는 권한 목록:**
- 컨트랙트 업그레이드
- 수수료 변경
- 긴급 일시정지
- 새 주소 화이트리스트 등록
- 자금 인출
- 파라미터 조정

이 모든 것이 **하나의 키**에 묶여 있다.

그 키를 가진 사람이 어떤 이유로든 탈취당하거나, 실수하거나, 배신하면
전부 한꺼번에 위험해진다.

더 심각한 문제가 있다.
업무상 "수수료 변경"만 할 수 있어야 하는 사람이
onlyOwner 키를 받는 순간 "자금 인출"도 할 수 있게 된다.

권한이 필요 이상으로 넓어지는 순간, 그 권한은 위험이 된다.

## 4) RBAC란 무엇인가

**RBAC(Role-Based Access Control, 역할 기반 접근 제어)**는
사용자에게 직접 권한을 주는 대신, **역할(Role)**에 권한을 부여하고
사용자는 역할을 통해 간접적으로 권한을 얻는 구조다.

핵심 원리 두 가지:

**① 역할 분리(Separation of Duties)**
서로 다른 기능은 서로 다른 역할이 담당한다.
"자금 집행"과 "자금 승인"을 같은 사람이 하지 않는다.

**② 최소 권한 원칙(Principle of Least Privilege)**
각 역할은 자신의 업무에 필요한 최소한의 권한만 갖는다.
수수료를 바꾸는 역할이 컨트랙트를 업그레이드할 이유는 없다.

핵심 문장:
> **RBAC는 "이 사람이 무엇을 할 수 있는가"가 아니라, "이 역할이 무엇을 할 수 있는가"를 정의한다.**

## 5) WEB3 서비스에서 자주 등장하는 역할 유형

서비스마다 다르지만, 다음 역할들이 반복적으로 등장한다.

| 역할 | 권한 범위 | 특징 |
|---|---|---|
| **Admin** | 역할 부여·취소, 전체 설정 변경 | 최상위 권한. Multisig로 보호되어야 함 |
| **Operator / Manager** | 파라미터 조정, 일상 운영 | Admin보다 낮은 권한. 빠른 실행이 필요 |
| **Pauser** | 컨트랙트 일시정지·재개 | 긴급 시 즉시 사용. 범위는 좁아야 함 |
| **Upgrader** | 컨트랙트 로직 업그레이드 | 가장 민감한 권한. 강한 통제 필요 |
| **Minter** | 토큰 발행 | 발행량 통제가 핵심 |
| **Withdrawer** | 자금 인출 | 재정 통제와 직결. 강한 제한 필요 |
| **Guardian** | 긴급 취소·거부권 | 독립적이어야 함. 실행권은 없음 |

설계자 포인트:
- 역할 목록이 명시되어 있지 않은 서비스는 권한 구조를 분석하기 어렵다.
- 역할 이름이 있어도 실제 권한 범위를 컨트랙트에서 확인해야 한다.
- "Admin이 모든 걸 한다"면 역할 분리가 없는 것과 같다.

---

## 6) 역할 부여와 취소 구조

RBAC에서 역할 자체를 "누가, 어떻게 부여하고 취소하는가"가 설계의 핵심이다.

### 역할 부여(Grant)
- Admin이 특정 주소에 역할을 부여한다.
- 역할 부여 자체가 강력한 권한이므로, Admin 역할은 Multisig로 보호되어야 한다.

### 역할 취소(Revoke)
- Admin이 역할을 회수한다.
- 역할자가 퇴직하거나, 키가 탈취됐거나, 오용이 발견됐을 때 즉시 취소 가능해야 한다.

### 역할 포기(Renounce)
- 역할 보유자가 스스로 역할을 반납한다.
- 프로토콜이 성숙해지면 Admin 역할을 아무도 갖지 않는 상태로 만들기도 한다.
  ("소유자 없는 컨트랙트" — immutable 상태로 만들겠다는 의사 표명)

설계자 포인트:
- 역할 부여·취소 이력이 On-chain 이벤트로 기록되는가?
- Admin 역할 자체가 단일 키인가, Multisig인가?
- 역할 취소가 즉각 반영되는가, 지연이 있는가?

## 7) onlyOwner vs. RBAC 구조 비교

| 구분 | onlyOwner | RBAC |
|---|---|---|
| 권한 주체 | 단일 Owner 주소 | 역할별 복수 주체 |
| 권한 범위 | 모든 관리 기능 | 역할별 최소 필요 기능 |
| 탈취 시 영향 | 전체 권한 탈취 | 해당 역할의 권한만 탈취 |
| 운영 유연성 | 낮음 (Owner만 실행) | 높음 (역할별 독립 실행) |
| 내부자 위험 | 단 1명의 배신으로 전체 노출 | 역할 범위 내에서만 피해 |
| 감사 추적 | 단일 주소 이력 | 역할별 이력 분리 가능 |

핵심 문장:
> **onlyOwner는 "열쇠 하나로 모든 문을 여는" 구조다. RBAC는 "각 문에 맞는 열쇠를 따로 만드는" 구조다.**

## 8) RBAC + Multisig + Timelock 조합

지금까지 배운 세 가지 메커니즘이 실제로 어떻게 조합되는지 보자.

**계층 구조:**

```
[최상위] Admin 역할
  └─ Multisig (3-of-5) + Timelock 48시간
     → 역할 부여·취소, 컨트랙트 업그레이드

[중간] Operator 역할
  └─ Multisig (2-of-3)
     → 파라미터 조정, 수수료 변경 (Timelock 없이도 허용)

[긴급] Pauser 역할
  └─ 단일 키 또는 소규모 Multisig (1-of-3)
     → 즉시 일시정지 (속도가 중요)

[독립] Guardian 역할
  └─ 독립 주체 (내부와 분리)
     → Timelock 제안 취소만 가능
```

이 구조에서:
- 가장 위험한 권한일수록 더 강한 합의 + 더 긴 지연이 필요하다.
- 빠른 대응이 필요한 권한은 범위를 좁히고 속도를 높인다.
- 권한의 강도와 실행 속도는 반비례 관계로 설계된다.

설계자 포인트:
> **권한이 강할수록 실행이 느려야 하고, 실행이 빨라야 할수록 권한이 좁아야 한다.**

## 9) DAO 거버넌스 기반 권한 구조

RBAC에서 Admin 역할을 특정 인물이나 팀이 아닌
**토큰 보유자 전체**에게 맡기는 방식이 DAO 거버넌스다.

기본 흐름:

```
제안 → 토큰 보유자 투표 → 통과 → Timelock → 실행
```

이 방식의 특징:
- 권한이 특정 주소가 아니라 **투표 결과**에 의해 결정된다.
- 투표 참여에 토큰 보유량이 영향을 미친다.
- 제안·투표·실행이 모두 On-chain에 기록된다.

Associate 수준에서 알아야 할 것:
- DAO 거버넌스도 결국 "누가 투표권을 얼마나 갖는가"라는 권한 구조다.
- 토큰이 특정 주소에 집중되어 있으면 사실상 단일 Admin과 다르지 않다.
- DAO라는 이름이 탈중앙화를 보장하지 않는다.

설계자 포인트:
- 이 프로토콜의 거버넌스 토큰이 어디에 집중되어 있는가?
- 투표가 실제로 일어나고 있는가, 아니면 형식적인가?

## 10) RBAC의 한계

**RBAC가 해결하지 못하는 것:**

1. **역할 설계 자체가 잘못된 경우**
   역할을 나눴지만 각 역할의 권한 범위가 여전히 너무 넓다면 분리 효과가 없다.
   형식적인 분리는 실질적인 위험 감소가 아니다.

2. **역할 보유자의 보안**
   역할이 세분화됐어도, 역할을 가진 키가 탈취되면 해당 역할 권한이 위협받는다.
   RBAC는 피해 범위를 줄이지만, 키 보안 자체를 해결하지는 않는다.

3. **역할 간 의존성**
   어떤 역할이 다른 역할의 실행을 사실상 강제할 수 있다면
   분리가 의미 없어진다. 역할 간 상호작용 설계도 중요하다.

4. **코드와 문서의 불일치**
   문서에는 RBAC가 명시되어 있지만 실제 컨트랙트에서는 onlyOwner로 구현된 경우가 있다.
   선언된 권한 구조와 실제 구현을 반드시 대조해야 한다.

핵심 문장:
> **RBAC는 "권한을 나눠야 한다"는 설계 의지를 구조로 만든 것이다. 의지가 없으면 형식만 남는다.**

---

## 11) Associate 관점: 서비스를 볼 때 확인하는 것

- 이 서비스의 권한 구조가 onlyOwner 단일 구조인가, 역할이 분리되어 있는가?
- 역할 목록이 명시되어 있는가? 각 역할의 권한 범위가 문서화되어 있는가?
- Admin 역할이 Multisig로 보호되고 있는가?
- 가장 민감한 권한(업그레이드, 자금 인출)이 가장 강한 통제 아래 있는가?
- 역할 부여·취소 이력이 On-chain에서 확인 가능한가?
- 선언된 역할 구조와 실제 컨트랙트 코드가 일치하는가?

다음 강의(ch4. Proxy와 Upgrade 패턴)에서는 RBAC에서 가장 강한 통제가 필요하다고 언급한
**Upgrader 역할**이 실제로 어떤 구조 위에서 작동하는지를 다룬다.

## 용어 보충 (Quick Reference)
- **RBAC(Role-Based Access Control)**: 역할에 권한을 부여하고, 주체는 역할을 통해 권한을 얻는 접근 제어 구조.
- **최소 권한 원칙(Principle of Least Privilege)**: 각 역할은 업무에 필요한 최소한의 권한만 가져야 한다는 원칙.
- **역할 분리(Separation of Duties)**: 서로 다른 기능은 서로 다른 역할이 담당해야 한다는 원칙.
- **역할 부여(Grant)**: 특정 주소에 역할 권한을 할당하는 행위.
- **역할 취소(Revoke)**: 특정 주소의 역할 권한을 회수하는 행위.
- **역할 포기(Renounce)**: 역할 보유자가 스스로 역할을 반납하는 행위.
- **onlyOwner**: 컨트랙트의 단일 소유자 주소만 실행할 수 있도록 제한하는 기본 권한 패턴.

## 헷갈리는 개념 보충
- RBAC를 사용한다고 해서 자동으로 안전한 것이 아니다. 역할 설계가 잘못되면 분리 효과가 없다.
- Admin 역할이 모든 역할을 부여할 수 있다면, Admin 탈취는 여전히 전체 위협이다. Admin 역할 자체에 대한 강한 보호가 전제되어야 한다.
- DAO 거버넌스는 탈중앙화된 RBAC의 한 형태다. 그러나 토큰 집중이 있으면 탈중앙화가 아니다.
