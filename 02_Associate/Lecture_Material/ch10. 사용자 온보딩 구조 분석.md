# 10강. 사용자 온보딩 구조 분석: 서비스 진입 단계에서 무슨 권한이 열리는가

## 1) 이 강의의 한 문장
**WEB3 온보딩은 단순한 "가입"이 아니다. 지갑 연결부터 자산 진입까지의 각 단계에서 사용자가 어떤 권한을 어떤 범위로 부여하는지가 서비스 전체 보안 구조의 첫 번째 경계를 만든다.**

## 2) 강의 목표 (수강생이 끝나고 할 수 있어야 하는 것)
이 강의를 마치면 수강생은:

1. WEB3 온보딩의 4단계(지갑 연결 → 네트워크 확인 → 권한 부여 → 자산 진입)를 순서대로 설명할 수 있다.
2. Approve / Permit / setApprovalForAll의 차이와 각각의 위험 수준을 말할 수 있다.
3. 무한 Approve가 구조적으로 왜 위험한지 설명할 수 있다.
4. 온보딩 분석에서 확인해야 할 항목을 도출할 수 있다.
5. 온보딩 취약점이 서비스 전체 리스크와 어떻게 연결되는지 설명할 수 있다.

---

## 3) WEB2 vs. WEB3 온보딩

**WEB2 서비스 가입:**
```
이메일 입력 → 비밀번호 설정 → 이용약관 동의 → 완료
```
사용자가 서비스에 주는 것: 개인 정보, 데이터 접근 동의.
서비스가 잃어도: 데이터 손실, 개인정보 유출.

**WEB3 서비스 온보딩:**
```
지갑 연결 → 네트워크 확인 → 권한 부여(Approve) → 자산 진입
```
사용자가 서비스에 주는 것: **자산에 대한 직접 접근 권한**.
서비스(또는 공격자)가 그 권한을 악용하면: 지갑의 자산이 직접 이동한다.

핵심 문장:
> **WEB3 온보딩에서 "동의"는 단순한 약관 동의가 아니라 블록체인 위에서 실제로 집행되는 권한 부여다.**

## 4) 온보딩 4단계

### 1단계: 지갑 연결 (Wallet Connection)

```
사용자 → 서비스의 "Connect Wallet" 클릭
          ↓
지갑 팝업 → "이 사이트에 연결하시겠습니까?"
          ↓
사용자 승인 → 서비스가 지갑 주소를 읽을 수 있게 됨
```

**이 단계에서 일어나는 것:**
- 서비스가 사용자의 **주소**를 알 수 있게 된다.
- 서비스가 잔액, 트랜잭션 이력을 읽을 수 있게 된다.
- 자산은 이동하지 않는다.
- 서명 요청을 보낼 수 있는 상태가 된다.

**이 단계에서 일어나지 않는 것:**
- 서비스가 아직 자산을 가져가거나 트랜잭션을 실행할 수 없다.
- 단순 연결 자체는 자산 리스크가 낮다.

설계자 포인트:
- 지갑 연결 요청이 온다고 해서 자산이 위험한 것은 아니다.
- 위험은 연결 **이후**의 서명 요청에서 시작된다.

---

### 2단계: 네트워크/체인 확인

```
서비스 → "이 네트워크로 전환하시겠습니까?" (예: Ethereum Mainnet)
          ↓
사용자 확인 → 지갑이 해당 체인으로 전환됨
```

**이 단계에서 확인해야 할 것:**
- 서비스가 어떤 체인에서 작동하는가?
- 사용자가 올바른 체인에 있는가?
- 동일한 주소가 다른 체인에서 다른 자산을 갖는다.

**네트워크 혼동 리스크:**
```
사용자: Ethereum Mainnet을 쓰려고 함
공격자: BNB Chain으로 전환 유도 → 가짜 서비스에서 서명 요청
결과: BNB Chain의 자산이 위험에 노출됨
```

또는 반대로:
```
사용자: Testnet에서 테스트하려고 함
실수로 Mainnet 연결 → 실제 자산으로 트랜잭션 실행
```

설계자 포인트:
- 서비스가 지원하는 체인 목록이 명확히 표시되는가?
- 네트워크 전환 요청이 왜 발생하는지 사용자에게 충분히 설명하는가?

---

### 3단계: 권한 부여 (Approval)

**이 단계가 온보딩에서 가장 중요하다.**

WEB3에서 사용자가 서비스에 자산 접근 권한을 주는 방법은 크게 세 가지다.

#### A. ERC-20 Approve

```
사용자 → Token Contract의 approve 함수 호출
         (spender: 서비스 컨트랙트 주소, amount: 허용 금액)
          ↓
Token Contract → allowance 상태 변경
          ↓
이후 서비스 컨트랙트가 transferFrom으로 토큰을 가져갈 수 있음
```

- 별도 트랜잭션이 필요하다. Gas가 든다.
- **amount를 얼마로 설정하느냐가 핵심이다.**
  - 정확한 금액: 필요한 만큼만 → 안전하지만 매번 Approve 필요
  - 무한 Approve(uint256 최대값): 한 번만 하면 됨 → 편하지만 위험

**무한 Approve의 구조적 위험:**
```
사용자: USDC 1,000개만 스왑하려고 무한 Approve 설정
          ↓
서비스 컨트랙트가 업그레이드됨 (ch4 Proxy 패턴 연결)
          ↓
새 로직이 악의적이라면: 사용자 지갑의 USDC 전체를 가져갈 수 있음
```

또는:
```
서비스 컨트랙트가 해킹됨
          ↓
공격자가 모든 무한 Approve 사용자의 토큰을 일괄 탈취
```

#### B. Permit (EIP-2612)

```
서비스 → 사용자에게 서명 요청 (트랜잭션 없음, Gas 없음)
          ↓
사용자 → 서명
          ↓
서비스 → 서명을 Token Contract에 제출 → allowance 생성
```

- Gas 없이 Approve 효과를 낸다. 사용자 경험이 편리하다.
- **트랜잭션 없이 권한이 생성되므로 피싱 위험이 높다.**
  - 사용자는 "그냥 서명했을 뿐"이라고 생각하기 쉽다.
  - 서명 내용을 확인하지 않으면 대규모 권한 부여가 일어날 수 있다.
- 서명에는 만료 시간(deadline)이 포함된다. 즉시 실행하지 않아도 유효하다.

#### C. setApprovalForAll (NFT 전체 승인)

```
사용자 → NFT 컨트랙트의 setApprovalForAll 호출
         (operator: 서비스 주소, approved: true)
          ↓
이후 서비스가 해당 컬렉션의 **모든 NFT**를 이동시킬 수 있음
```

- 특정 NFT 하나가 아니라 **전체 컬렉션**에 대한 권한을 준다.
- NFT 마켓플레이스, 게임 등에서 자주 사용된다.
- setApprovalForAll이 악의적인 컨트랙트에 주어지면 컬렉션 전체가 위험하다.

**세 가지 권한 부여 방식 비교:**

| 방식 | 범위 | Gas | 피싱 위험 | 취소 방법 |
|---|---|---|---|---|
| ERC-20 Approve (정액) | 지정 금액 | 필요 | 낮음 | amount=0으로 재Approve |
| ERC-20 Approve (무한) | 무제한 | 필요 | 중간 | amount=0으로 재Approve |
| Permit | 서명 시 지정 금액 | 불필요 | 높음 | 만료 전 취소 어려움 |
| setApprovalForAll | 전체 컬렉션 | 필요 | 높음 | approved=false로 재호출 |

---

### 4단계: 자산 진입 (Asset Entry)

권한 부여 이후 사용자가 실제로 자산을 서비스에 넣는 단계다.

**유형 1: 직접 전송**
```
사용자 → 서비스 컨트랙트로 토큰 전송 (transfer)
```
- 즉시 자산이 이동한다.
- 컨트랙트 주소가 맞는지 확인하지 않으면 영구 손실.

**유형 2: Approve 후 컨트랙트가 가져감 (Pull)**
```
사용자 → Approve (3단계 완료)
서비스 컨트랙트 → transferFrom으로 사용자 지갑에서 토큰을 가져감
```
- 스왑, Lending, 유동성 공급 등 대부분의 DeFi 패턴.

**유형 3: 네이티브 코인 전송 (ETH)**
```
사용자 → 트랜잭션에 value를 포함해 ETH 전송
```
- Approve 불필요. 직접 이동.
- 잘못된 컨트랙트로 보내면 복구 불가.

## 5) 온보딩 구조 분석: 무엇을 확인하는가

**체크 1: Approve 범위**
- 서비스가 기본값으로 무한 Approve를 요청하는가?
- 정확한 금액으로 Approve할 옵션이 있는가?
- Approve 내용(spender, amount)이 사용자에게 명확히 표시되는가?

**체크 2: Permit 사용 방식**
- Permit 서명 요청 시 내용(spender, amount, deadline)이 표시되는가?
- Permit의 만료 시간이 합리적인가? 무기한 Permit은 아닌가?

**체크 3: setApprovalForAll 사용**
- NFT 전체 승인을 요구하는 서비스인가?
- 최소한 필요한 범위로 제한되어 있는가?
- 서비스 종료 후 setApprovalForAll을 취소하는 안내가 있는가?

**체크 4: Approve 대상 컨트랙트**
- Approve의 spender가 실제 서비스 컨트랙트인가?
- 그 컨트랙트가 Proxy인가? (ch4 연결) 업그레이드되면 새 로직이 Approve를 악용할 수 있다.

**체크 5: 온보딩 흐름의 명확성**
- 각 서명 요청에 대한 설명이 충분한가?
- 사용자가 무엇에 서명하는지 이해하기 어려운 구조인가?

## 6) 온보딩 취약점이 서비스 전체와 연결되는 방식

온보딩 취약점은 단독으로 끝나지 않는다.
ch9에서 배운 교차 분석의 핵심이 여기서 드러난다.

**무한 Approve + Proxy Upgrade (ch4):**
```
사용자가 서비스 컨트랙트에 무한 Approve
          ↓
서비스 컨트랙트가 업그레이드됨
          ↓
새 로직이 transferFrom으로 사용자 잔액 전체 탈취 가능
```

**무한 Approve + 내부자 위협 (ch8 Custody 연결):**
```
서비스 내부자가 컨트랙트에 악성 함수 삽입
          ↓
무한 Approve된 사용자들의 토큰 일괄 탈취
```

**setApprovalForAll + 피싱:**
```
사용자가 가짜 서비스에서 setApprovalForAll 실행
          ↓
공격자 주소에 NFT 컬렉션 전체 접근 권한 부여
          ↓
공격자가 전체 NFT를 자신의 지갑으로 이전
```

설계자 포인트:
- 온보딩 단계의 권한 설계가 서비스 전체 보안의 첫 번째 방어선이다.
- 온보딩에서 과도한 권한이 열리면, 이후의 모든 보안 장치가 의미를 잃을 수 있다.

## 7) 온보딩 분석을 Architecture Read Sheet에 기록하는 방법

ch9에서 소개한 Architecture Read Sheet의 "온보딩 구조" 섹션을 채우는 방법.

**기록 형식:**
```
[온보딩 단계별 분석]

1단계: 지갑 연결
 - 연결 방식: MetaMask, WalletConnect 등
 - 연결 시 요청 권한: 주소 읽기 / 서명 요청 가능 여부
 - 특이사항: 없음 / 있음 (내용 기록)

2단계: 네트워크 확인
 - 지원 체인: Ethereum Mainnet, Arbitrum 등
 - 네트워크 전환 요청 발생 여부: 있음/없음
 - 특이사항: 없음 / 있음

3단계: 권한 부여
 - Approve 대상 컨트랙트 주소: 0x...
 - Approve 기본값: 정액 / 무한 → (무한이면 취약점 메모)
 - Permit 사용 여부: 있음/없음
 - setApprovalForAll 요구 여부: 있음/없음
 - Approve 컨트랙트가 Proxy인지 여부: 예/아니오

4단계: 자산 진입
 - 자산 진입 방식: 직접 전송 / Pull / 네이티브 전송
 - 진입 시 추가 확인 절차: 있음/없음

[온보딩 취약점 메모]
 - (발견된 취약점과 연결된 다른 취약점 기록)
```

## 8) Associate 관점: 온보딩 구조 평가 기준

온보딩 구조를 평가할 때 "좋다/나쁘다"의 이분법이 아니라
**"필요한 수준을 넘는 권한이 있는가"**를 기준으로 본다.

| 평가 항목 | 허용 수준 | 주의 수준 | 취약 수준 |
|---|---|---|---|
| Approve 범위 | 정확한 금액 | 약간 여유 있는 금액 | 무한 Approve |
| Approve 대상 | 불변 컨트랙트 | Multisig+Timelock Proxy | 단일 키 Proxy |
| setApprovalForAll | 없음 또는 명확한 안내 | 있으나 사용 범위 제한 | 묻지도 않고 전체 승인 |
| Permit 사용 | deadline 짧고 금액 정확 | deadline 길고 금액 과다 | 무기한·무제한 Permit |
| 권한 설명 | 명확한 UI 안내 | 부분적 안내 | 설명 없음 |

## 용어 보충 (Quick Reference)
- **온보딩 (Onboarding)**: 사용자가 서비스를 처음 사용하기까지 거치는 일련의 단계. WEB3에서는 권한 부여와 자산 진입을 포함한다.
- **Approve**: ERC-20 토큰 컨트랙트에서 특정 주소(spender)가 사용자의 토큰을 가져갈 수 있도록 허용하는 함수.
- **무한 Approve (Unlimited Approve)**: amount를 uint256 최대값으로 설정해 금액 제한 없이 spender에게 권한을 주는 방식.
- **Permit (EIP-2612)**: 서명만으로 Gas 없이 Approve 효과를 내는 방식. 트랜잭션 없이 권한이 생성된다.
- **setApprovalForAll**: ERC-721/1155에서 특정 주소가 사용자의 전체 NFT 컬렉션을 이동시킬 수 있도록 허용하는 함수.
- **spender**: Approve에서 권한을 받는 주소. 보통 서비스 컨트랙트.
- **allowance**: 특정 spender가 가져갈 수 있도록 허용된 토큰 금액. Approve로 설정된다.
- **Pull 방식**: 컨트랙트가 사용자에게서 transferFrom으로 자산을 가져오는 방식. Approve가 선행되어야 한다.

## 헷갈리는 개념 보충
- 지갑 연결 자체는 자산 위험과 무관하다. 자산이 위험해지는 것은 Approve나 서명 이후다.
- 무한 Approve는 한 번 설정하면 명시적으로 취소(amount=0으로 재Approve)하지 않는 한 유지된다.
- Permit은 사용자가 서명했지만 아직 서비스에 제출되지 않았을 수 있다. 서명 자체가 유출되면 누구든 그 서명을 제출해 권한을 행사할 수 있다.
