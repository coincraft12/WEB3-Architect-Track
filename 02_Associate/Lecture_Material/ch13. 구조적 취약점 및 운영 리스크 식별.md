# 13강. 구조적 취약점 및 운영 리스크 식별: 3축 분석을 통합하여 리스크를 도출하라

## 1) 이 강의의 한 문장
**구조적 취약점은 서비스 설계 자체에 내재된 위험이고, 운영 리스크는 그 구조가 실제로 운영되는 방식에서 발생하는 위험이다. 두 가지를 구분하고 3축 분석을 통합해야 완전한 리스크 그림이 나온다.**

## 2) 강의 목표 (수강생이 끝나고 할 수 있어야 하는 것)
이 강의를 마치면 수강생은:

1. 구조적 취약점과 운영 리스크의 차이를 설명할 수 있다.
2. 온보딩·자산흐름·권한책임 3축 분석 결과를 통합하는 방법을 말할 수 있다.
3. 3축 교차점에서 심각한 취약점이 나타나는 이유를 설명할 수 있다.
4. 취약점 도출 결과를 Architecture Read Sheet에 기록할 수 있다.
5. Module 1(필기)의 분석 결과가 Module 2, 3(실기)의 입력으로 연결되는 구조를 이해한다.

## 3) 구조적 취약점 vs. 운영 리스크

### 구조적 취약점 (Structural Vulnerability)

서비스 설계 자체에서 비롯된 취약점.
코드, 아키텍처, 권한 모델이 잘못 설계되어 있어 사람이 실수를 하지 않아도 위험이 존재한다.

**특징:**
- 서비스가 정상적으로 운영되는 동안에도 잠재된 위험
- 설계 변경(코드 수정, 권한 구조 재설계)으로만 해결 가능
- 발견 즉시 즉각적인 위험은 아니지만, 특정 조건이 갖춰지면 발현

**예시:**
```
- Admin이 단일 EOA → 키 탈취 시 서비스 전체 위험
- Proxy 업그레이드 권한이 Admin과 동일 → 악의적 로직 교체 가능
- 무한 Approve 구조 → 미래의 컨트랙트 취약점이 사용자 자산에 영향
- 출금 없는 진입 경로 → 자산이 영구적으로 잠길 수 있음
```

### 운영 리스크 (Operational Risk)

올바르게 설계된 구조도 운영 과정에서 발생하는 위험.
사람의 실수, 절차 부재, 외부 환경 변화로 발생한다.

**특징:**
- 정상 구조에서도 운영 방식에 따라 위험 수준이 달라짐
- 절차 개선, 교육, 모니터링으로 일부 완화 가능
- 예방 가능한 경우가 많지만 인적 요소가 개입됨

**예시:**
```
- Multisig 서명자 키 관리 소홀 → 서명자 탈취 가능
- 파라미터 변경 전 충분한 검토 절차 부재 → 잘못된 값 설정
- 외부 프로토콜 감사 없이 연동 → 연동 후 해킹 피해
- 비상 권한 행사 기준 불명확 → 임의적 사용 가능
```

### 구분 기준 정리

| 구분 기준 | 구조적 취약점 | 운영 리스크 |
|---|---|---|
| 발생 원인 | 설계 결함 | 운영 방식 결함 |
| 해결 방법 | 코드/아키텍처 변경 | 절차/인적 개선 |
| 잠재성 | 항상 존재 | 상황에 따라 다름 |
| 발현 조건 | 특정 조건 충족 시 | 실수/외부 변화 시 |

## 4) 3축 분석 통합

ch9에서 정의한 3축 분석 프레임:
1. **온보딩 축** (ch10): 사용자가 서비스에 진입하는 경로와 권한 부여 구조
2. **자산흐름 축** (ch11): 자산의 이동·변환·출구 경로
3. **권한·책임 축** (ch12): 역할, 권한 위임 경로, 책임 경계

이 세 축은 독립적이지 않다. **교차점에서 가장 심각한 취약점이 나타난다.**

### 3축 교차 분석

**교차점 1: 온보딩 × 자산흐름**
```
온보딩 취약점: 과도한 Approve 허용
자산흐름 취약점: 외부 프로토콜 재예치
---
교차 리스크:
사용자가 A 서비스에 무한 Approve를 했는데,
A 서비스가 사용자 자산을 외부 B 프로토콜에 재예치한다.
B 프로토콜이 해킹되면 → A 서비스의 무한 Approve 경로로 추가 자산 인출 가능.
```

**교차점 2: 자산흐름 × 권한·책임**
```
자산흐름 취약점: 관리자 직접 인출 경로 존재 (onlyOwner)
권한·책임 취약점: Admin이 단일 EOA
---
교차 리스크:
Admin 키 하나만 탈취되면 모든 사용자 자산을 즉시 인출 가능.
Multisig나 Timelock 없이 즉시 실행 가능.
```

**교차점 3: 온보딩 × 권한·책임**
```
온보딩 취약점: KYC/검증 없이 누구나 가입 가능
권한·책임 취약점: 역할 부여 로직에 취약점 존재
---
교차 리스크:
악의적 주체가 온보딩 후 역할 부여 로직을 악용하여
자신에게 Operator 또는 Admin 역할을 부여할 가능성.
```

**3축 모두 교차:**
```
온보딩 취약점 + 자산흐름 집중 + 권한 집중
---
최대 리스크:
단일 Admin이 전체 자산을 통제하고,
사용자는 무한 Approve로 연결되어 있으며,
모든 자산이 단일 컨트랙트에 집중되어 있다.
→ Admin 탈취 = 전체 사용자 자산 탈취
```

## 5) 취약점 도출 절차

### Step 1: 각 축에서 취약점 목록 작성

**온보딩 축 취약점:**
```
O-1: 무한 Approve 요구 (Approve 한도 제한 없음)
O-2: 진입 자산 검증 없음 (어떤 토큰도 진입 가능)
O-3: 연결 지갑 검증 없음 (스마트 컨트랙트 지갑 구분 없음)
```

**자산흐름 축 취약점:**
```
F-1: 외부 Yield 프로토콜 재예치 (명시적 공개 없음)
F-2: pause() 권한으로 출금 차단 가능
F-3: 수수료 → 단일 EOA Fee Collector
```

**권한·책임 축 취약점:**
```
A-1: Fee Collector가 단일 EOA (책임 경계 불명확)
A-2: Admin이 업그레이드 권한 동시 보유
A-3: Risk Manager 서명자 목록 비공개
```

### Step 2: 교차점 분석으로 복합 리스크 도출

```
F-3 (수수료 단일 EOA) × A-1 (Fee Collector 단일 EOA)
→ 동일 취약점의 확인: 수수료 흐름 전체가 단일 키에 의존

F-2 (pause로 출금 차단) × A-2 (Admin이 upgrade 권한 보유)
→ 복합 리스크: Admin이 출금을 막고 → 컨트랙트 로직을 바꿔 자산 이동 가능

O-1 (무한 Approve) × F-1 (외부 재예치)
→ 복합 리스크: 외부 프로토콜 해킹 + 무한 Approve = 추가 피해 경로
```

### Step 3: 리스크 분류

| 리스크 ID | 유형 | 내용 | 발생 가능성 | 영향도 |
|---|---|---|---|---|
| R-1 | 구조적 취약점 | Admin 단일 키 + 업그레이드 권한 = 전체 통제 가능 | 중 | 극대 |
| R-2 | 구조적 취약점 | 무한 Approve + 외부 재예치 = 추가 피해 경로 | 중 | 대 |
| R-3 | 운영 리스크 | 외부 프로토콜 감사 절차 불명확 | 중 | 대 |
| R-4 | 구조적 취약점 | pause 권한 = 출금 차단 가능 | 저 | 대 |
| R-5 | 운영 리스크 | Fee Collector 단일 EOA 키 관리 절차 불명확 | 중 | 중 |

## 6) 구조적 취약점 vs. 운영 리스크: 실제 사례 패턴

**사례 1: Rug Pull (구조적 취약점)**
```
구조: Admin이 모든 자산에 직접 접근 가능 (withdraw onlyOwner)
발현: Admin이 모든 자산을 인출 후 사라짐
→ 구조 자체가 문제: 이 기능이 존재한 시점부터 위험
```

**사례 2: 파라미터 오설정 (운영 리스크)**
```
구조: Risk Manager가 LTV를 조정하는 구조 (합리적 설계)
발현: 90% LTV 설정 오류 → 담보 불충분 대출 발생
→ 구조는 괜찮지만 운영 절차(검토, 시뮬레이션, 다중 승인)가 없음
```

**사례 3: 오라클 조작 (구조적 취약점 + 운영 리스크 복합)**
```
구조: 단일 오라클에 의존 (구조적 취약점)
운영: 오라클 이상 징후 모니터링 없음 (운영 리스크)
발현: 오라클 가격 조작 → 부당 대출 → 청산 연쇄
→ 구조 개선(다중 오라클) + 운영 개선(모니터링 절차) 모두 필요
```

## 8) Module 1 → Module 2, 3 연결

이 강의(ch13)는 Module 1(필기)의 마지막 단계이자 Module 2, 3(실기)의 입력 단계다.

```
Module 1 출력:
  Architecture Read Sheet
    └─ 섹션 1: 서비스 개요
    └─ 섹션 2: 온보딩 구조
    └─ 섹션 3: 자산흐름 구조
    └─ 섹션 4: 권한·책임 구조
    └─ 섹션 5: 취약점 및 리스크 목록 ← 이 강의의 산출물

       ↓ 입력으로 연결

Module 2 (권한 통제 설계):
  취약점 목록 중 권한 관련 항목 선별
  → Authority Control Matrix 작성 (ch16)

Module 3 (자금 흐름 기반 리스크 평가):
  취약점 목록 중 자산흐름 관련 항목 선별
  → Fund Flow Risk Sheet 작성 (ch20)
```

**설계자 포인트:**
- ch13의 취약점 도출이 정확하고 완전할수록 실기 답안의 질이 높아진다.
- 실기에서 "왜 이 취약점이 문제인가"를 설명하려면 ch1~ch11의 개념이 바탕이 되어야 한다.

## 용어 보충 (Quick Reference)
- **구조적 취약점 (Structural Vulnerability)**: 서비스 설계 자체에 내재된 취약점. 설계 변경 없이는 완전히 해소되지 않는다.
- **운영 리스크 (Operational Risk)**: 정상 구조에서도 운영 방식에 따라 발생하는 위험. 절차 개선으로 완화 가능.
- **복합 리스크 (Composite Risk)**: 두 개 이상의 취약점이 교차할 때 발생하는 증폭된 위험.
- **3축 교차 분석**: 온보딩·자산흐름·권한책임의 취약점이 서로 어떻게 연결되는지를 분석하는 방법.
- **우선순위 (Priority)**: 발생가능성과 영향도를 종합하여 리스크를 대응할 순서를 정하는 것. Module 3에서 세부 기준을 학습한다.

## 헷갈리는 개념 보충
- 구조적 취약점이 더 심각하다고 볼 수 없다. 설계가 완벽해도 운영 리스크로 큰 손실이 발생할 수 있다. 두 가지 모두 분석해야 한다.
- 취약점을 발견했다고 즉시 위험한 것은 아니다. 발생 가능성과 영향도를 함께 평가해야 한다. (Module 3에서 세부 기준 학습)
- 교차 리스크가 중요한 이유: 각각의 취약점이 단독으로는 낮은 위험도여도, 교차 시 증폭될 수 있다.
