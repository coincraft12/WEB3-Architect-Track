# 21강. 종합 시나리오 분석: Module 1~3의 통합 적용

## 1) 이 강의의 한 문장
**종합 시나리오 분석은 하나의 실제 서비스를 대상으로 온보딩→자산흐름→권한구조→취약점→통제안·개선방향의 전체 흐름을 Module 1~3의 방법론을 통합하여 완성하는 실습이다.**

## 2) 강의 목표 (수강생이 끝나고 할 수 있어야 하는 것)
이 강의를 마치면 수강생은:

1. Module 1~3에서 학습한 분석 방법론을 하나의 서비스에 통합 적용할 수 있다.
2. Architecture Read Sheet → ACM → FFRS의 산출물 연결 구조를 이해한다.
3. 실기 시험에서 주어진 서비스를 분석하고 두 가지 실기 산출물을 작성할 수 있다.
4. 분석 결과를 논리적으로 서술하는 형식을 갖출 수 있다.

## 4) 종합 시나리오: DEX + Lending 복합 서비스 (가상 사례)

### 서비스 개요

**서비스명:** DeFiX Protocol
**서비스 유형:** DEX(AMM) + Lending 복합 서비스
**주요 기능:**
- USDC/ETH 스왑 (AMM)
- LP 토큰을 담보로 한 USDC 대출
- 청산 메커니즘

**핵심 구조 정보:**
- AMM Pool 컨트랙트 (Proxy 구조, UUPS)
- Lending 컨트랙트 (Proxy 구조, Transparent)
- Oracle: Chainlink 단독 (Twap 미적용)
- 역할:
  - Admin: 3-of-5 Multisig (Timelock 없음)
  - Risk Manager: 2-of-3 Multisig
  - Fee Collector: 단일 EOA (0xFEE...)
  - Liquidator: Public (누구나)
- 외부 의존: Chainlink Oracle (가격 데이터)
- 감사: AMM 컨트랙트 감사 완료, Lending 컨트랙트 감사 진행 중

---

### Step 1~5: Module 1 분석 (Architecture Read Sheet)

**섹션 1: 서비스 개요**
```
서비스 유형: AMM + Lending 복합
TVL: 추정 $80M (AMM $50M, Lending $30M)
체인: Ethereum
Proxy 구조: 양쪽 모두 적용
감사 현황: AMM 완료, Lending 진행 중
```

**섹션 2: 온보딩 구조**
```
[지갑 연결]
- MetaMask / WalletConnect 지원
- 스마트 컨트랙트 지갑 구분 없음 (취약점 후보)

[AMM Approve]
- USDC, ETH에 대해 무한 Approve 요구 확인
- 취약점: 무한 Approve → AMM 침해 시 사용자 지갑 전체 노출

[Lending Approve]
- LP 토큰 담보 시 무한 Approve 요구
- 취약점: AMM과 동일한 무한 Approve 문제

[진입 검증]
- 최소 금액 없음, 모든 지갑 접근 가능
- KYC 없음 (DeFi 특성)
```

**섹션 3: 자산흐름 구조**
```
[AMM 자금 흐름]
진입점: addLiquidity(ETH, USDC) → LP 토큰 발행
내부: LP 토큰 보유 / 스왑 수수료 풀 누적
출구: removeLiquidity(LP 토큰) → ETH + USDC 반환
수수료: 스왑 수수료 0.3% → 0.25% LP 분배 + 0.05% Treasury(0xFEE...)

[Lending 자금 흐름]
진입점: deposit(LP 토큰) → 담보 등록
내부: LP 토큰 담보 보관 / LTV 80% 기준 USDC 대출
출구 A: repay(USDC+이자) → LP 토큰 반환
출구 B (청산): Liquidator가 부채 상환 → LP 토큰 할인 수령
수수료: 이자의 10% → Treasury(0xFEE...)

[숨겨진 경로 확인]
- 관리자 직접 인출: 없음 (감사 보고서 확인)
- pause 기능: 있음 (Admin Multisig 권한)
- emergencyWithdraw: 없음
```

**섹션 4: 권한·책임 구조**
```
역할 목록:
1. Admin: 3-of-5 Multisig
   - 권한: pause(), unpause(), upgrade() (양쪽), setOracle(), grantRole()
   - Timelock: 없음

2. Risk Manager: 2-of-3 Multisig
   - 권한: setLTV(), setLiquidationThreshold(), addAsset()

3. Fee Collector: 단일 EOA (0xFEE...)
   - 권한: collectFees() → Treasury

4. Liquidator: Public
   - 권한: liquidate(borrower)

5. Oracle (External): Chainlink
   - 역할: ETH/USDC 가격 데이터

권한 위임 경로:
Admin (Root of Trust)
  ├─ grantRole → Risk Manager
  ├─ grantRole → Fee Collector
  └─ upgrade() → AMM + Lending 양쪽 컨트랙트

책임 경계 평가:
- Admin 행동: Multisig 서명 이력 온체인 기록 (추적 가능)
- Timelock 없음: 즉시 실행 → 사용자 사전 대응 불가
- Fee Collector: 단일 EOA → 책임 귀속 불명확
```

**섹션 5: 취약점 목록**
```
구조적 취약점:
SV-01: Admin Timelock 없음 (업그레이드 즉시 실행 가능)
SV-02: Fee Collector 단일 EOA (수수료 흐름 단일 통제점)
SV-03: Lending 컨트랙트 감사 미완료 (재진입 등 미검증)
SV-04: Oracle 단일 소스 (Chainlink 장애/조작 시 전체 영향)
SV-05: LP 토큰 담보의 복합 리스크 (AMM + Lending 연결)
SV-06: 무한 Approve (AMM, Lending 양쪽)

운영 리스크:
OR-01: Timelock 없어 급격한 파라미터 변경 가능
OR-02: Oracle 장애 모니터링 절차 불명확

복합 리스크:
CR-01: SV-04(Oracle 조작) + SV-05(LP 담보) = LP 가격 조작으로 과잉 대출
CR-02: SV-01(업그레이드 즉시) + Admin 탈취 = 즉각 악성 로직 삽입
```

### Step 6~8: Module 2 (Authority Control Matrix)

**ACM:**

| 역할 | 보유 권한 | 통제 수준 | 취약점 | 개선 방향 |
|---|---|---|---|---|
| Admin (3-of-5) | upgrade() (AMM+Lending) | 3 (보통) | [업그레이드 권한 집중] Timelock 없음, 즉시 악성 업그레이드 가능 | 72시간 Timelock 적용. UPGRADER_ROLE 분리 (4-of-7 Multisig) 검토 |
| Admin (3-of-5) | pause() / setOracle() | 3 (보통) | [단일 관리자 집중] Multisig 탈취 3개로 전체 정지 및 Oracle 교체 | 긴급 정지 별도 Security Council 구성. setOracle()에 24시간 Timelock |
| Risk Manager (2-of-3) | setLTV() / setLiquidationThreshold() | 3 (보통) | [역할 미분리] LTV 변경 즉시 적용, 사용자 포지션 즉각 영향 | LTV 변경에 24시간 Timelock. 변경 폭 ±5% 내 제한 |
| Fee Collector (단일 EOA) | collectFees() | 1 (미흡) | [단일 관리자 집중] 단일 키 탈취 시 수수료 전체 통제 + 목적지 변경 가능 | 2-of-3 Multisig 전환. Treasury 주소 하드코딩 |
| Liquidator (Public) | liquidate() | 5 (우수) | 없음 (경제적 인센티브 기반 공개 함수) | 유지. 청산 보너스 적정성 정기 검토 |
| User | deposit(), borrow(), repay(), removeLiquidity() | 3 (보통) | [무한 Approve] AMM·Lending 양쪽 무한 Approve 요구 | 정확한 금액 Approve로 전환. ERC-2612 Permit 검토 |

**개선 우선순위:**
```
P0: 없음 (즉각 서비스 중단 필요 수준 아님)
P1: Fee Collector Multisig 전환 / upgrade() Timelock 적용
P2: Lending 컨트랙트 감사 완료 / 무한 Approve 개선 / Oracle 복수 소스
P3: UPGRADER_ROLE 분리 / LTV 변경 Timelock / Risk Manager 역할 세분화
```

### Step 9~12: Module 3 (Fund Flow Risk Sheet)

**자금 흐름 다이어그램:**
```
[AMM 흐름]
사용자 ──(무한 Approve + addLiquidity)──▶ AMM Pool
                                           │ LP 토큰 발행
                                           ▼
                                       사용자 (LP 토큰 보유)
스왑 사용자 ──(swap)──▶ AMM Pool ──(수수료 0.05%)──▶ Treasury (0xFEE...)

[Lending 흐름]
사용자 ──(무한 Approve + deposit LP)──▶ Lending Contract
                                           │
                              Chainlink Oracle ◀── 가격 조회
                                           │ LTV 80% 기준
                                           ▼
                                       사용자 (USDC 대출 수령)

[청산 흐름]
가격 하락 → LTV 초과
                │
                ▼
Liquidator ──(liquidate)──▶ Lending Contract
                                │ LP 토큰 할인 수령
                                ▼
                            Liquidator (LP 토큰)

[수수료 흐름]
이자 수익의 10% ──▶ Treasury (0xFEE... 단일 EOA)
```

**FFRS:**

| 흐름 구간 | 리스크 유형 | 심각도 | L×I | 개선 방향 |
|---|---|---|---|---|
| 사용자 → AMM Pool (LP 공급) | 무한 Approve: AMM 침해 시 사용자 지갑 전체 노출 | 3 (Medium) | M×M | 정확한 금액 Approve 정책. ERC-2612 Permit 도입 검토 |
| 사용자 → Lending (LP 담보) | 무한 Approve: Lending 침해 시 사용자 LP 토큰 전체 노출 | 3 (Medium) | M×M | 동일: 정확한 금액 Approve. Permit 도입 |
| Lending → Chainlink (가격 조회) | Oracle 단일 소스: 장애 또는 조작 시 전체 영향 | 4 (High) | M×H | Chainlink + TWAP 복수 Oracle. 편차 5% 초과 서킷 브레이커 |
| Lending → Chainlink (가격 조회) | 플래시론 + Oracle 조작: LP 가격 순간 조작 후 과잉 대출 | 4 (High) | M×H | LP 가격에 TWAP 적용 (최소 30분). 블록 내 가격 변화 임계값 |
| AMM Pool → Lending (LP 담보 가치) | AMM-Lending 복합 리스크: AMM 유동성 고갈 시 LP 가치 급락 → Lending 담보 부족 | 4 (High) | M×H | LP 담보 LTV 보수적 설정 (80% → 60%). AMM 유동성 하한선 모니터링 |
| Liquidator → Lending (청산) | 연쇄 청산: LP 가격 하락 → 청산 → LP 추가 매도 → 가격 추가 하락 | 3 (Medium) | M×M | 청산 보너스 최적화. 단계적 청산 (한 번에 최대 50%) 메커니즘 |
| Lending → 차입자 (대출) | 감사 미완료: 재진입 등 미검증 취약점 존재 가능 | 4 (High) | M×H | 즉시: 대출 한도 임시 축소. 단기: Lending 감사 완료 후 정상화 |
| 수수료 → Treasury (0xFEE...) | 수수료 중앙화: 단일 EOA 탈취 시 수수료 전액 통제 | 4 (High) | H×M | 2-of-3 Multisig 전환. Treasury 주소 하드코딩 |
| Admin → 컨트랙트 (업그레이드) | 즉시 업그레이드: Timelock 없이 악성 로직 교체 가능 | 4 (High) | M×H | 72시간 Timelock 적용. 업그레이드 사전 커뮤니티 공지 |
| AMM Pool → 사용자 (LP 출금) | 유동성 고갈: 대형 LP 동시 출금 시 소형 LP 인출 어려움 | 2 (Low) | L×M | 최대 단일 출금 비율 제한 (1회 최대 Pool의 20%). 유동성 집중도 모니터링 |

**FFRS 개선 우선순위:**
```
P1 (단기, 1~4주):
- 수수료 Multisig 전환 (심각도 4: 즉시 실현 가능)
- Lending 감사 완료 전 대출 한도 임시 축소 (심각도 4: 즉각 완화)
- upgrade() Timelock 적용 (심각도 4)

P2 (중기, 1~3개월):
- 복수 Oracle 도입 (심각도 4, 기술 복잡)
- LP 담보 TWAP 적용 (심각도 4)
- AMM-Lending 복합 리스크: LP LTV 재설정 (심각도 4)
- 무한 Approve → Permit 전환 (심각도 3)
- 연쇄 청산 방지 메커니즘 (심각도 3)

P3 (장기, 3개월 이상):
- 유동성 집중도 관리 (심각도 2)
```

## 5) 분석 결과 서술 형식

실기에서 분석 결과를 서술할 때의 표준 흐름:

```
[서비스 개요 요약]
→ 서비스 유형, TVL, 주요 구조 특징을 2~3문장으로 요약

[핵심 취약점 요약]
→ 가장 심각한 취약점 2~3개를 이유와 함께 서술

[ACM 핵심 분석]
→ 가장 위험한 역할/권한 조합과 개선 방향 강조

[FFRS 핵심 분석]
→ 가장 위험한 자금 흐름 구간과 개선 방향 강조

[통합 개선 로드맵]
→ P1~P3 우선순위 목록 제시
→ 어떤 순서로 개선하면 가장 효과적인지 근거 설명
```

**서술 예시 (핵심 취약점 부분):**
```
DeFiX Protocol의 가장 심각한 구조적 취약점은 세 가지다.

첫째, Lending 컨트랙트 감사 미완료(FFRS R-07, 심각도 4)다.
현재 $30M TVL의 Lending 컨트랙트가 감사 없이 운영 중이며,
재진입 공격 등의 취약점이 잠재한다.
즉시 대출 한도를 축소하고 감사 완료를 최우선 과제로 해야 한다.

둘째, AMM-Lending 복합 리스크(FFRS R-05, 심각도 4)다.
LP 토큰을 담보로 사용하는 구조에서 AMM 유동성 고갈이
Lending의 담보 가치 급락으로 이어지는 연쇄 취약점이 있다.
LP 담보 LTV를 80%에서 60%로 낮추고 AMM 유동성 하한선 모니터링을 도입해야 한다.

셋째, 수수료 중앙화(ACM Fee Collector, 통제 수준 1)다.
모든 수수료가 단일 EOA로 집중되어 있어 탈취 시
수수료 흐름 전체가 통제된다. 즉시 2-of-3 Multisig 전환이 필요하다.
```

## 6) Associate 과정 전체 요약

```
Module 1: WEB3 아키텍처 분석 (ch1~ch13)
│
├─ [권한 통제 메커니즘] ch1~ch4
│   - Multisig: 다중 서명으로 단일 키 의존 제거
│   - Timelock: 시간 지연으로 사전 공개 강제
│   - RBAC: 역할별 최소 권한 분리
│   - Proxy: 업그레이드 구조와 위험성
│
├─ [자산·자금 흐름 구조] ch5~ch8
│   - AMM: 유동성 풀, x*y=k, LP 토큰
│   - Lending: 담보-대출-청산 구조
│   - Bridge: Lock-Mint/Burn-Unlock, 검증자
│   - Custody: Hot/Cold, MPC
│
└─ [서비스 단위 분석 방법론] ch9~ch13
    - 3축 분석: 온보딩 / 자산흐름 / 권한책임
    - Architecture Read Sheet
    - 구조적 취약점 vs. 운영 리스크

Module 2: 권한 통제 설계 (ch14~ch16)
│
├─ 4가지 취약점 유형: 집중/Approve/업그레이드/미분리
├─ 유형별 통제 메커니즘 선택 기준
└─ Authority Control Matrix (ACM) 작성

Module 3: 자금 흐름 기반 리스크 평가 (ch17~ch21)
│
├─ 자금 흐름 추적: 입금→보관→출금 구간
├─ 리스크 유형: 서비스별 / 컨트랙트 상호작용
├─ 발생가능성×영향도 심각도 매트릭스
└─ Fund Flow Risk Sheet (FFRS) 작성
```

## 용어 보충 (Quick Reference)
- **Architecture Read Sheet**: Module 1 분석의 통합 산출물. 서비스 개요, 온보딩, 자산흐름, 권한구조, 취약점 목록을 포함.
- **Authority Control Matrix (ACM)**: Module 2 실기 산출물. 역할×권한×통제수준×취약점×개선방향.
- **Fund Flow Risk Sheet (FFRS)**: Module 3 실기 산출물. 흐름구간×리스크유형×심각도×개선방향.
- **통합 개선 로드맵**: ACM과 FFRS에서 도출된 개선 과제를 우선순위별로 정리한 실행 계획.

## 헷갈리는 개념 보충
- ACM과 FFRS에서 같은 취약점이 다르게 표현될 수 있다. ACM에서 "Fee Collector 단일 EOA (통제 수준 1)"로 표현된 것이 FFRS에서 "수수료 흐름 구간 - 수수료 중앙화 리스크 (심각도 4)"로 나타난다. 이는 같은 취약점을 권한 관점과 자금 흐름 관점에서 각각 본 것이다.
- 종합 분석에서 중요한 것은 취약점의 "상호 연결"을 이해하는 것이다. AMM의 유동성 문제가 Lending의 담보 가치에 영향을 주는 것처럼, 서비스 유형 간 상호작용에서 복합 리스크가 나타난다.
- 실기 답안은 완벽한 분석이 아니라 "논리적 근거를 갖춘 분석"이 중요하다. 모든 취약점을 찾는 것보다 발견한 취약점에 대해 ch1~ch20의 개념과 연결하여 설명하는 것이 좋은 답안이다.
