# 6강. Lending 구조와 자산 흐름: 담보를 맡기고 빌리고 청산되는 구조

## 1) 이 강의의 한 문장
**WEB3 Lending 프로토콜은 담보를 컨트랙트에 잠그고 그 가치 범위 안에서 다른 자산을 빌려주는 구조다. 담보 가치가 일정 수준 아래로 떨어지면 청산이 자동으로 실행되며, 이 과정에서 자산 흐름이 세 방향으로 갈라진다.**

## 2) 강의 목표 (수강생이 끝나고 할 수 있어야 하는 것)
이 강의를 마치면 수강생은:

1. Lending 프로토콜의 기본 구조(공급자·차용자·청산자)를 구분할 수 있다.
2. 과담보(Overcollateralization)가 왜 필요한지 설명할 수 있다.
3. LTV와 청산 임계값의 의미를 말할 수 있다.
4. 청산이 발생하는 조건과 자산 흐름을 추적할 수 있다.
5. 이자율이 어떻게 결정되는지 이용률 기반 메커니즘을 설명할 수 있다.
6. Lending 구조의 주요 리스크(오라클 조작·청산 연쇄·유동성 경색)를 말할 수 있다.

---

## 3) Lending 프로토콜이란 무엇인가

전통 금융에서 대출은 이렇게 작동한다.
- 은행이 예금자에게 이자를 주고 자금을 모은다.
- 대출 심사를 통해 신용이 검증된 사람에게 빌려준다.
- 상환 능력을 검토하고, 담보를 요구하기도 한다.

WEB3 Lending 프로토콜은 다르다.

- 신원 확인이 없다.
- 신용 심사가 없다.
- 컨트랙트가 규칙을 집행한다.

그렇다면 어떻게 상환을 보장하는가?

> **상환을 "보장"하지 않는다. 대신 빌리는 금액보다 더 많은 담보를 먼저 잠근다.**

이것이 **과담보(Overcollateralization)** 구조다.

## 4) 세 가지 참여자와 자산 흐름

Lending 프로토콜에는 세 종류의 참여자가 있고 자산 흐름이 각각 다르다.

### A. 공급자 (Supplier / Lender)

자산을 프로토콜에 예치하고 이자를 받는 참여자.

```
공급자 → USDC 1,000개 예치
          ↓
프로토콜 → aUSDC 1,000개 발행 (이자 누적 증표)
          ↓
        시간 경과 (이자 누적)
          ↓
공급자 → aUSDC 반환
          ↓
프로토콜 → USDC 1,000 + 이자 반환
```

- aUSDC(Aave), cUSDC(Compound) 등은 공급 지분의 증표 토큰이다.
- Basic의 증표 토큰 개념과 동일하다. "내 자산이 프로토콜 안에 있다"는 권리 표시.
- 시간이 지날수록 1 aUSDC로 교환할 수 있는 USDC가 늘어난다.

### B. 차용자 (Borrower)

담보를 잠그고 다른 자산을 빌리는 참여자.

```
차용자 → ETH 1개를 담보로 잠금
          ↓
프로토콜 → LTV 기준으로 USDC 최대 2,400개까지 대출 가능
          ↓
차용자 → USDC 2,000개 대출
          ↓
        시간 경과 (이자 누적)
          ↓
차용자 → USDC 2,000 + 이자 상환
          ↓
프로토콜 → ETH 1개 반환
```

### C. 청산자 (Liquidator)

담보 가치가 하락해 청산 조건이 충족되면 개입하는 참여자.

```
ETH 가격 하락 → 담보 가치 < 청산 임계값
                  ↓
청산자 → 차용자의 빚 일부 대신 상환
          ↓
프로토콜 → 청산자에게 담보(ETH)를 할인가에 제공
          (청산 보너스: 보통 5~15%)
```

설계자 포인트:
- 청산자는 수익을 위해 자발적으로 참여한다. 청산 인센티브가 없으면 프로토콜이 부실화된다.
- 청산자가 없는 상황(시장 급락, 네트워크 혼잡)은 프로토콜 전체의 리스크다.

## 5) 담보 비율: LTV와 청산 임계값

Lending 프로토콜의 핵심 파라미터는 두 가지다.

**LTV (Loan-to-Value, 담보 대비 대출 비율)**
- 담보 가치 대비 최대 얼마까지 빌릴 수 있는가.
- ETH의 LTV가 80%라면: ETH 3,000달러 담보 → 최대 2,400달러 대출 가능.

**청산 임계값 (Liquidation Threshold)**
- 이 비율을 넘으면 청산이 트리거된다.
- LTV보다 높게 설정된다. (예: LTV 80%, 청산 임계값 85%)
- 사용자가 LTV 한도까지 빌린 후 담보 가치가 하락해 청산 임계값을 넘으면 청산된다.

**헬스 팩터 (Health Factor)**
- 현재 포지션의 안전 수준을 나타내는 지표.
- 1.0 미만이 되면 청산 가능 상태.
- 1.0 이상일수록 안전. 높을수록 담보 여유가 많다.

수식으로 보면:
```
헬스 팩터 = (담보 가치 × 청산 임계값) / 총 빚
헬스 팩터 < 1.0 → 청산 가능
```

예시:
```
ETH 담보: 3,000달러, 청산 임계값: 85%
대출 금액: 2,400달러

헬스 팩터 = (3,000 × 0.85) / 2,400 = 1.0625 → 안전

ETH 가격이 3,000 → 2,600달러로 하락하면:
헬스 팩터 = (2,600 × 0.85) / 2,400 = 0.9208 → 청산 가능
```

## 6) 이자율 결정: 이용률 기반 메커니즘

WEB3 Lending에서 이자율은 중앙 기관이 정하지 않는다.
**이용률(Utilization Rate)**에 따라 자동으로 결정된다.

```
이용률 = 총 대출 금액 / 총 예치 금액
```

이용률이 높다는 것은:
- 예치된 자산 중 많은 부분이 대출 중이다.
- 공급자가 인출하려고 해도 유동성이 부족할 수 있다.

이자율 곡선:
```
이용률 낮음 → 이자율 낮음 (공급 유도, 대출 장려)
이용률 높음 → 이자율 급격히 상승 (추가 공급 유도, 대출 억제)
```

설계자 포인트:
- 이용률이 지나치게 높아지면 공급자가 인출을 요청해도 자금이 없는 **유동성 경색** 상태가 된다.
- 이 상태에서 급격한 가격 하락이 겹치면 청산도 안 되고 인출도 안 되는 최악의 상황이 된다.

## 7) 청산 흐름 상세

청산은 Lending 프로토콜에서 가장 중요한 자산 흐름이다.
정상 작동할 때는 프로토콜을 보호하지만, 실패하면 전체가 부실화된다.

**정상 청산 흐름:**
```
1. 담보 가치 하락 → 헬스 팩터 < 1.0
2. 청산자 발견 → 청산 가능 포지션 확인
3. 청산자 → 빚의 일부(최대 50%) 대신 상환
4. 프로토콜 → 청산자에게 담보를 청산 패널티만큼 할인해서 제공
5. 남은 담보는 차용자에게 반환
```

**청산 연쇄(Cascade Liquidation) 위험:**
```
ETH 가격 급락
  ↓
대량 포지션 동시 청산 조건 충족
  ↓
청산자들이 담보(ETH)를 시장에 매도
  ↓
ETH 가격 추가 하락
  ↓
더 많은 포지션 청산 조건 충족
  ↓
반복 → 가격 폭락 가속
```

설계자 포인트:
- 단일 담보 자산(특히 변동성 높은 자산)에 집중된 프로토콜은 연쇄 청산 리스크가 높다.
- 청산 패널티가 너무 낮으면 청산자가 참여하지 않는다. 너무 높으면 차용자 손실이 과도하다.

## 8) 오라클 의존성: Lending의 핵심 취약점

Lending 프로토콜은 담보 가치를 알아야 LTV와 청산을 계산할 수 있다.
이 담보 가치를 외부에서 가져오는 것이 **오라클(Oracle)**이다.

```
오라클 → "현재 ETH 가격은 3,000달러" → 프로토콜 참조
```

문제는 오라클이 조작될 수 있다는 것이다.

**가격 조작 공격 시나리오:**
```
1. 공격자 → AMM 풀에서 ETH 가격을 일시적으로 급등시킴 (플래시론 이용)
2. 프로토콜이 조작된 가격을 참조 → 담보 가치가 부풀려짐
3. 공격자 → 부풀려진 담보 가치 기준으로 대규모 대출
4. ETH 가격 원상복구 → 담보 가치 폭락 → 전액 대출금 탈취
```

설계자 포인트:
- 단일 AMM 풀의 순간 가격을 오라클로 쓰는 것은 위험하다.
- Chainlink 같은 분산형 오라클이나 시간 가중 평균 가격(TWAP)을 쓰는 것이 기준이다.
- 오라클 조작은 Lending 프로토콜 해킹의 가장 빈번한 경로 중 하나다.

## 9) AMM과 Lending의 연결: 컴포저빌리티 리스크

ch5(AMM)와 이번 강의는 독립적이지 않다. 실제 DeFi에서는 연결된다.

**연결 사례:**
- AMM 풀에 유동성을 공급하고 받은 LP 토큰을 Lending 프로토콜에 담보로 사용한다.
- Lending에서 빌린 자산을 다시 AMM에서 교환해 추가 수익을 낸다 (레버리지 파밍).

이 연결이 만드는 리스크:
```
AMM 풀 붕괴
  ↓
LP 토큰 가치 폭락
  ↓
LP 토큰을 담보로 쓴 Lending 포지션 일제히 청산
  ↓
청산으로 인한 자산 매도 → AMM 풀 추가 압박
  ↓
연쇄 붕괴
```

설계자 포인트:
- 단일 프로토콜의 리스크가 아니라 연결된 프로토콜 전체의 시스템 리스크를 봐야 한다.
- 이것이 ch5~ch8을 별도로 배우는 이유다. 각 구조를 알아야 연결 리스크가 보인다.

## 10) Associate 관점: 서비스를 볼 때 확인하는 것

**담보 구조 관점:**
- 어떤 자산이 담보로 허용되는가? 변동성이 높은 자산일수록 리스크가 크다.
- LTV와 청산 임계값이 자산별로 다르게 설정되어 있는가?
- 단일 담보 자산에 집중되어 있는가, 다양화되어 있는가?

**오라클 관점:**
- 담보 가격을 어디서 가져오는가? 단일 AMM 풀 가격인가, 분산형 오라클인가?
- TWAP(시간 가중 평균 가격)을 사용하는가?

**청산 메커니즘 관점:**
- 청산 패널티가 청산자 참여를 유도하기에 충분한가?
- 시장 급락 시 청산자가 없는 상황에 대한 대비책이 있는가?
- 청산 파라미터 변경 권한이 어떻게 통제되는가? (ch3 RBAC 연결)

**유동성 관점:**
- 현재 이용률은 얼마인가? 과도하게 높지 않은가?
- 유동성 경색 시 긴급 대응 메커니즘이 있는가?

## 용어 보충 (Quick Reference)
- **Lending 프로토콜**: 담보를 잠기고 다른 자산을 빌려주는 탈중앙화 대출 구조.
- **과담보 (Overcollateralization)**: 빌리는 금액보다 더 큰 가치의 담보를 먼저 예치하는 구조.
- **LTV (Loan-to-Value)**: 담보 가치 대비 최대 대출 가능 비율.
- **청산 임계값 (Liquidation Threshold)**: 이 비율을 초과하면 청산이 발동되는 기준값. LTV보다 높다.
- **헬스 팩터 (Health Factor)**: 포지션의 안전도 지표. 1.0 미만이면 청산 가능.
- **이용률 (Utilization Rate)**: 총 예치 금액 중 대출 중인 비율. 이자율 결정의 기준.
- **청산 연쇄 (Cascade Liquidation)**: 가격 하락 → 청산 → 담보 매도 → 추가 하락이 반복되는 구조적 위험.
- **aToken / cToken**: Aave/Compound에서 공급 지분을 나타내는 증표 토큰. 시간이 지나면서 이자가 누적된다.
- **오라클 (Oracle)**: 블록체인 외부의 가격 정보를 컨트랙트에 제공하는 데이터 소스.
- **TWAP (Time-Weighted Average Price)**: 일정 시간 동안의 평균 가격. 순간 가격 조작에 강하다.

## 헷갈리는 개념 보충
- 공급자(Lender)와 차용자(Borrower)는 같은 프로토콜에 동시에 존재한다. 공급자가 예치한 자산을 차용자가 빌린다.
- LTV와 청산 임계값은 다르다. LTV는 "최대 얼마까지 빌릴 수 있나"이고, 청산 임계값은 "언제 청산되나"의 기준이다. 보통 청산 임계값이 LTV보다 높다.
- 헬스 팩터는 변한다. 담보 가격이 오르면 올라가고, 빚이 늘거나 담보 가격이 내리면 떨어진다. 실시간으로 모니터링해야 한다.
