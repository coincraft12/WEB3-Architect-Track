# 15강. 취약점에서 통제안 도출: 유형별 대응 메커니즘과 선택 기준

## 1) 이 강의의 한 문장
**취약점 유형마다 적합한 통제 메커니즘이 다르며, 언제 Multisig를 쓰고 언제 Timelock을 쓰고 언제 RBAC를 쓰는지 판단하는 것이 권한 통제 설계의 핵심이다.**

## 2) 강의 목표 (수강생이 끝나고 할 수 있어야 하는 것)
이 강의를 마치면 수강생은:

1. ch14의 4가지 취약점 유형에 대한 대응 메커니즘을 매핑할 수 있다.
2. Multisig / Timelock / RBAC / Proxy 통제 각각을 언제 적용해야 하는지 판단 기준을 말할 수 있다.
3. 상황별로 적합한 통제 조합을 선택하는 이유를 설명할 수 있다.
4. 개선 방향을 구체적인 형식으로 제시할 수 있다.

## 3) 취약점 → 통제안 매핑 원칙

통제안 도출은 두 단계로 이루어진다:

**Step 1: 취약점 유형 확인**
- ch14에서 학습한 4가지 유형 중 어디에 해당하는가?

**Step 2: 유형에 맞는 메커니즘 선택**
- 어떤 메커니즘이 이 취약점을 완화하는가?
- 여러 메커니즘의 조합이 필요한가?
- 서비스의 규모·성격·단계에서 현실적으로 적용 가능한가?

## 4) 유형 1 대응: 단일 관리자 집중 → Multisig 적용

### 핵심 원칙
단일 키 의존을 다중 서명으로 분산한다.
한 명이 탈취되어도 임계값 미달로 실행 불가능한 구조를 만든다.

### 메커니즘: Multisig

**적용 방법:**
```
Before:
Admin = 0xAAAA (단일 EOA)
→ 키 하나 탈취 = 전체 권한 탈취

After:
Admin = Gnosis Safe 3-of-5 Multisig
→ 5명의 서명자 중 3명이 동의해야 실행 가능
```

**임계값 선택 기준:**

| 상황 | 권장 임계값 | 이유 |
|---|---|---|
| 소규모 팀 (3명) | 2-of-3 | 최소 다수결, 단일 탈취 방지 |
| 중간 팀 (5명) | 3-of-5 | 과반 합의, 운영 유연성 유지 |
| 대형 팀 / 고위험 기능 | 4-of-7 | 높은 보안, 느린 의사결정 감수 |
| 핵심 기능 (자산 이동, 업그레이드) | N/2 초과 | 과반수 이상 동의 필수 |

**개선 방향 제시 형식:**
```
현황: Admin이 단일 EOA (0xAAAA)
취약점: 키 탈취 시 즉시 전체 권한 이전
개선 방향: Admin을 3-of-5 Multisig로 전환
  - 서명자 선정: 팀 내 독립적 역할을 가진 5명
  - 서명자 키 관리: 하드웨어 지갑 사용 권장
  - Multisig 주소: Gnosis Safe 등 감사된 구현체 사용
```

### 언제 Multisig를 쓰는가
- Admin 역할이 핵심 자산 이동, 정지, 업그레이드 권한을 가질 때
- 단일 키 의존으로 인한 단일 실패점이 식별될 때
- 팀 운영 중 여러 구성원이 공동 책임을 가져야 할 때

## 5) 유형 1 + 취약한 업그레이드 대응: Timelock 적용

### 핵심 원칙
권한이 있어도 즉시 실행하지 못하게 시간 지연을 강제한다.
이 지연 기간 동안 서비스 운영 주체는 변경 내용을 다시 검토하고, 필요하면 사용자는 사전 대응할 시간을 확보할 수 있다.

### 메커니즘: Timelock

**적용 방법:**
```
Before:
Admin → upgrade() 즉시 실행
→ 사용자 인지 전에 악성 로직 배포 가능

After:
Admin → Timelock에 업그레이드 예약 (queue)
→ 48시간 이상 지연 → 변경 내용 재검토
→ 지연 후 실행 (execute)
→ 이상 발견 시 취소 (cancel) 가능
```

**지연 시간 선택 기준:**

| 기능 유형 | 권장 지연 시간 | 이유 |
|---|---|---|
| 파라미터 조정 (소폭) | 24시간 | 빠른 대응 필요, 영향 제한적 |
| 파라미터 조정 (대폭) | 48~72시간 | 사용자 대응 시간 필요 |
| 컨트랙트 업그레이드 | 48시간 이상 | 코드 검토 + 사용자 출금 기회 |
| 긴급 상황 | 별도 예외 절차 검토 | 정상 변경과 동일 취급 시 대응 지연 가능 |

**개선 방향 제시 형식:**
```
현황: upgrade() 함수가 Multisig로만 보호, 즉시 실행 가능
취약점: Multisig 탈취 시 사용자 인지 전 악성 업그레이드 가능
개선 방향: 업그레이드 함수에 Timelock 적용
  - 지연 시간: 48시간 (최소 권장)
  - 실행 주체: Admin Multisig → Timelock 컨트랙트 → 대상 컨트랙트
  - 취소 권한: Admin Multisig
  - 예외 상황 처리 기준: 별도 운영 절차로 정리
```

### 언제 Timelock을 쓰는가
- 업그레이드 권한이 존재할 때 (거의 필수)
- 파라미터 변경이 사용자 자산에 직접 영향을 줄 때
- 사용자에게 변경 내용을 사전 공개해야 할 때

## 6) 유형 4 대응: 역할 미분리 → RBAC 적용

### 핵심 원칙
권한을 역할별로 분리하여 각 역할이 필요한 최소 권한만 갖도록 한다.
최소 권한 원칙(Principle of Least Privilege)을 구현한다.

### 메커니즘: RBAC (Role-Based Access Control)

**적용 방법:**
```
Before:
onlyOwner: setFee() + pause() + withdraw() + upgrade()
→ Admin 하나가 모든 기능 접근

After:
FEE_MANAGER_ROLE: setFee()
PAUSER_ROLE: pause(), unpause()
ASSET_MANAGER_ROLE: withdraw() ← 추가 통제 필요
UPGRADER_ROLE: upgrade() ← Timelock 조합 필요
DEFAULT_ADMIN_ROLE: 역할 부여/회수만
```

**역할 설계 원칙:**
1. **기능 단위로 역할 분리**: 하나의 역할이 하나의 기능 범주를 담당
2. **성격이 다른 권한 혼합 금지**: 일상 운영 권한과 비상 통제 권한을 분리
3. **자산 이동 권한 별도 보호**: withdraw 계열은 Multisig + Timelock 조합
4. **역할 부여 권한 분리**: 누가 역할을 줄 수 있는지도 제한

**개선 방향 제시 형식:**
```
현황: 모든 관리 기능이 onlyOwner로 단일 역할에 집중
취약점: 역할 미분리 → Admin 탈취 = 전체 기능 통제
개선 방향: RBAC 구조로 역할 분리
  - FEE_MANAGER_ROLE: 수수료 관련 파라미터만
  - PAUSER_ROLE: 긴급 정지/해제만
  - UPGRADER_ROLE: 업그레이드 권한 (+ Timelock 적용)
  - DEFAULT_ADMIN_ROLE: 역할 관리만 (자산 접근 불가)
  - 각 역할을 독립적인 Multisig 주소에 할당
```

### 언제 RBAC를 쓰는가
- 관리 기능이 두 가지 이상이고 성격이 다를 때
- 팀 내 역할이 분리되어 있을 때 (개발팀 / 리스크팀 / 운영팀)
- 파트너 또는 외부 주체에게 제한적 권한을 부여해야 할 때
- 단일 Admin에 너무 많은 권한이 집중되어 있을 때

## 7) 유형 3 대응: 업그레이드 권한 집중 → Proxy 통제 강화

### 핵심 원칙
업그레이드 권한을 Admin에서 분리하고, 별도의 통제 체계(Multisig + Timelock)로 보호한다.

### 메커니즘: Upgrader 역할 분리 + Timelock 조합

**적용 방법:**
```
Before:
Admin Multisig (3-of-5) → upgrade() 즉시 실행

After:
Upgrader Multisig (4-of-7) → Timelock (72시간) → upgrade() 실행
                               ↑
                       Admin Multisig는 업그레이드 권한 없음
```

**설계 포인트:**
- Upgrader를 Admin보다 높은 임계값의 Multisig로 설정 (더 신중한 의사결정)
- 변경 권한과 실행 권한을 분리해 검토 단계를 확보
- 예외 상황에서도 최소한의 검토 기준을 둘 것

**개선 방향 제시 형식:**
```
현황: Admin Multisig (3-of-5)가 upgrade() 권한 보유, Timelock 없음
취약점: Admin 탈취 + 즉시 악성 업그레이드 가능
개선 방향:
  1. UPGRADER_ROLE을 Admin과 분리
  2. Upgrader = 4-of-7 Multisig (Admin보다 높은 임계값)
  3. upgrade() 실행 시 Timelock (72시간) 적용
  4. 예외 상황 처리 기준을 별도 문서로 정리
```

## 8) 유형 2 대응: 무한 Approve → Approve 정책 개선

### 핵심 원칙
사용자에게 필요한 최소 금액만 Approve하도록 유도하거나, 무한 Approve의 리스크를 명시적으로 안내한다.

### 메커니즘: 정책 및 UX 개선

**적용 방법:**

**A. 서비스 측 Approve 정책 변경:**
```
Before:
approve(contract, type(uint256).max) // 무한

After:
approve(contract, exactAmount) // 사용량 만큼만
또는
approve(contract, exactAmount + 10%) // 소폭 버퍼
```

**B. ERC-2612 Permit 활용:**
```
Permit을 지원하는 토큰(USDC, DAI 등):
- 별도 Approve 트랜잭션 없이 서명만으로 1회성 권한 부여
- 사용 후 자동 만료 → 잔여 Approve 리스크 없음
```

**C. Permit2 (Uniswap) 활용:**
```
Permit2 컨트랙트를 통한 표준화된 서명 기반 Approve:
- 유효 기간 설정 가능
- 금액 제한 가능
- 사용자가 잔여 Approve를 직접 관리
```

**개선 방향 제시 형식:**
```
현황: 서비스 진입 시 type(uint256).max Approve 요구
취약점: 서비스 컨트랙트 해킹 시 사용자 지갑 전체 잔액 위험
개선 방향:
  1. (단기) 트랜잭션 단위 Approve로 변경 (정확한 금액만)
  2. (중기) ERC-2612 Permit 지원 토큰에 Permit 방식 도입
  3. (UI) Approve 요청 시 금액과 의미를 명확히 표시
  4. (가이드) 사용자에게 Approve Revoke 방법 안내
```

### 언제 각 방법을 쓰는가

| 상황 | 권장 방법 |
|---|---|
| 즉시 개선 가능 | 정확한 금액 Approve로 변경 |
| 토큰이 Permit 지원 | ERC-2612 Permit 방식 도입 |
| 복수 토큰/복잡한 흐름 | Permit2 검토 |
| 개선 전 임시 조치 | 사용자 안내 및 Revoke 가이드 |

## 9) 통제 메커니즘 선택 기준 종합

| 취약점 유형 | 1차 메커니즘 | 추가 강화 | 적용 기준 |
|---|---|---|---|
| 단일 관리자 집중 | Multisig | Timelock | Admin이 핵심 자산 권한 보유 시 |
| 무한 Approve | Approve 정책 개선 | Permit/Permit2 | 사용자 자산 직접 접근 구조 시 |
| 업그레이드 권한 집중 | Upgrader 분리 + Timelock | 실행 전 추가 검토 절차 | Proxy 구조 존재 시 |
| 역할 미분리 | RBAC | 역할별 Multisig | 관리 기능이 다수이고 성격이 다를 때 |

**통제 조합 선택 시 고려 사항:**
1. **서비스 단계**: 초기 스타트업 vs. 성숙한 프로토콜
2. **팀 규모**: 서명자를 몇 명이나 확보할 수 있는가
3. **운영 속도**: 긴급 대응이 얼마나 중요한가
4. **사용자 규모**: TVL(Total Value Locked)이 클수록 더 높은 통제 필요

## 10) 개선 방향 제시 형식 가이드

실기에서 개선 방향을 제시할 때 사용하는 표준 형식:

```
[취약점 ID]: [취약점 유형]
현황: [현재 구조의 구체적 설명]
리스크: [이 구조가 어떤 위험으로 이어지는지]
개선 방향:
  1. [1차 통제안 - 메커니즘 + 구체적 파라미터]
  2. [추가 강화 방안 - 있다면]
  3. [운영 절차 개선 - 있다면]
우선순위: 높음 / 중간 / 낮음
적용 시 고려 사항: [트레이드오프, 제약 조건]
```

**작성 예시:**
```
[취약점 ID]: SV-01 단일 관리자 집중
현황: Admin이 단일 EOA (0xAAAA), pause/upgrade/withdraw 모두 보유
리스크: 키 탈취 1회로 서비스 전체 통제권 이전, 즉시 자산 인출 가능
개선 방향:
  1. Admin을 Gnosis Safe 3-of-5 Multisig로 전환
     - 서명자: 팀 내 독립적 역할 5명 (개발 2, 운영 2, 법무/재무 1)
     - 하드웨어 지갑 사용 권장
  2. upgrade() 함수에 48시간 Timelock 적용
  3. withdraw()는 별도 ASSET_MANAGER_ROLE 분리 + Multisig
우선순위: 높음
적용 시 고려 사항: Multisig 전환 후 기존 owner 키 폐기 절차 필요
```

## 용어 보충 (Quick Reference)
- **최소 권한 원칙 (Principle of Least Privilege)**: 각 주체는 자신의 임무를 수행하는 데 필요한 최소한의 권한만 가져야 한다는 보안 원칙.
- **ERC-2612 Permit**: 별도의 Approve 트랜잭션 없이 서명(off-chain)으로 토큰 사용 권한을 부여하는 표준. gasless Approve.
- **Permit2**: Uniswap이 개발한 범용 서명 기반 Approve 표준. 유효기간 설정 가능.
- **TVL (Total Value Locked)**: 서비스에 예치된 전체 자산 규모. TVL이 클수록 통제 강화의 필요성이 높아진다.

## 헷갈리는 개념 보충
- Multisig와 Timelock은 목적이 다르다. Multisig는 "여러 명의 동의"를, Timelock은 "시간 지연을 통한 사전 공개"를 제공한다. 가장 강한 보호는 두 가지 조합이다.
- RBAC는 역할을 분리하지만, 각 역할에 적절한 Multisig를 연결해야 의미 있다. 역할만 분리하고 모두 단일 EOA라면 취약점이 분산될 뿐이다.
- 개선 방향은 현실 가능성을 고려해야 한다. 이상적인 구조라도 팀 규모나 운영 단계에서 적용 불가능한 통제안은 실효성이 없다.
