# 20.5강. Bridge/Oracle/Approval 실제 사고 리플레이

## 1) 이 강의의 한 문장
**실전 사고 리플레이의 목적은 사건을 암기하는 것이 아니라, 공격 경로를 구조적으로 재구성해 "다음 분석에서 같은 패턴을 즉시 식별"하는 데 있다.**

## 2) 강의 목표 (수강생이 끝나고 할 수 있어야 하는 것)
이 강의를 마치면 수강생은:

1. Bridge, Oracle, Approval 관련 실제 사고의 공통 공격 패턴을 설명할 수 있다.
2. 사건을 "전제 조건 -> 공격 단계 -> 피해 확산 -> 통제 실패" 구조로 재구성할 수 있다.
3. 사고 패턴을 Attack Surface Map과 연결해 우선순위를 조정할 수 있다.
4. 사건별 재발 방지 통제를 ACM/FFRS 항목으로 작성할 수 있다.
5. 시험 시나리오에서 유사 패턴을 빠르게 매칭할 수 있다.

---

## 3) 리플레이 방법론 (Associate 표준)

모든 사건은 아래 5단계로 정리한다.

1. **전제 조건**: 공격이 성립하기 위해 필요했던 구조
2. **트리거**: 공격 시작 이벤트 또는 권한 획득
3. **공격 체인**: 자금/권한이 이동한 순서
4. **피해 확산**: 어떤 자산과 사용자에게 영향이 전파됐는지
5. **통제 실패 지점**: 어떤 통제가 없었거나 무력화됐는지

---

## 4) Case A: Bridge 사고 리플레이 (검증자/검증로직 실패형)

### A-1. 전제 조건

- 브릿지가 멀티시그/검증자 네트워크에 의존
- 검증 임계값이 특정 운영 집단에 과집중
- 검증 메시지 무결성 확인 로직이 불완전하거나 운영 통제가 약함

### A-2. 공격 체인 (요약)

```
검증자 키 탈취 또는 검증로직 우회
  -> 정상 잠금 없이 출금/민트 승인 메시지 생성
  -> 목적 체인에서 자산 인출 또는 래핑 토큰 발행
  -> 원본 담보와 발행량 정합성 붕괴
```

### A-3. 통제 실패 포인트

- 검증자 독립성 부족
- 키 운영 보안 미흡
- 이상 발행/출금 탐지 지연

### A-4. 재발 방지 (Associate 수준)

- 검증자 분산도 기준 수립 (조직/인프라 분리)
- 임계값 조정 + 긴급 중지 조건 명확화
- 발행-잠금 정합성 모니터링 상시화

---

## 5) Case B: Oracle 사고 리플레이 (가격 참조 오염형)

### B-1. 전제 조건

- 단일 가격 소스 의존 또는 짧은 관측 구간
- 유동성 얕은 시장 가격이 즉시 반영
- 대출/청산 로직이 가격 급변 방어 장치 부족

### B-2. 공격 체인 (요약)

```
플래시 유동성 동원
  -> 참조 시장 가격 급변 유도
  -> 오염된 가격이 담보평가/청산판정에 반영
  -> 과대 대출/오청산/담보 탈취
```

### B-3. 통제 실패 포인트

- TWAP/다중 오라클 부재
- 가격 편차 상한, 회로차단기(circuit breaker) 부재
- 비정상 거래량 탐지/차단 미흡

### B-4. 재발 방지 (Associate 수준)

- 다중 소스 + 시간가중 가격 적용
- 가격 편차 초과 시 대출/청산 일시 제한
- 고위험 자산에 보수적 LTV 적용

---

## 6) Case C: Approval 사고 리플레이 (권한 남용형)

### C-1. 전제 조건

- 사용자 다수가 무한 approve를 허용
- spender 컨트랙트가 업그레이드 가능 또는 침해 노출
- revoke 안내/경고 UX 부족

### C-2. 공격 체인 (요약)

```
무한 approve 상태 사용자 다수 확보
  -> spender 권한 탈취(업그레이드/키침해/악성 코드)
  -> transferFrom 연쇄 실행
  -> 사용자 지갑 자산 대량 인출
```

### C-3. 통제 실패 포인트

- 최소 권한 원칙 미적용
- spender 권한 변경 감시 부재
- 사용자 위험 고지 및 취소 경로 안내 부족

### C-4. 재발 방지 (Associate 수준)

- 정액 approve 기본값화, 만료 기반 permit 권장
- spender 변경/업그레이드 알림 체계 도입
- revoke 가이드 및 위험 경고 UI 의무화

---

## 7) 패턴 통합: 세 사건의 공통점

| 축 | Bridge | Oracle | Approval |
|---|---|---|---|
| 신뢰 가정 붕괴 지점 | 검증자/검증로직 | 가격 소스/반영로직 | spender 권한/운영권한 |
| 즉시 피해 매커니즘 | 무담보 발행/인출 | 오평가 기반 대출/청산 | allowance 기반 연쇄 인출 |
| 공통 통제 키워드 | 분산, 검증, 모니터링 | 다중화, 지연, 차단 | 최소권한, 가시성, 취소가능성 |

핵심:
> 세 사건은 기술 스택이 달라도 "신뢰 가정이 깨진 경계"에서 시작된다.

## 8) 산출물 연결 예시

### A. ACM 입력 예시

| 역할 | 권한 | 통제수준 | 취약점 | 개선방향 |
|---|---|---|---|---|
| Bridge Admin | validator set 변경 | 중 | 검증자 집중 | 다중 조직 분산 + 변경 Timelock |
| Protocol Admin | oracle source 변경 | 하 | 단일 오라클 의존 | 다중 소스 + 변경 승인 이중화 |
| Spender Contract Admin | spender logic 업그레이드 | 하 | approve 남용 위험 | 정액 approve 기본 + 업그레이드 공지 |

### B. FFRS 입력 예시

| 흐름구간 | 리스크유형 | 심각도 | 개선방향 |
|---|---|---|---|
| Bridge mint 구간 | 무담보 발행 | High | 발행-잠금 정합성 알림 및 자동 중지 |
| Oracle 반영 구간 | 가격 오염 | High | TWAP + 편차 임계치 초과 시 지연 |
| 사용자 진입 구간 | 무한 approve 노출 | Medium~High | 정액 approve 및 revoke UX |

## 9) 시험 대응용 60초 패턴 매칭

시나리오를 받으면 아래 순서로 본다.

1. 신뢰 가정이 어디에 모여 있는가? (검증자/오라클/spender)
2. 그 가정이 깨지면 즉시 어떤 함수가 악용되는가?
3. 자금은 어떤 경로로 빠져나가는가?
4. 기존 통제에서 빈 구멍이 무엇인가?
5. ACM/FFRS에 어떤 항목으로 기록할 것인가?

---

## 용어 보충 (Quick Reference)

- **사고 리플레이**: 실제 사건의 공격 경로를 단계별로 재구성하는 분석 방식.
- **정합성 붕괴**: 시스템이 가정한 1:1 대응/검증 일치가 깨진 상태.
- **가격 오염**: 참조 가격이 조작 또는 비정상 변동으로 신뢰를 잃은 상태.
- **권한 남용**: 정상 기능의 권한 범위를 공격자가 악용하는 상태.

## 헷갈리는 개념 보충

- "과거 사건과 동일한 기술"이 없어도 동일한 패턴은 재발한다.
- 사건 이름을 모르는 것은 감점 포인트가 아니다. 공격 체인 재구성이 핵심이다.
- Associate 답안에서는 포렌식 디테일보다 통제 실패 지점과 개선 방향의 명확성이 중요하다.
