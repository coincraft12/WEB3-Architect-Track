# 16강. Authority Control Matrix 정의 및 작성 기준: 권한 통제 설계의 최종 산출물

## 1) 이 강의의 한 문장
**Authority Control Matrix(ACM)는 서비스의 역할, 보유 권한, 통제 수준, 식별된 취약점, 개선 방향을 하나의 구조화된 표로 정리하는 실기 산출물이다. ACM을 완성하면 서비스의 권한 구조를 한눈에 평가하고 개선 우선순위를 결정할 수 있다.**

## 2) 강의 목표 (수강생이 끝나고 할 수 있어야 하는 것)
이 강의를 마치면 수강생은:

1. Authority Control Matrix의 구조와 각 컬럼의 의미를 설명할 수 있다.
2. 주어진 서비스 정보에서 ACM을 작성할 수 있다.
3. 통제 수준을 평가하는 기준을 적용할 수 있다.
4. ACM을 통해 개선 우선순위를 결정하는 방법을 말할 수 있다.
5. 실기 답안 형식에 맞게 ACM을 작성할 수 있다.

---

## 3) Authority Control Matrix란 무엇인가

### 정의
서비스 내 모든 역할과 그 역할이 보유한 권한을, 통제 수준·취약점·개선 방향과 함께 한 표에 정리한 문서.

### 목적
- 서비스의 권한 구조를 한눈에 파악
- 통제가 부족한 역할/권한을 식별
- 개선 우선순위 결정
- 실기 답안의 근거 문서

### ACM이 답해야 하는 질문
1. 이 서비스에는 어떤 역할이 있는가?
2. 각 역할은 어떤 권한을 갖는가?
3. 그 권한에 어떤 통제가 적용되어 있는가?
4. 통제가 부족하다면 어떤 취약점이 있는가?
5. 어떻게 개선해야 하는가?

## 4) ACM의 구조 (5컬럼 기본 형식)

| 역할 (Role) | 보유 권한 (Authority) | 통제 수준 (Control Level) | 취약점 (Vulnerability) | 개선 방향 (Improvement) |
|---|---|---|---|---|

### 각 컬럼 정의

**① 역할 (Role)**
- 서비스 내 권한을 갖는 주체의 분류
- 기술 방식: 역할명 + 구현 형태 + 주소 유형
- 예: `Admin (3-of-5 Multisig)`, `Operator (단일 EOA)`, `User (누구나)`

**② 보유 권한 (Authority)**
- 해당 역할이 실행할 수 있는 핵심 함수/기능 목록
- 기술 방식: 함수명 또는 기능 설명
- 예: `pause()`, `upgrade()`, `setLTV()`, `collectFees()`

**③ 통제 수준 (Control Level)**
- 해당 권한에 적용된 통제 메커니즘의 강도
- 평가 기준: 아래 통제 수준 등급 참조

**④ 취약점 (Vulnerability)**
- 현재 통제 수준이 부족한 경우의 리스크
- ch14의 4가지 유형 코드 참조 가능 (집중/Approve/업그레이드/미분리)
- 기술 방식: 취약점 유형 + 구체적 위험

**⑤ 개선 방향 (Improvement)**
- 취약점을 완화하기 위한 구체적인 통제 방안
- ch15의 메커니즘 선택 기준 적용
- 기술 방식: 적용할 메커니즘 + 구체적 파라미터

## 5) 통제 수준 등급 기준

| 등급 | 명칭 | 기준 | 색상 코드 |
|---|---|---|---|
| 1 | 미흡 | 단일 EOA, 통제 메커니즘 없음 | 적색 |
| 2 | 부족 | Multisig 없거나 임계값 낮음 (1-of-N 등) | 주황 |
| 3 | 보통 | Multisig 적용, Timelock 미적용 | 황색 |
| 4 | 양호 | Multisig + Timelock, RBAC 부분 적용 | 연녹색 |
| 5 | 우수 | Multisig + Timelock + RBAC + 거버넌스 연동 | 녹색 |

**통제 수준 평가 예시:**
```
Admin (단일 EOA, pause/upgrade 보유) → 1 (미흡)
Admin (3-of-5 Multisig, upgrade 즉시 실행) → 3 (보통)
Admin (3-of-5 Multisig, upgrade에 48h Timelock) → 4 (양호)
Operator (2-of-3 Multisig, 파라미터 조정만, Timelock) → 4 (양호)
Fee Collector (단일 EOA, collectFees만) → 2 (부족)
```

## 6) ACM 작성 절차

### Step 1: 역할 목록 추출
Architecture Read Sheet 섹션 4에서 역할 목록을 가져온다.
```
1. Admin: Gnosis Safe 3-of-5 Multisig
2. Risk Manager: 2-of-3 Multisig
3. Fee Collector: 단일 EOA
4. Liquidator: Public (누구나)
5. User: 일반 사용자
```

### Step 2: 역할별 권한 매핑
각 역할이 접근 가능한 함수/기능을 나열한다.
```
Admin: pause(), unpause(), upgrade(), setOracle(), grantRole()
Risk Manager: setLTV(), setLiquidationThreshold(), addAsset()
Fee Collector: collectFees()
Liquidator: liquidate(borrower)
User: deposit(), borrow(), repay(), withdraw()
```

### Step 3: 통제 수준 평가
각 역할+권한 조합에 통제 메커니즘을 확인하고 등급을 부여한다.
```
Admin (pause/upgrade): 3-of-5 Multisig, Timelock 없음 → 3 (보통)
Risk Manager (파라미터): 2-of-3 Multisig, Timelock 없음 → 3 (보통)
Fee Collector (collectFees): 단일 EOA → 1 (미흡)
```

### Step 4: 취약점 기술
통제가 부족한 부분의 취약점을 ch14 유형을 참조하여 기술한다.

### Step 5: 개선 방향 제시
ch15 메커니즘 선택 기준을 적용하여 구체적인 개선 방향을 작성한다.

## 7) ACM 작성 예시 (Lending 프로토콜)

**서비스 정보:**
- 서비스 유형: Lending Protocol
- 주요 역할: Admin (3-of-5 Multisig), Risk Manager (2-of-3 Multisig), Fee Collector (단일 EOA), Liquidator (Public), User
- 특이 사항: Proxy 구조 사용, Timelock 없음, 외부 Chainlink Oracle 의존

**ACM:**

| 역할 | 보유 권한 | 통제 수준 | 취약점 | 개선 방향 |
|---|---|---|---|---|
| Admin (3-of-5 Multisig) | pause(), unpause(), upgrade(), setOracle(), grantRole() | 3 (보통) | [업그레이드 권한 집중] Timelock 없어 즉시 업그레이드 가능. Multisig 탈취 시 악성 로직 즉시 배포 가능 | upgrade()에 48시간 Timelock 적용. UPGRADER_ROLE 분리 (4-of-7 Multisig) 검토 |
| Admin (3-of-5 Multisig) | setOracle() | 3 (보통) | [단일 관리자 집중] 잘못된 Oracle 주소 설정 시 전체 프로토콜 가격 오류 | setOracle()에 24시간 Timelock 적용. 변경 전 커뮤니티 사전 공지 절차 수립 |
| Risk Manager (2-of-3 Multisig) | setLTV(), setLiquidationThreshold(), addAsset() | 3 (보통) | [역할 미분리] LTV 변경이 즉시 적용되어 사용자 포지션에 즉각 영향 | LTV 변경에 24시간 Timelock 적용. 변경 폭 제한 (±5% 이내) 파라미터 거버넌스 도입 |
| Fee Collector (단일 EOA) | collectFees() → Treasury | 1 (미흡) | [단일 관리자 집중] 단일 키 탈취 시 수수료 흐름 전체 통제 가능. 수수료 목적지 변경 가능성 | Fee Collector를 2-of-3 Multisig로 전환. collectFees() 목적지 하드코딩 또는 변경 시 Admin 승인 요구 |
| Liquidator (Public) | liquidate(borrower) | 5 (우수) | 없음 (공개 함수, 경제적 인센티브 기반) | 유지. 단, 청산 플래시론 악용 방지 로직 확인 필요 |
| User | deposit(), borrow(), repay(), withdraw() | 4 (양호) | [무한 Approve] 서비스 진입 시 무한 Approve 요구 가능성 확인 필요 | Approve 금액을 트랜잭션 단위로 제한. UI에서 Approve 금액 명시 |

## 8) ACM 기반 개선 우선순위 결정

ACM 작성 후 통제 수준 1~2인 항목을 우선 개선 대상으로 선정한다.

**우선순위 결정 매트릭스:**

| 통제 수준 | 권한의 영향도 | 우선순위 |
|---|---|---|
| 1 (미흡) | 자산 이동 / 업그레이드 | 즉시 개선 (P0) |
| 1 (미흡) | 파라미터 조정 / 수수료 | 단기 개선 (P1) |
| 2 (부족) | 자산 이동 / 업그레이드 | 단기 개선 (P1) |
| 2 (부족) | 파라미터 조정 | 중기 개선 (P2) |
| 3 (보통) | 업그레이드 포함 | 중기 개선 (P2) |
| 3 (보통) | 파라미터 조정 | 장기 개선 (P3) |
| 4~5 (양호~우수) | 모든 유형 | 모니터링 유지 |

**예시 우선순위 결정:**
```
P0 (즉시): Fee Collector 단일 EOA → Multisig 전환
P1 (단기): Admin upgrade() → Timelock 48시간 적용
P2 (중기): Risk Manager LTV 변경 → Timelock 24시간 적용
P3 (장기): UPGRADER_ROLE 분리 → 별도 고임계값 Multisig
```

## 9) 실기 답안 작성 가이드

### 주어지는 것
- 서비스 개요 (유형, 주요 기능)
- 역할 및 권한 정보
- 계약 구조 (Proxy 여부, 외부 의존성)

### 작성해야 하는 것
1. ACM 표 작성 (역할 × 권한 × 통제 수준 × 취약점 × 개선 방향)
2. 개선 우선순위 목록 (P0~P3)
3. 핵심 취약점 2~3개에 대한 개선 방향 상세 서술

### 시간 배분 전략
```
읽기 및 역할 파악: 5분
역할 × 권한 매핑: 10분
통제 수준 평가: 5분
취약점 및 개선 방향: 15분
우선순위 결정: 5분
검토: 5분
총 45분
```

### 자주 하는 실수
- 역할 목록 불완전: 외부 계약(Oracle, Bridge)도 역할로 포함
- 취약점 없음 표기 오류: 통제 수준 4~5여도 취약점이 0은 아닐 수 있음
- 개선 방향 너무 추상적: "Multisig 도입" → "3-of-5 Multisig로 전환"처럼 구체적으로
- 우선순위 기준 없이 나열: 영향도와 통제 수준을 함께 고려하여 근거 제시

## 10) ACM과 다른 산출물의 관계

```
Architecture Read Sheet (Module 1 산출물)
    │
    ├─ 섹션 4 (권한·책임 구조) ──→ ACM 역할/권한 컬럼의 입력
    └─ 섹션 5 (취약점 목록) ──────→ ACM 취약점 컬럼의 입력

                 ↓
    Authority Control Matrix (Module 2 실기 산출물)
                 │
                 └─→ 개선 방향이 Module 3 리스크 평가로 연결
                       (권한 구조 개선이 자금 흐름 리스크에 미치는 영향)
```

## 용어 보충 (Quick Reference)
- **Authority Control Matrix (ACM)**: 역할, 권한, 통제 수준, 취약점, 개선 방향을 통합한 구조화된 평가 문서.
- **통제 수준 (Control Level)**: 특정 권한에 적용된 보안 메커니즘의 강도. 1(미흡)~5(우수) 5단계로 평가.
- **개선 우선순위 (Improvement Priority)**: 통제 수준과 권한 영향도를 종합하여 먼저 개선해야 할 항목의 순서.
- **P0~P3**: 우선순위 코드. P0=즉시, P1=단기(1~4주), P2=중기(1~3개월), P3=장기(3개월 이상).

## 헷갈리는 개념 보충
- ACM의 각 행은 역할 단위가 아니라 "역할+권한 그룹" 단위일 수 있다. Admin이 pause()와 upgrade()를 가진다면, 두 권한의 통제 수준이 다를 경우 행을 분리할 수 있다.
- 통제 수준 5(우수)도 개선 방향이 없을 수 있지만, "유지 및 모니터링"을 명시하는 것이 완성된 답안이다.
- ACM은 현황 분석과 개선 방향을 동시에 담는다. "현재 상태"와 "목표 상태"를 명확히 구분하여 작성한다.
