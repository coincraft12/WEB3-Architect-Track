# 20강. Fund Flow Risk Sheet 정의 및 작성 기준: 자금 흐름 기반 리스크 평가의 최종 산출물

## 1) 이 강의의 한 문장
**Fund Flow Risk Sheet(FFRS)는 서비스의 자금 흐름 구간별로 리스크 유형, 심각도, 개선 방향을 정리한 실기 산출물이다. FFRS를 완성하면 자금 흐름의 어느 구간이 가장 취약한지를 구조적으로 제시할 수 있다.**

## 2) 강의 목표 (수강생이 끝나고 할 수 있어야 하는 것)
이 강의를 마치면 수강생은:

1. Fund Flow Risk Sheet의 구조와 각 컬럼의 의미를 설명할 수 있다.
2. ch17~ch19의 분석 결과를 FFRS에 통합하는 방법을 말할 수 있다.
3. 주어진 서비스 정보에서 FFRS를 작성할 수 있다.
4. FFRS 기반으로 개선 우선순위를 결정하는 방법을 말할 수 있다.
5. 실기 답안 형식에 맞게 FFRS를 작성할 수 있다.

---

## 3) Fund Flow Risk Sheet란 무엇인가

### 정의
자금이 이동하는 각 구간에서 발생 가능한 리스크의 유형, 심각도, 개선 방향을 체계적으로 정리한 문서.

### 목적
- 자금 흐름의 어느 구간에 리스크가 집중되어 있는지 파악
- 구간별 리스크 우선순위 결정
- 개선 방향의 근거 문서
- 실기 답안의 핵심 산출물

### FFRS가 답해야 하는 질문
1. 자금 흐름에서 어떤 구간이 존재하는가?
2. 각 구간에서 어떤 리스크가 발생할 수 있는가?
3. 그 리스크의 심각도는 어느 수준인가?
4. 어떻게 개선해야 하는가?
5. 어떤 구간을 먼저 개선해야 하는가?

---

## 4) FFRS의 구조 (5컬럼 기본 형식)

| 흐름 구간 (Flow Segment) | 리스크 유형 (Risk Type) | 심각도 (Severity) | 발생가능성/영향도 | 개선 방향 (Improvement) |
|---|---|---|---|---|

### 각 컬럼 정의

**① 흐름 구간 (Flow Segment)**
- ch17에서 추출한 자금 흐름의 단계별 구간
- 기술 방식: 시작점 → 도착점 형식
- 예: `사용자 → Lending Pool (입금)`, `Pool → Oracle (가격 조회)`

**② 리스크 유형 (Risk Type)**
- ch18에서 분류한 리스크 유형
- 기술 방식: 유형명 + 구체적 내용
- 예: `Oracle 조작: 단일 Oracle 소스 조작 시 가격 왜곡`

**③ 심각도 (Severity)**
- ch19의 심각도 매트릭스 결과
- 기술 방식: 등급 숫자 + 등급명
- 예: `4 (High)`, `3 (Medium)`

**④ 발생가능성/영향도**
- 심각도의 근거를 명시하는 보조 컬럼
- 기술 방식: L×I 형식
- 예: `M×H`, `L×M`

**⑤ 개선 방향 (Improvement)**
- ch19에서 도출한 리스크별 대응 방안
- 기술 방식: 핵심 개선 조치 요약 (상세 내용은 별도 서술)
- 예: `가격 소스 단일 의존 완화 검토`

## 5) FFRS 작성 절차

### Step 1: 자금 흐름 구간 목록 작성 (ch17 연결)
```
1단계: 입금 경로에서 구간 추출
  구간 01: 사용자 → Service Contract (입금)

2단계: 보관 경로에서 구간 추출
  구간 02: Service Contract → External Protocol (재예치)
  구간 03: External Protocol → Service Contract (이자 회수)
  구간 04: Service Contract → Oracle (가격 조회)

3단계: 출금 경로에서 구간 추출
  구간 05: Service Contract → 사용자 (출금)
  구간 06: Liquidator → Service Contract (청산)

4단계: 수수료 흐름에서 구간 추출
  구간 07: Service Contract → Treasury (수수료 이동)
```

### Step 2: 구간별 리스크 유형 매핑 (ch18 연결)
```
구간 01 (입금): 무한 Approve 리스크
구간 02 (재예치): 외부 의존성 리스크
구간 03 (이자 회수): 재진입 공격 가능성
구간 04 (Oracle): Oracle 조작, 플래시론 공격
구간 05 (출금): 재진입 공격, 유동성 부족
구간 06 (청산): Oracle 조작, 연쇄 청산
구간 07 (수수료): 권한 집중 리스크
```

### Step 3: 심각도 평가 (ch19 연결)
각 구간의 리스크에 발생가능성×영향도 매트릭스 적용.

### Step 4: 개선 방향 작성
심각도에 따른 개선 방향 작성. 우선순위 순으로 정렬.

## 6) FFRS 작성 예시 (Lending Protocol)

**서비스 정보:**
- 서비스 유형: Lending Protocol
- 주요 기능: 담보 예치, 대출, 청산
- 특이 사항: 단일 Chainlink Oracle, 외부 Yield 재예치 없음, Admin 3-of-5 Multisig

**자금 흐름 구간 (먼저 ch17 방식으로 추출):**
```
[입금]   사용자 → Lending Pool (담보 예치)
[평가]   Lending Pool → Chainlink Oracle (담보 가격 조회)
[대출]   Lending Pool → 차입자 (USDC 대출 실행)
[이자]   이자 누적 → Lending Pool 잔액 증가
[청산]   Liquidator → Lending Pool (청산 실행)
[상환]   차입자 → Lending Pool (부채 상환)
[출금]   Lending Pool → 공급자 (공급 자산 회수)
[수수료] Lending Pool → Treasury (수수료 분배)
```

**FFRS:**

| 흐름 구간 | 리스크 유형 | 심각도 | L×I | 개선 방향 |
|---|---|---|---|---|
| 사용자 → Lending Pool (담보 예치) | 무한 Approve: 컨트랙트 침해 시 사용자 지갑 전체 노출 | 3 (Medium) | M×M | 트랜잭션 단위 정확한 금액 Approve 정책. ERC-2612 Permit 도입 검토 |
| Lending Pool → Oracle (가격 조회) | Oracle 단일 소스: Chainlink 단독 의존, 장애·조작 시 가격 오류 | 4 (High) | M×H | 가격 소스 다변화 또는 이상값 완화 장치 검토 |
| Lending Pool → Oracle (가격 조회) | 플래시론 공격: 순간 가격 조작 후 과잉 대출 가능성 | 3 (Medium) | L×H | 순간 가격 영향 완화 장치 검토 |
| Lending Pool → 차입자 (대출 실행) | 과잉 대출: LTV 설정 오류로 담보 대비 과도한 대출 | 2 (Low) | L×M | LTV 변경에 24시간 Timelock + Risk Manager 다중 서명 적용 |
| Liquidator → Lending Pool (청산) | 연쇄 청산: 담보 급락 시 청산 연쇄 반응으로 추가 가격 하락 | 3 (Medium) | M×M | 담보 기준과 청산 조건의 보수성 재점검 |
| Liquidator → Lending Pool (청산) | 청산 부족: 청산자 없거나 지연으로 담보 부족 상태 지속 | 3 (Medium) | M×M | 청산 참여 유인과 대응 절차 점검 |
| Lending Pool → 공급자 (출금) | 재진입 공격: 출금 함수에서 상태 업데이트 전 외부 호출 가능성 | 3 (Medium) | L×H | 외부 호출 순서와 재진입 방지 통제 확인 |
| Lending Pool → 공급자 (출금) | 유동성 부족: 대출 이용률 100% 시 즉시 출금 불가 | 2 (Low) | M×L | 유동성 한도와 경보 기준 재점검 |
| Lending Pool → Treasury (수수료) | 수수료 중앙화: Fee Collector 단일 EOA 집중 | 4 (High) | H×M | Fee Collector를 2-of-3 Multisig로 전환. collectFees() 목적지 하드코딩 |

## 7) FFRS 기반 개선 우선순위

FFRS 작성 후 심각도 순으로 우선순위를 결정한다.

**우선순위 목록 (예시):**

```
P1 (단기 개선, 1~4주):
- Oracle 단일 소스 의존 (심각도 4): 가격 소스 다변화 또는 이상값 완화 장치 검토
- 수수료 중앙화 (심각도 4): Fee Collector Multisig 전환

P2 (중기 개선, 1~3개월):
- 무한 Approve (심각도 3): Permit 방식 도입 검토
- 플래시론 공격 (심각도 3): 순간 가격 영향 완화 장치 검토
- 연쇄 청산 (심각도 3): 담보 기준과 청산 조건 재점검
- 재진입 공격 (심각도 3): 코드 수준 통제 재확인

P3 (장기 개선, 3개월 이상):
- 과잉 대출 (심각도 2): 파라미터 변경 절차 정비
- 유동성 부족 (심각도 2): 유동성 관리 기준 고도화
```

## 8) FFRS vs. Authority Control Matrix 비교

| 구분 | Authority Control Matrix (ACM) | Fund Flow Risk Sheet (FFRS) |
|---|---|---|
| 분석 대상 | 권한 구조 (누가 무엇을 할 수 있는가) | 자금 흐름 (자금이 어떻게 이동하는가) |
| 행(Row) 단위 | 역할 × 권한 | 흐름 구간 |
| 핵심 컬럼 | 통제 수준, 취약점 유형 | 리스크 유형, 심각도 (L×I) |
| 개선 방향 근거 | ch14~ch15 (권한 통제 메커니즘) | ch18~ch19 (자금 흐름 리스크 + 심각도) |
| 산출물 모듈 | Module 2 (실기 1: 권한 통제 설계) | Module 3 (실기 2: 자금 흐름 리스크 평가) |

두 산출물은 독립적이지 않다.
ACM에서 발견된 권한 취약점이 FFRS의 특정 구간에서 리스크로 나타난다.

## 9) 실기 답안 작성 가이드

### 주어지는 것
- 서비스 개요 및 자금 흐름 정보
- 역할 및 권한 정보
- 외부 의존성 정보

### 작성해야 하는 것
1. 자금 흐름 다이어그램 (ch17 방식)
2. FFRS 표 작성 (구간 × 리스크 × 심각도 × 개선 방향)
3. 개선 우선순위 목록 (P1~P3)
4. 핵심 리스크 2~3개에 대한 개선 방향 상세 서술

### 시간 배분 전략
```
읽기 및 구간 파악: 5분
흐름 다이어그램 작성: 10분
구간별 리스크 매핑: 10분
심각도 평가: 5분
개선 방향 작성: 15분
우선순위 결정 및 검토: 5분
총 50분
```

### 자주 하는 실수
- 구간 누락: 수수료 흐름, 외부 의존성 구간 빠뜨리기
- 심각도 근거 없이 숫자만 기입: L×I를 명시하지 않으면 설득력 저하
- 개선 방향 추상적: "Oracle 개선 필요" 대신 "가격 소스 다변화 검토"처럼 리스크와 통제 방향이 연결되게 작성
- 권한 구조 리스크를 FFRS에 포함하지 않음: 수수료 집중, Admin 권한 등 권한 리스크도 자금 흐름 구간에 반영

## 10) FFRS와 다른 산출물의 관계

```
ch17 자금 흐름 추적
    └─ 흐름 구간 목록 → FFRS 1열 입력

ch18 리스크 유형 분류
    └─ 구간별 리스크 유형 → FFRS 2열 입력

ch19 리스크 평가 기준
    └─ 심각도 매트릭스 결과 → FFRS 3~4열 입력
    └─ 개선 방향 → FFRS 5열 입력

           ↓

Fund Flow Risk Sheet (Module 3 실기 산출물)

           ↓

ch21 종합 시나리오 분석
    FFRS + ACM + Architecture Read Sheet 통합 적용
```

## 용어 보충 (Quick Reference)
- **Fund Flow Risk Sheet (FFRS)**: 자금 흐름 구간별로 리스크 유형, 심각도, 개선 방향을 정리한 실기 산출물.
- **흐름 구간 (Flow Segment)**: 자금이 이동하는 각 단계를 시작점→도착점으로 표현한 단위. FFRS의 기본 분석 단위.
- **서킷 브레이커 (Circuit Breaker)**: 이상 상황 감지 시 관련 기능을 중단하는 메커니즘. Associate에서는 필요성 식별 수준으로 이해한다.
- **Permit (ERC-2612)**: Approve 대신 서명으로 토큰 사용 권한을 부여하는 표준. 무한 Approve 리스크의 개선 방법.

## 헷갈리는 개념 보충
- FFRS의 구간은 기술적 컨트랙트 단위가 아니라 "자금 이동의 의미 있는 단계"로 정의한다. 동일한 컨트랙트 내에서도 의미가 다른 이동이라면 별도 구간으로 분리한다.
- 심각도가 낮다고 개선 방향이 없는 것은 아니다. 낮은 심각도에도 "모니터링 유지" 또는 "장기 개선"을 명시하면 완성도 있는 답안이 된다.
- FFRS와 ACM은 서로 다른 관점에서 같은 서비스를 분석한다. 두 산출물에서 동일한 취약점이 다른 형식으로 나타날 수 있으며, 이는 오류가 아니라 취약점의 다면적 분석을 보여주는 것이다.
