# 4강. Proxy와 Upgrade 패턴: 불변 컨트랙트를 바꾸는 구조의 원리와 위험

## 1) 이 강의의 한 문장
**스마트컨트랙트는 한번 배포되면 코드를 바꿀 수 없다. Proxy 패턴은 이 불변성을 우회해 로직을 교체할 수 있게 만드는 구조이며, 그 업그레이드 권한은 WEB3에서 가장 강력하고 위험한 권한 중 하나다.**

## 2) 강의 목표 (수강생이 끝나고 할 수 있어야 하는 것)
이 강의를 마치면 수강생은:

1. 스마트컨트랙트의 불변성이 왜 문제가 되는지 설명할 수 있다.
2. Proxy 패턴의 기본 원리(로직과 스토리지 분리)를 말할 수 있다.
3. Transparent Proxy와 UUPS의 구조적 차이를 개요 수준에서 구분할 수 있다.
4. 업그레이드 권한이 왜 가장 민감한 권한인지 설명할 수 있다.
5. Proxy 구조가 있는 서비스를 볼 때 확인해야 할 것들을 말할 수 있다.

---

## 3) 문제에서 시작: 컨트랙트는 코드를 바꿀 수 없다

Basic 과정에서 배운 스마트컨트랙트의 핵심 속성 중 하나는 **불변성(Immutability)**이다.

컨트랙트가 한번 블록체인에 배포되면, 그 코드는 변경되지 않는다.
이것은 신뢰의 근거였다.

> "코드가 규칙이다. 아무도 그 규칙을 몰래 바꾸지 못한다."

하지만 현실에서는 문제가 생긴다.

- 배포 후 버그가 발견됐다.
- 보안 취약점이 발견됐다.
- 사용자 요구나 시장 환경이 바뀌어 로직 개선이 필요하다.
- 규제 요건이 추가됐다.

불변 컨트랙트는 이 모든 상황에서 손을 쓸 수 없다.
심각한 버그가 있어도 "코드가 규칙"이라는 이유로 그대로 실행된다.

이 문제를 해결하기 위해 나온 것이 **Proxy 패턴**이다.

---

## 4) Proxy 패턴의 기본 원리: 로직과 스토리지 분리

Proxy 패턴의 핵심 아이디어는 단순하다.

> **사용자가 호출하는 주소(Proxy)와 실제 로직이 있는 주소(Implementation)를 분리한다.**

```
사용자
  ↓
Proxy 컨트랙트 (주소 고정, 스토리지 보관)
  ↓ (delegatecall)
Implementation 컨트랙트 (로직만 있음, 교체 가능)
```

각 역할:

**Proxy 컨트랙트**
- 사용자가 실제로 상호작용하는 주소다. 이 주소는 바뀌지 않는다.
- 자산 잔액, 사용자 데이터 등 **상태(스토리지)**를 보관한다.
- 호출을 받으면 Implementation에게 위임(delegatecall)한다.

**Implementation 컨트랙트 (로직 컨트랙트)**
- 실제 비즈니스 로직(함수들)이 여기 있다.
- 이 컨트랙트는 교체될 수 있다. 교체해도 Proxy의 주소와 스토리지는 그대로다.
- 버그 수정, 기능 추가가 필요하면 새 Implementation을 배포하고 Proxy가 가리키는 주소만 바꾼다.

핵심 문장:
> **사용자는 Proxy 주소를 계속 쓴다. Proxy 뒤에서 로직이 조용히 교체된다.**

---

## 5) delegatecall: Proxy가 작동하는 방식

Proxy가 Implementation에게 위임하는 방식은 `delegatecall`이라는 특수한 호출 방식이다.

일반 call vs. delegatecall의 차이:

| 구분 | 일반 call | delegatecall |
|---|---|---|
| 코드 실행 위치 | 호출받은 컨트랙트 | 호출받은 컨트랙트 |
| 스토리지 변경 위치 | 호출받은 컨트랙트 | **호출한 컨트랙트 (Proxy)** |
| msg.sender | 호출한 컨트랙트 주소 | **원래 사용자 주소** |

delegatecall은 "Implementation의 코드를 Proxy의 컨텍스트에서 실행한다"는 뜻이다.
그래서 Implementation의 로직이 실행되지만, 상태 변화는 Proxy에 기록된다.

설계자 포인트:
- delegatecall의 이 특성 때문에 Proxy와 Implementation의 **스토리지 레이아웃이 일치해야 한다**.
- 업그레이드 시 스토리지 구조가 바뀌면 기존 데이터가 깨진다. 이것이 업그레이드의 핵심 위험 중 하나다.

Associate 수준에서 delegatecall의 작동 원리를 코드로 이해할 필요는 없다.
다만 "Proxy를 통해 로직이 실행될 때 스토리지는 Proxy에 남는다"는 구조적 사실을 알아야 한다.

## 6) 주요 Proxy 패턴 두 가지

Proxy 패턴에는 여러 변형이 있다. 가장 많이 사용되는 두 가지를 구조 관점에서 본다.

### A. Transparent Proxy

```
사용자 → Proxy → Implementation
Admin  → Proxy (직접 관리 함수 호출)
```

특징:
- 업그레이드 함수가 **Proxy 자체**에 있다.
- Admin 주소와 일반 사용자 주소를 구분해서, Admin은 Proxy 관리 함수를, 사용자는 Implementation 로직을 호출한다.
- 이 구분이 없으면 "function selector clash" 문제가 생길 수 있어, Proxy가 호출자를 보고 라우팅한다.

장점: 구조가 명확하다.
단점: Proxy 자체가 무거워진다. Gas 비용이 높다.

### B. UUPS (Universal Upgradeable Proxy Standard)

```
사용자 → Proxy → Implementation (업그레이드 함수 포함)
```

특징:
- 업그레이드 함수가 **Implementation** 안에 있다.
- Proxy 자체는 매우 단순하다.
- 업그레이드는 Implementation의 함수를 통해 실행된다.

장점: Proxy가 가볍다. Gas 효율이 좋다.
단점: 새 Implementation에 업그레이드 함수가 없으면 영구적으로 업그레이드 불가 상태가 된다.

설계자 포인트:
- 어떤 패턴을 쓰는지보다, **업그레이드 권한이 어떻게 보호되고 있는지**가 더 중요하다.
- 패턴 자체가 안전한 게 아니라, 그 패턴 위에 누가 어떤 권한으로 업그레이드를 실행하는지가 안전을 결정한다.

## 7) 업그레이드 권한: 왜 가장 위험한 권한인가

ch3에서 역할 목록을 보면 **Upgrader**가 특별 주의 대상이었다.
이유가 여기 있다.

업그레이드 권한은 단순히 "코드를 바꾸는" 권한이 아니다.

**업그레이드로 할 수 있는 것:**
- 컨트랙트의 모든 로직을 교체한다.
- 수수료 계산 방식을 바꾼다.
- 출금 조건을 바꾼다.
- 기존에 없던 함수를 추가해 자금을 빼낼 수 있는 경로를 만든다.
- 사용자 자산을 이동시키는 로직을 삽입한다.

핵심 문장:
> **업그레이드 권한을 가진 사람은 이론상 컨트랙트의 모든 약속을 뒤집을 수 있다.**

이것이 Timelock과 가장 강한 Multisig가 Upgrader 역할에 결합되어야 하는 이유다.

## 8) 업그레이드 권한의 통제 패턴

업그레이드 권한을 어떻게 통제하는가에 따라 서비스의 신뢰 수준이 달라진다.

### 패턴 1: 단일 Admin 업그레이드 (가장 위험)
- 단일 키가 즉시 업그레이드 가능.
- "탈중앙화" 서비스라고 표방해도 사실상 운영자가 모든 것을 바꿀 수 있다.

### 패턴 2: Multisig 업그레이드 (기본 수준)
- M명의 합의가 있어야 업그레이드 가능.
- 단독 행동은 막지만, 즉시 실행은 가능하다.

### 패턴 3: Multisig + Timelock 업그레이드 (권장 수준)
- M명의 합의 후 지연 기간이 지나야 실행.
- 업그레이드 내용이 공개되고 사용자가 반응할 시간이 생긴다.

### 패턴 4: DAO 거버넌스 업그레이드
- 토큰 보유자 투표로 업그레이드 결정.
- 투표 → Timelock → 실행 흐름.
- 가장 분산된 구조이나, 거버넌스 토큰 집중 여부에 따라 실질이 달라진다.

### 패턴 5: 업그레이드 권한 포기 (Immutable)
- 업그레이드 권한 자체를 제거한다.
- 버그 수정 불가, 기능 추가 불가.
- 불변성을 가장 강하게 보장하지만, 운영 유연성이 없다.

설계자 포인트:
- 어떤 패턴도 절대적으로 좋거나 나쁘지 않다.
- 서비스의 성격, 자금 규모, 사용자 기대에 따라 적합한 패턴이 다르다.
- 중요한 것은 **선택한 패턴을 명시적으로 공개하고 있는가**다.

## 9) Proxy 구조의 추가 위험 요소

업그레이드 권한 외에도 Proxy 구조가 만드는 위험이 있다.

### 스토리지 충돌 (Storage Collision)
Proxy와 Implementation이 같은 스토리지 슬롯을 다른 목적으로 사용하면 데이터가 덮어써진다.
업그레이드 시 스토리지 레이아웃을 잘못 변경하면 기존 사용자 데이터가 손상될 수 있다.

### 초기화 문제 (Initialization)
일반 컨트랙트의 `constructor`는 Proxy 패턴에서 제대로 작동하지 않는다.
대신 `initialize` 함수를 사용하는데, 이 함수가 중복 실행될 수 있는 취약점이 존재한다.
"초기화 함수가 누구나 다시 호출할 수 있는 상태"는 심각한 취약점이다.

### 로직 컨트랙트 직접 초기화
Implementation 컨트랙트 자체가 초기화되지 않은 채로 배포되면,
공격자가 직접 그 컨트랙트의 초기화 함수를 호출해 소유권을 탈취할 수 있다.

Associate 수준에서 이 취약점들의 코드를 분석할 필요는 없다.
다만 "Proxy 구조에는 업그레이드 권한 외에도 스토리지와 초기화 관련 위험이 있다"는 사실을 알아야 한다.

## 10) Proxy가 없는 서비스의 의미

모든 서비스가 Proxy 패턴을 쓰는 것은 아니다.

Proxy 없이 불변 컨트랙트로 운영하는 서비스는:
- "코드를 바꾸지 않겠다"는 강한 신뢰 신호를 줄 수 있다.
- 버그 발견 시 대응이 불가능한 위험을 감수한다.
- 일부 DeFi 프로토콜(초기 Uniswap 등)은 불변 컨트랙트를 선택했다.

Proxy 패턴을 쓰는 서비스는:
- 유연성을 가지지만, 업그레이드 권한이라는 신뢰 부채를 진다.
- "운영자를 얼마나 신뢰하는가"라는 질문이 항상 따라온다.

설계자 포인트:
- Proxy 사용 여부 자체가 "좋다/나쁘다"가 아니다.
- Proxy를 쓴다면 업그레이드 권한이 어떻게 통제되는지,
  쓰지 않는다면 버그 대응 계획이 무엇인지를 확인한다.

## 11) Module 1 권한 통제 메커니즘 파트 정리

ch1~ch4는 WEB3 서비스의 권한 통제를 구성하는 핵심 메커니즘 네 가지를 다뤘다.

| 메커니즘 | 해결하는 문제 | 핵심 위험 |
|---|---|---|
| Multisig | 단독 행동 방지 | 서명자 공모, 형식적 분산 |
| Timelock | 즉시 실행 방지 | 긴급 대응 지연, 사용자 무관심 |
| RBAC | 권한 범위 최소화 | 역할 설계 부실, Admin 집중 |
| Proxy/Upgrade | 로직 교체 가능 | 업그레이드 권한 남용, 스토리지 충돌 |

이 네 가지는 독립적으로 쓰이기도 하고 조합되어 쓰이기도 한다.
다음 강의부터는 자산이 실제로 어떻게 흐르는지, 서비스 유형별 구조로 넘어간다.

## 12) Associate 관점: 서비스를 볼 때 확인하는 것

- 이 서비스의 컨트랙트가 Proxy 패턴을 사용하는가? (etherscan 등에서 확인 가능)
- 업그레이드 권한이 누구에게 있는가? 단일 키인가, Multisig인가?
- 업그레이드에 Timelock이 적용되어 있는가? 지연 기간은 얼마인가?
- 최근 업그레이드 이력이 있는가? 어떤 변경이 있었는가?
- 업그레이드 권한이 DAO 거버넌스로 이전되었는가, 팀이 계속 보유하는가?
- Proxy를 사용하지 않는다면, 버그 발생 시 대응 계획이 있는가?

## 용어 보충 (Quick Reference)
- **Proxy 컨트랙트**: 사용자가 상호작용하는 고정 주소 컨트랙트. 스토리지를 보관하고 호출을 Implementation에 위임한다.
- **Implementation 컨트랙트**: 실제 비즈니스 로직이 담긴 컨트랙트. 교체(업그레이드) 가능하다.
- **delegatecall**: 호출받은 컨트랙트의 코드를 호출한 컨트랙트의 컨텍스트(스토리지)에서 실행하는 특수 호출 방식.
- **Transparent Proxy**: 업그레이드 함수가 Proxy에 있고, Admin과 사용자 호출을 분리하는 패턴.
- **UUPS**: 업그레이드 함수가 Implementation에 있는 경량 Proxy 패턴.
- **스토리지 충돌(Storage Collision)**: Proxy와 Implementation이 같은 스토리지 슬롯을 다른 용도로 사용해 데이터가 덮어써지는 문제.
- **초기화(Initialization)**: Proxy 패턴에서 constructor 대신 사용하는 최초 설정 함수. 중복 실행 방지 설계가 필요하다.

## 헷갈리는 개념 보충
- Proxy 패턴을 쓴다고 해서 서비스가 불안전한 것이 아니다. 업그레이드 권한이 어떻게 통제되는지가 핵심이다.
- Proxy 컨트랙트의 주소는 바뀌지 않는다. 바뀌는 것은 Proxy가 가리키는 Implementation 주소다.
- UUPS에서 새 Implementation에 업그레이드 함수를 빠뜨리면 영구적으로 업그레이드가 불가능해진다. 이것은 의도할 수도 있고, 실수일 수도 있다.
