# 19강. 리스크 평가 기준과 개선 방향 도출: 발생가능성×영향도로 심각도를 결정하라

## 1) 이 강의의 한 문장
**자금 흐름 리스크의 심각도는 발생가능성과 영향도의 곱으로 결정되며, 심각도에 따라 개선 방향의 우선순위를 정하고 구체적인 대응 방안을 제시해야 한다.**

## 2) 강의 목표 (수강생이 끝나고 할 수 있어야 하는 것)
이 강의를 마치면 수강생은:

1. 발생가능성과 영향도를 각각 3단계로 평가하는 기준을 적용할 수 있다.
2. 심각도 매트릭스를 사용하여 리스크 우선순위를 결정할 수 있다.
3. 심각도 등급에 따른 개선 방향 제시 형식을 사용할 수 있다.
4. 리스크 유형별 개선 방향을 구체적으로 작성할 수 있다.

---

## 3) 리스크 평가의 두 축

### 발생가능성 (Likelihood)

해당 리스크 시나리오가 실제로 발생할 가능성.

| 등급 | 명칭 | 기준 |
|---|---|---|
| H (높음) | High | 조건이 갖춰져 있고 이미 유사 사례 존재. 특별한 조건 없이도 발생 가능 |
| M (중간) | Medium | 특정 조건(시장 상황, 공격자 능력)이 갖춰져야 발생. 유사 사례 있음 |
| L (낮음) | Low | 발생하려면 매우 특수한 조건 필요. 이론적으로만 가능하거나 사례 없음 |

**발생가능성 평가 시 고려 사항:**
- 현재 서비스 구조에 이미 취약점이 있는가?
- 유사한 공격 사례가 실제로 발생한 적 있는가?
- 공격자가 이 리스크를 악용하려면 어떤 자원/능력이 필요한가?
- 서비스의 TVL이 높아 공격자의 동기가 충분한가?

### 영향도 (Impact)

리스크가 발생했을 때 서비스와 사용자에게 미치는 피해 규모.

| 등급 | 명칭 | 기준 |
|---|---|---|
| H (높음) | High | 대규모 자산 손실 (TVL의 상당 부분), 서비스 운영 불가, 회복 불가능한 신뢰 손상 |
| M (중간) | Medium | 일부 사용자 자산 손실 또는 일시적 서비스 중단. 회복 가능 |
| L (낮음) | Low | 최소한의 자산 영향, 일시적 불편, 자동 복구 가능 |

**영향도 평가 시 고려 사항:**
- 피해가 발생하면 몇 명의 사용자, 얼마의 자산이 영향받는가?
- 손실이 영구적인가, 일시적인가?
- 서비스 신뢰도와 생태계 전체에 미치는 영향은?
- 보험, 준비금 등 완충 장치가 있는가?

## 4) 심각도 매트릭스

발생가능성(H/M/L) × 영향도(H/M/L)로 9개 조합의 심각도를 결정한다.

```
                    영향도 (Impact)
                  L         M         H
발생  H    │  중간(3)  │  높음(4)  │  치명(5)  │
가능  M    │  낮음(2)  │  중간(3)  │  높음(4)  │
성    L    │  무시(1)  │  낮음(2)  │  중간(3)  │
```

| 심각도 | 등급명 | 대응 우선순위 |
|---|---|---|
| 5 | 치명 (Critical) | P0: 즉시 대응 (서비스 중단 검토) |
| 4 | 높음 (High) | P1: 단기 대응 (1~4주) |
| 3 | 중간 (Medium) | P2: 중기 대응 (1~3개월) |
| 2 | 낮음 (Low) | P3: 장기 대응 (3개월 이상) |
| 1 | 무시 (Negligible) | 모니터링만 유지 |

## 5) 리스크 평가 적용 예시

**대상 서비스: Lending Protocol (TVL: $50M)**

**리스크 목록 평가:**

| 리스크 ID | 리스크 유형 | 발생가능성 | 영향도 | 심각도 | 우선순위 |
|---|---|---|---|---|---|
| R-01 | Oracle 단일 소스 의존 (조작 가능성) | M | H | 4 (High) | P1 |
| R-02 | Admin 단일 EOA (키 탈취) | M | H | 4 (High) | P1 |
| R-03 | 출금 시 재진입 취약점 가능성 | L | H | 3 (Medium) | P2 |
| R-04 | 연쇄 청산 (담보 자산 변동성 高) | M | M | 3 (Medium) | P2 |
| R-05 | 수수료 흐름 단일 EOA 집중 | H | M | 4 (High) | P1 |
| R-06 | 유동성 고갈 (대형 LP 집중) | L | M | 2 (Low) | P3 |
| R-07 | 비영구적 손실 | M | L | 2 (Low) | P3 |

**발생가능성 평가 근거 예시:**

```
R-01 (Oracle 단일 소스): M
→ 현재 Chainlink 단일 Oracle 사용
→ Chainlink는 비교적 신뢰할 수 있으나 장애 사례 있음
→ 가격 조작 자체는 어렵지만 Chainlink 장애 가능성은 M

R-02 (Admin 단일 EOA): M
→ 단일 키 탈취는 특수한 공격 능력 필요 (피싱, 키로거 등)
→ 발생 자체는 M이지만, 발생 시 영향이 커서 H 영향도

R-05 (수수료 흐름 단일 EOA): H
→ 현재 수수료가 EOA 0xDDDD로 집중
→ 이미 구조가 취약 (특별한 공격 없이도 내부자/탈취로 실현 가능)
```

## 6) 개선 방향 도출 원칙

심각도에 따라 개선 방향의 강도와 형식이 달라진다.

### 심각도 5 (치명): 즉시 개선

**특징:**
- 서비스 운영 중단 또는 즉각적인 구조 변경 검토
- 단기 완화 조치 + 장기 구조 개선 동시 제시
- 사용자 공지 포함

**작성 형식:**
```
[단기 완화 조치] (즉시~1주)
- 서비스 해당 기능 일시 중단 또는 제한 검토
- 사용자에게 리스크 공지

[중기 구조 개선] (1~4주)
- 구체적인 메커니즘 적용 (Multisig 전환, Oracle 추가 등)

[장기 거버넌스 강화] (1~3개월)
- 변경 사항 감사 및 커뮤니티 공개
```

### 심각도 4 (높음): 단기 개선

**특징:**
- 4주 이내 구체적인 메커니즘 적용
- 임시 완화 조치 가능하면 즉시 적용

**작성 형식:**
```
[개선 방향]
1. 적용 메커니즘: [구체적 메커니즘 + 파라미터]
2. 타임라인: [단기 목표 시점]
3. 완화 조치: [구조 변경 전 임시 완화 가능 여부]
```

### 심각도 3 (중간): 중기 개선

**특징:**
- 1~3개월 내 구조 개선 계획 수립 및 실행
- 운영 절차 개선이 포함될 수 있음

### 심각도 2 (낮음): 장기 개선

**특징:**
- 3개월 이상 장기 로드맵에 포함
- 모니터링 강화로 임시 대응

### 심각도 1 (무시): 모니터링 유지

## 7) 리스크 유형별 개선 방향 가이드

### Oracle 리스크 개선 방향

```
현황: 단일 Chainlink Oracle 사용
심각도: 4 (High)

개선 방향:
1. 복수 Oracle 소스 추가
   - Chainlink + Uniswap TWAP + Band Protocol 중 2개 이상 조합
   - 편차 임계값 초과 시 거래 일시 중단 (서킷 브레이커)

2. TWAP 최소 기간 설정
   - 가격 조작 저항을 위해 최소 30분 TWAP 적용
   - 급격한 가격 변화 시 수동 개입 절차

3. 오라클 장애 감지 및 대응
   - 오라클 업데이트 지연 감지 알림 설정
   - 오라클 장애 시 관련 기능 자동 일시 중단
```

### 재진입 공격 개선 방향

```
현황: 출금 함수에서 외부 호출 후 상태 업데이트 가능성
심각도: 3 (Medium) - 단, 코드 확인 후 조정 가능

개선 방향:
1. Checks-Effects-Interactions 패턴 준수 코드 감사
   - 모든 함수에서 상태 업데이트 순서 검토
   - 외부 호출 전 상태 변경 완료 확인

2. ReentrancyGuard 적용
   - OpenZeppelin ReentrancyGuard의 nonReentrant modifier 사용
   - 외부 호출이 있는 모든 함수에 적용

3. Pull over Push 패턴 검토
   - 직접 전송 대신 사용자가 인출하는 방식으로 전환 검토
```

### 청산 리스크 개선 방향

```
현황: 연쇄 청산 가능성 / 청산 인센티브 불충분
심각도: 3 (Medium)

개선 방향:
1. LTV 보수적 설정
   - 담보 자산 변동성에 따른 LTV 차등 적용
   - 고변동성 자산 LTV 낮춤 (50% → 40%)

2. 청산 인센티브 최적화
   - 청산 보너스 적정 수준 유지 (5~10%)
   - 청산자 진입 장벽 낮춤 (가스비 최적화)

3. 청산 지연 대비 안전 버퍼
   - 건전성 비율(Health Factor) 임계값 높게 설정
   - 급락 상황 시뮬레이션 정기 실시
```

### Bridge 검증자 리스크 개선 방향

```
현황: 검증자 임계값이 낮거나 검증자 탈중앙화 미흡
심각도: 4~5 (High~Critical)

개선 방향:
1. 검증자 수 및 임계값 증가
   - 현행 5-of-9 → 7-of-13으로 상향
   - 지리적/법인적으로 독립된 검증자 선정

2. 검증자 스테이킹 요구
   - 검증자 슬래싱 메커니즘 도입 (부정 행위 시 스테이킹 삭감)

3. 출금 한도 설정
   - 24시간 출금 한도 설정 (대규모 즉시 탈취 방지)
   - 한도 초과 시 지연 + 추가 검증

4. 긴급 정지 체계
   - 이상 감지 시 즉시 정지 가능한 비상 권한 구성
```

## 8) 개선 방향 제시 표준 형식

실기에서 개선 방향을 제시할 때 사용하는 표준 형식:

```
[리스크 ID]: [리스크 유형]
발생가능성: H / M / L
영향도: H / M / L
심각도: [1~5 등급명]
우선순위: P0 / P1 / P2 / P3

[개선 방향]
1. [핵심 개선 내용]: [구체적 조치]
   - 세부 방법 또는 파라미터
2. [추가 강화]: [있다면]
3. [모니터링]: [지속적으로 확인해야 할 지표]

[완화 조치] (구조 개선 전 임시):
- [즉시 적용 가능한 완화 조치]

[개선 근거]:
- ch[번호]의 [메커니즘]이 이 취약점에 적합한 이유
```

**작성 예시:**

```
[R-01]: Oracle 단일 소스 의존
발생가능성: M
영향도: H
심각도: 4 (High)
우선순위: P1

[개선 방향]
1. 복수 Oracle 소스 도입:
   - Chainlink + Uniswap V3 TWAP (30분) 조합
   - 두 Oracle 편차 5% 이상 시 관련 기능 자동 일시 중단
2. Oracle 이상 감지 모니터링:
   - Chainlink 업데이트 heartbeat 감지 자동 알림
   - 이상 시 긴급 pause 절차 수립 (Admin Multisig)
3. 장기: 분산 오라클 네트워크 추가 검토

[완화 조치]:
- 즉시: Oracle 장애 감지 알림 설정 (Tenderly, OZ Defender)
- 단기: 긴급 pause 절차 문서화 및 팀 훈련

[개선 근거]:
- ch18의 Oracle 조작 리스크: 단일 소스 의존이 핵심 취약점
- 복수 Oracle + 편차 감지가 조작에 대한 표준 방어 패턴
```

## 9) 평가 결과 종합

리스크 평가를 완료하면 종합 요약을 작성한다.

**종합 리스크 프로파일 예시:**

```
서비스명: [Lending Protocol]
평가일: [날짜]
TVL: [$50M]

심각도 분포:
- 치명 (5): 0개
- 높음 (4): 3개 (R-01, R-02, R-05)
- 중간 (3): 2개 (R-03, R-04)
- 낮음 (2): 2개 (R-06, R-07)

최우선 개선 과제 (P1):
1. R-02: Admin 단일 EOA → 3-of-5 Multisig 전환
2. R-01: Oracle 복수 소스 도입
3. R-05: Fee Collector Multisig 전환

전반적 리스크 평가:
권한 구조 취약점(R-02)과 Oracle 의존성(R-01)이 결합될 경우
복합 리스크 발생 가능. 단기 내 P1 과제 완료가 필요.
```

## 용어 보충 (Quick Reference)
- **발생가능성 (Likelihood)**: 리스크 시나리오가 실제로 발생할 확률. H/M/L 3단계 평가.
- **영향도 (Impact)**: 리스크 발생 시 서비스와 사용자에게 미치는 피해 규모. H/M/L 3단계 평가.
- **심각도 (Severity)**: 발생가능성과 영향도를 조합하여 결정하는 리스크의 전체적 위험 수준. 1~5 등급.
- **서킷 브레이커 (Circuit Breaker)**: 이상 상황(가격 급변, Oracle 오류 등) 감지 시 자동으로 관련 기능을 중단하는 메커니즘.
- **완화 조치 (Mitigation)**: 구조적 개선 전에 리스크를 일시적으로 줄이기 위한 조치. 알림 설정, 운영 절차 강화 등.
- **리스크 프로파일 (Risk Profile)**: 서비스의 전체 리스크를 종합하여 한눈에 볼 수 있는 요약 문서.

## 헷갈리는 개념 보충
- 발생가능성 H가 반드시 심각도 H를 의미하지 않는다. 영향도가 L이면 심각도는 중간에 그칠 수 있다.
- 리스크가 복합적으로 발생할 때(교차 리스크)는 각 리스크의 심각도보다 높게 평가해야 할 수 있다. ch13의 교차 리스크 개념을 연결하여 판단한다.
- "개선 방향"은 완전한 해결책이 아닐 수 있다. 트레이드오프(속도 vs. 보안, 탈중앙화 vs. 효율성)를 인정하고 현실 가능한 방향을 제시하는 것이 좋은 답안이다.
