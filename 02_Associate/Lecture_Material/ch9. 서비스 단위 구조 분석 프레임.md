# 9강. 서비스 단위 구조 분석 프레임: 전체를 하나의 구조로 읽는 방법론

## 1) 이 강의의 한 문장
**ch1~ch8에서 패턴을 배웠다면, ch9부터는 그 패턴을 가지고 실제 서비스 전체를 분석하는 방법론을 다룬다. 서비스를 온보딩·자산흐름·권한·책임 3축으로 분해하고, Architecture Read Sheet로 구조화하는 것이 Associate 분석의 출발점이다.**

## 2) 강의 목표 (수강생이 끝나고 할 수 있어야 하는 것)
이 강의를 마치면 수강생은:

1. Basic의 S-A-A-R이 트랜잭션 단위 분석임을 재확인하고, 서비스 단위 분석과의 차이를 설명할 수 있다.
2. 서비스 분석 3축(온보딩 / 자산흐름 / 권한·책임)이 무엇을 보는지 말할 수 있다.
3. 서비스 분석의 진입점(어디서 시작하는가)을 알고 실제로 적용할 수 있다.
4. Architecture Read Sheet의 구조와 작성 목적을 말할 수 있다.
5. 분석 순서(구조 파악 → 흐름 추적 → 취약점 식별)를 이해하고 설명할 수 있다.

---

## 3) 전환점: 패턴 지식에서 분석 방법론으로

ch1~ch8은 **무엇을 알아야 하는가**에 대한 답이었다.

- 권한 통제 메커니즘: Multisig / Timelock / RBAC / Proxy
- 자산·자금 흐름 구조: AMM / Lending / Bridge / Custody

이것을 알면 특정 메커니즘이 서비스에 있을 때 그게 무엇인지 알아볼 수 있다.

하지만 **"이 서비스는 어떤 구조인가?"**라는 질문에 답하려면 다른 능력이 필요하다.

> 전체를 한눈에 읽어서 구조를 파악하는 능력.

실제 서비스 앞에 섰을 때:
- 어디서 시작해야 하는가?
- 무엇을 먼저 봐야 하는가?
- 어떤 순서로 분석하는가?
- 분석 결과를 어떻게 정리하는가?

ch9~ch13은 이 질문에 답하는 파트다.

## 4) Basic S-A-A-R과의 차이

Basic 12강에서 S-A-A-R 프레임을 배웠다.

```
S (Subject)  : 누가 행동을 시작했나?
A (Action)   : 무엇을 요청했나?
A (Asset)    : 어떤 자산이 움직이나?
R (Rule)     : 어떤 규칙(컨트랙트) 아래에서?
+ Constraints: 실행 조건은?
+ Result     : 상태 변화는?
```

이것은 **트랜잭션 1건**을 읽는 프레임이다.
"이 Approve 트랜잭션에서 무슨 일이 일어났는가?"를 분석한다.

Associate에서 분석 대상은 다르다.
**서비스 전체**다.

서비스는 수십~수백 개의 컨트랙트, 수많은 역할, 복잡한 자산 경로로 구성된다.
S-A-A-R 카드 하나로 전체를 읽을 수 없다.

| 구분 | Basic S-A-A-R | Associate 서비스 분석 |
|---|---|---|
| 분석 단위 | 트랜잭션 1건 | 서비스/프로젝트 전체 |
| 분석 시간 | 단일 시점 | 지속적 구조 |
| 질문 | "이 트랜잭션에서 무슨 일이 일어났나?" | "이 서비스는 어떻게 설계되어 있는가?" |
| 출력 | 구조 해석 카드 | Architecture Read Sheet |

## 5) 서비스 분석 3축

Associate 서비스 분석의 핵심은 **3축으로 분해하는 것**이다.

```
         서비스
        /   |   \
       /    |    \
  온보딩  자산흐름  권한·책임
```

### 축 1: 온보딩 구조
**"사용자가 서비스에 어떻게 진입하는가?"**

- 사용자가 서비스를 처음 쓰기 위해 어떤 단계를 거치는가?
- 어떤 권한(approve 등)을 어느 시점에 주는가?
- 온보딩 과정에서 어떤 자산이 이동하는가?

### 축 2: 자산흐름 구조
**"자산이 서비스 안에서 어떻게 움직이는가?"**

- 자산의 진입점(Entry Point)은 어디인가?
- 서비스 내부에서 자산이 어떤 경로로 이동하는가?
- 자산이 어떻게 변환되는가? (토큰 → LP 토큰 → 이자 토큰 등)
- 자산의 출구(Exit Point)는 어디인가?

### 축 3: 권한·책임 구조
**"누가 무엇을 통제하고, 누가 무엇에 책임지는가?"**

- 서비스 내 역할은 무엇이 있는가?
- 각 역할이 어떤 권한을 갖는가?
- 권한이 어떻게 통제되는가? (Multisig? Timelock? RBAC?)
- 사고 발생 시 책임 경계는 어디인가?

설계자 포인트:
- 3축은 독립적이지 않다. 온보딩 과정에서 권한이 설정되고, 그 권한이 자산 흐름을 통제한다.
- 3축을 각각 분석한 후 **교차점**을 보면 취약점이 드러난다.

## 6) 분석 진입점: 어디서 시작하는가

실제 서비스를 분석할 때 가장 막히는 것이 "어디서 시작하는가"다.

**진입점 1: 공식 문서와 백서**
- 서비스가 스스로 설명하는 구조를 먼저 읽는다.
- "어떤 서비스인가", "어떤 역할이 있는가", "자산은 어떻게 흐르는가"를 파악한다.
- 주의: 문서와 실제 컨트랙트가 다를 수 있다. 선언과 구현의 일치 여부를 확인해야 한다.

**진입점 2: 컨트랙트 주소와 Etherscan**
- 메인 컨트랙트 주소를 찾는다 (공식 문서, GitHub 등에서 확인).
- Etherscan에서 해당 컨트랙트가 Proxy인지 확인한다.
- Proxy라면 Implementation 주소를 찾아 실제 로직을 본다.
- 컨트랙트 Owner 또는 Admin 주소를 확인한다. 단일 키인가, Multisig인가?

**진입점 3: 실제 사용자 트랜잭션 추적**
- 실제 사용자 트랜잭션(예: 큰 입금, 큰 스왑)을 하나 골라서 S-A-A-R로 분석한다.
- 그 트랜잭션이 어떤 컨트랙트를 거쳤는지 추적한다.
- 컨트랙트 간 호출 경로가 서비스 구조의 실제 흐름을 보여준다.

**진입점 4: 감사 보고서(Audit Report)**
- 보안 감사 보고서는 서비스 구조를 이미 분석한 문서다.
- 발견된 취약점 목록이 이전 취약점 패턴을 보여준다.
- 취약점이 수정됐는지, 수정 방식이 적절한지 확인한다.

설계자 포인트:
- 진입점 중 어느 하나만으로는 전체 그림이 안 나온다. 여러 진입점을 교차 확인한다.
- 처음엔 공식 문서로 "서비스가 주장하는 구조"를 파악하고, 컨트랙트로 "실제 구조"를 검증한다.

## 7) 분석 순서

서비스 전체를 한꺼번에 보려고 하면 길을 잃는다.
순서가 있어야 한다.

```
1단계: 서비스 개요 파악
  → 어떤 종류의 서비스인가? (AMM / Lending / Bridge / Custody / 복합)
  → 주요 참여자는 누구인가?
  → 주요 컨트랙트는 무엇인가?

2단계: 3축 개별 분석
  → 온보딩: 사용자 진입 경로 파악
  → 자산흐름: 입금~출금 경로 추적
  → 권한·책임: 역할 목록 및 통제 방식 확인

3단계: 교차 분석
  → 3축이 서로 어떻게 연결되는가?
  → 어느 지점에서 권한과 자산흐름이 교차하는가?
  → 온보딩 과정의 권한이 자산흐름에 어떤 영향을 주는가?

4단계: 취약점 식별
  → 3축 분석에서 드러난 구조적 문제는 무엇인가?
  → ch1~ch8에서 배운 패턴과 비교해 미흡한 부분은 무엇인가?

5단계: Architecture Read Sheet 작성
  → 분석 결과를 정리된 형식으로 문서화
```

## 8) Architecture Read Sheet: 분석 결과를 구조화하는 도구

**Architecture Read Sheet**는 서비스 전체 구조 분석 결과를 하나의 문서로 정리하는 산출물이다.

Basic의 "구조 해석 카드"가 트랜잭션 1건의 결과를 담았다면,
Architecture Read Sheet는 **서비스 전체의 구조를 담는다**.

### 구성 요소

**섹션 1: 서비스 개요**
- 서비스 명칭 / 유형 / 주요 컨트랙트 주소
- 어떤 서비스인가 (AMM / Lending / Bridge / Custody / 복합)
- 분석 시점과 참조 자료

**섹션 2: 온보딩 구조**
- 사용자 진입 경로 (단계별 서술)
- 온보딩 과정에서 부여되는 권한 목록
- 온보딩 관련 취약점 메모

**섹션 3: 자산흐름 구조**
- 자산 진입점 (입금 경로)
- 서비스 내부 자산 변환 경로
- 자산 출구 (출금 경로)
- 자산흐름 다이어그램 (텍스트 또는 도식)
- 흐름 관련 취약점 메모

**섹션 4: 권한·책임 구조**
- 역할 목록 (역할명 / 권한 범위 / 보유 주체)
- 권한 통제 방식 (단일 키 / Multisig / Timelock / RBAC)
- 업그레이드 가능 여부 및 통제 방식
- 권한·책임 관련 취약점 메모

**섹션 5: 종합 취약점 메모**
- 3축 분석을 통합해 도출된 구조적 취약점 목록
- 각 취약점의 영향 범위 및 관련 섹션 참조

### Architecture Read Sheet 작성 원칙
- 서비스가 주장하는 것이 아니라 **실제 확인된 것**을 기록한다.
- 불확실한 것은 "확인 필요"로 표시한다. 추측으로 채우지 않는다.
- 취약점은 "문제가 있다"가 아니라 "무엇이, 왜 문제인가"로 기록한다.

## 9) 예시: 간단한 AMM 서비스 분석 시작

실제로 어떻게 3축 분석이 시작되는지 간단한 예시로 보자.

**분석 대상: 단순 AMM DEX (가상)**

**1단계: 서비스 개요**
- 서비스 유형: AMM DEX
- 주요 컨트랙트: Router, Factory, Pool (Pair)
- Proxy 구조: Factory는 Proxy, Pool은 불변 컨트랙트

**2단계: 3축 초기 분석**

*온보딩:*
- 사용자가 토큰을 스왑하려면: 지갑 연결 → 토큰 Approve → 스왑 실행
- Approve 범위: 무한 Approve가 기본값으로 설정되어 있는가? → 확인 필요

*자산흐름:*
- 스왑 경로: 사용자 지갑 → Router → Pool → 사용자 지갑
- LP 예치 경로: 사용자 지갑 → Pool → LP 토큰 발행 → 사용자 지갑
- 수수료 흐름: 스왑 수수료 → Pool 누적 → LP 인출 시 분배

*권한·책임:*
- Factory Owner: 누구인가? → Etherscan 확인
- Pool 수수료 변경 권한: Factory Owner가 갖는가?
- Router 업그레이드 가능 여부: Proxy인가? → 확인 필요

**3단계: 교차 분석에서 드러나는 것**
- 무한 Approve(온보딩) + Router 업그레이드 가능(권한) 조합이 위험
  → Router가 업그레이드되면 무한 Approve된 토큰을 모두 가져갈 수 있는 경로가 생긴다.

설계자 포인트:
- 각 축을 따로 보면 "무한 Approve는 일반적"이고 "업그레이드 가능은 일반적"이다.
- 조합해서 보면 치명적인 취약점이 드러난다.

## 10) Associate 관점: 분석할 때 반복되는 함정

**함정 1: 이름을 믿는다**
- "탈중앙화 거래소"라고 해서 Owner 키가 없는 게 아니다.
- "DAO 거버넌스"라고 해서 중앙화가 아닌 게 아니다.
- 이름이 아니라 컨트랙트 구조를 확인한다.

**함정 2: 하나의 축만 본다**
- 자산 흐름만 보면 권한 취약점을 놓친다.
- 권한 구조만 보면 온보딩 단계의 리스크를 놓친다.
- 3축을 모두 보고 교차점을 찾는다.

**함정 3: 문서와 구현을 동일시한다**
- 백서에 쓰인 구조와 실제 컨트랙트가 다를 수 있다.
- 항상 컨트랙트를 직접 확인하거나 감사 보고서로 교차 검증한다.

**함정 4: 취약점을 단독으로 본다**
- 단독으로 보면 별것 아닌 취약점이 조합되면 치명적이다.
- 각 취약점이 다른 취약점과 어떻게 연결되는지 항상 생각한다.

## 용어 보충 (Quick Reference)
- **Architecture Read Sheet**: 서비스 전체 구조를 3축(온보딩 / 자산흐름 / 권한·책임)으로 분석한 결과를 정리하는 산출물.
- **분석 3축**: 서비스 구조를 온보딩 / 자산흐름 / 권한·책임의 세 관점으로 분해하는 분석 프레임.
- **진입점 (Entry Point)**: 자산이 서비스에 처음 들어오는 지점. 또는 분석을 시작하는 출발 자료.
- **교차 분석**: 두 개 이상의 축을 동시에 보면서 개별 분석에서 드러나지 않던 취약점을 찾는 과정.

## 헷갈리는 개념 보충
- Architecture Read Sheet는 완성된 보고서가 아니라 분석 과정의 작업 문서다. 분석하면서 계속 채워나간다.
- 3축 분석의 순서는 고정되어 있지 않다. 서비스 유형에 따라 자산흐름부터 시작할 수도 있고, 권한 구조부터 볼 수도 있다. 중요한 것은 세 가지 모두를 다루는 것이다.
- "취약점"은 반드시 해킹 가능한 버그만을 의미하지 않는다. 운영 리스크, 중앙화 위험, 투명성 부재도 구조적 취약점이다.
