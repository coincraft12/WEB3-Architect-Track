# 17강. 자금 흐름 추적 방법론: 입금에서 출금까지 경로를 완전히 그리는 방법

## 1) 이 강의의 한 문장
**자금 흐름 추적은 입금→보관→출금의 전체 경로를 단계별로 분해하고, 외부 의존성과 숨겨진 경로까지 포함하여 완전한 흐름 다이어그램을 그리는 작업이다.**

## 2) 강의 목표 (수강생이 끝나고 할 수 있어야 하는 것)
이 강의를 마치면 수강생은:

1. 자금 흐름 추적의 3단계(입금/보관/출금)를 정의할 수 있다.
2. 각 단계에서 확인해야 할 항목을 나열할 수 있다.
3. 외부 의존성이 포함된 자금 흐름을 다이어그램으로 표현할 수 있다.
4. 서비스 유형별(AMM/Lending/Bridge/Custody) 흐름 추적의 핵심 차이를 말할 수 있다.
5. 완성된 자금 흐름 다이어그램을 Fund Flow Risk Sheet의 입력으로 연결할 수 있다.

## 4) 자금 흐름 추적의 3단계

### 단계 1: 입금 (Deposit / Inflow)

자금이 서비스 컨트랙트로 처음 들어오는 과정.

**확인 항목:**
```
1. 진입 함수: 어떤 함수를 호출하여 입금이 이루어지는가?
   예: deposit(address token, uint256 amount)

2. 수용 자산 유형: 어떤 자산이 입금 가능한가?
   예: ETH 네이티브 / ERC-20 토큰 / 특정 화이트리스트 토큰만

3. 입금 검증 절차: 입금 시 어떤 검증이 이루어지는가?
   예: 최소 금액 / 최대 금액 / 토큰 화이트리스트 / KYC 연동

4. 입금 후 상태 변화: 입금 즉시 어떤 일이 발생하는가?
   예: 잔액 업데이트 / 증표 토큰 발행 / 외부 프로토콜 즉시 전송

5. 입금 경로 수: 여러 경로로 입금이 가능한가?
   예: 직접 전송 + 브릿지 경유 + 스왑 후 입금
```

**입금 단계 다이어그램 표기:**
```
[사용자 지갑]
    │ USDC 1,000 (approve → deposit 호출)
    ▼
[Service Contract: deposit()]
    │ 검증: 화이트리스트 토큰 확인, 최대 한도 미초과
    ▼
[잔액 기록 업데이트]
    │ 증표 토큰 발행 (share 토큰 or 없음)
    ▼
[다음 단계로 →]
```

### 단계 2: 보관 (Storage / Custody)

입금된 자금이 어디에 어떤 형태로 보관되는가.

**확인 항목:**
```
1. 보관 위치: 자산이 실제로 어디에 있는가?
   - 서비스 자체 컨트랙트에 잠김
   - 외부 프로토콜에 재예치
   - 여러 곳에 분산 보관

2. 보관 형태 변환: 자산이 다른 형태로 바뀌는가?
   - 원본 토큰 그대로 보관
   - 증표 토큰(share/aToken/LP 토큰)으로 변환
   - 담보로 잠김

3. 외부 의존성: 외부 프로토콜에 의존하는 보관이 있는가?
   - 어떤 프로토콜에 얼마나 예치?
   - 해당 프로토콜의 감사 현황?
   - 외부 프로토콜이 해킹되면 손실 규모?

4. 보관 중 이자/수익 발생 구조:
   - 이자가 어디서 발생하는가?
   - 이자가 어떻게 분배되는가?
   - 수수료는 어디로 흐르는가?

5. 보관 기간 및 락업:
   - 언제까지 출금 불가인가?
   - 락업 조건은 무엇인가?
```

**보관 단계 다이어그램 표기:**
```
[Service Contract]
    ├─ 보관 분할:
    │    80% → Vault 컨트랙트 (직접 보관)
    │    20% → 외부 Yield 프로토콜 (수익 창출용)
    │
    ├─ 이자 발생:
    │    외부 프로토콜 이자 → Vault 잔액 증가
    │
    └─ 수수료 흐름:
         수익의 10% → Treasury (0xTREASURY)
```

### 단계 3: 출금 (Withdrawal / Outflow)

자금이 서비스에서 사용자에게 반환되는 과정.

**확인 항목:**
```
1. 출금 함수: 어떤 함수로 출금이 가능한가?
   예: withdraw(uint256 shares), redeem(uint256 amount)

2. 출금 조건: 출금에 어떤 조건이 있는가?
   - 락업 기간 종료 여부
   - 부채 상환 완료 여부
   - 최소 보유 기간 충족 여부

3. 출금 지연: 즉시 출금인가, 지연이 있는가?
   - 즉시 처리
   - 출금 큐 (요청 → 대기 → 실행)
   - 유동성 풀 상황에 따라 가변

4. 긴급 출금 경로: 비상 시 출금이 가능한가?
   - emergencyWithdraw() 존재 여부
   - pause 상태에서의 출금 가능 여부
   - 서킷 브레이커 적용 여부

5. 출금 제한: 일일 한도, 최대 출금액이 있는가?
```

**출금 단계 다이어그램 표기:**
```
[사용자]
    │ withdraw(1000 shares) 호출
    ▼
[Service Contract: withdraw()]
    │ 조건 확인: 락업 해제, 잔액 충분
    ▼
[Vault → 외부 프로토콜에서 자산 회수]
    │ shares 소각 → 원금 + 이자 계산
    ▼
[사용자 지갑] USDC 1,050 (원금 + 수익)
```

## 5) 외부 의존성 추적

단순한 내부 보관이 아닌 외부 프로토콜 연동이 있는 경우, 외부 의존성을 명시적으로 추적해야 한다.

**외부 의존성 추적 체크리스트:**
```
□ 어떤 외부 프로토콜을 사용하는가? (이름, 주소)
□ 얼마나 많은 자산이 외부에 있는가? (비율 또는 금액)
□ 해당 외부 프로토콜은 감사를 받았는가?
□ 외부 프로토콜이 실패하면 어떤 손실이 발생하는가?
□ 외부 의존성이 사용자에게 공개되어 있는가? (문서, UI)
□ 외부 의존성을 제거하거나 교체할 수 있는가? (누가, 얼마나 빨리)
```

**복잡한 외부 의존성 다이어그램 예시 (Yield Aggregator):**
```
[사용자]
    │ USDC 예치
    ▼
[Yield Aggregator Contract]
    ├─ 30% → Aave에 예치 (aUSDC 수령)
    ├─ 40% → Compound에 예치 (cUSDC 수령)
    └─ 30% → Curve Pool에 유동성 공급 (LP 토큰 수령)

    [수익 집계] ← aUSDC 이자 + cUSDC 이자 + Curve 수수료
    │
    ▼
[Vault 잔액 증가]
    │ 출금 요청 시 외부 프로토콜에서 순서대로 회수
    ▼
[사용자] 원금 + 수익
```

## 6) 서비스 유형별 자금 흐름 추적 핵심

ch5~ch8에서 학습한 서비스 유형별로 추적 방법이 달라진다.

### AMM 자금 흐름 추적

```
[유동성 공급자 (LP)]
    │ ETH + USDC (approve + addLiquidity)
    ▼
[AMM Pool 컨트랙트]
    │ LP 토큰 발행 (풀 지분 증표)
    ▼
[LP 보유]

[스왑 사용자]
    │ ETH (approve + swap)
    ▼
[Router → Pool]
    │ x*y=k 계산으로 USDC 수령
    │ 스왑 수수료 (0.3%) → Pool 잔액 증가 (LP에게 귀속)
    ▼
[스왑 사용자] USDC 수령

[유동성 회수]
LP 토큰 → burn → 원금 + 누적 수수료 (ETH + USDC)
```

**추적 핵심 포인트:**
- 수수료가 풀에 누적되는 방식 확인
- 비영구적 손실(Impermanent Loss) 발생 조건
- 유동성이 낮을 때 출금 가능 여부

### Lending 자금 흐름 추적

```
[공급자 (Supplier)]
    │ USDC 예치 → aUSDC/cUSDC 수령
    ▼
[Lending Pool]

[차입자 (Borrower)]
    │ ETH 담보 예치
    ▼
[Lending Pool] 담보 보관
    │ LTV 한도 내 USDC 대출
    ▼
[차입자] USDC 수령 + 이자 발생

[청산 흐름]
담보 가치 하락 → LTV 초과
    ▼
[Liquidator] 부채 대신 상환 → 담보 할인 수령
    ▼
[Pool 정상화]
```

**추적 핵심 포인트:**
- Oracle 가격이 청산 기준에 미치는 영향
- 청산 시 담보가 어디로 이동하는지
- 공급자 자산이 대출자에게 흐르는 경로

### Bridge 자금 흐름 추적

```
[원본 체인 (Ethereum)]
    │ ETH 잠금 (Lock)
    ▼
[Bridge 컨트랙트] ETH 보관
    │ 검증자 서명 수집
    ▼
[대상 체인 (Polygon)]
    │ wETH 발행 (Mint) → 사용자 수령

[역방향 (브릿지백)]
    │ wETH 소각 (Burn)
    ▼
[검증자 확인]
    ▼
[원본 체인] ETH 잠금 해제 (Unlock) → 사용자
```

**추적 핵심 포인트:**
- Lock된 자산이 보관되는 컨트랙트의 통제 구조
- 검증자 서명 경로와 임계값
- 비상 시 Lock된 자산의 반환 가능 여부

### Custody 자금 흐름 추적

```
[사용자]
    │ 입금 주소로 전송 (온체인)
    ▼
[Custody 서비스 Hot Wallet]
    │ 집금 (주기적)
    ▼
[Cold Storage] 대부분의 자산 보관
    │ 출금 요청 시
    ▼
[Hot Wallet 보충]
    │ 출금 처리
    ▼
[사용자 외부 주소]
```

**추적 핵심 포인트:**
- Hot/Cold 비율 (Hot에 너무 많으면 해킹 위험)
- 입금 주소 생성 구조 (중앙화 의존)
- 출금 승인 프로세스 (MPC/Multisig)

## 7) 자금 흐름 추적 다이어그램 작성 표준

### 표기 규칙
```
[참여자/컨트랙트]: 박스로 표시
    │ 이동하는 자산: 화살표 위에 자산 유형과 조건 표기
    ▼
[다음 참여자/컨트랙트]
    ├─ 분기점: 자산이 여러 경로로 나뉘는 경우
    └─ 다른 경로

※ 역방향: 출금 또는 반환 흐름을 역방향으로 표기
※ 외부 의존: [외부 프로토콜명] 표기 후 → 의존성 명시
```

### 다이어그램 완성 체크리스트
```
□ 모든 진입점이 포함되어 있는가?
□ 외부 프로토콜 의존성이 명시되어 있는가?
□ 수수료 흐름이 포함되어 있는가?
□ 출금 경로가 완전히 그려져 있는가?
□ 자산 변환(토큰 변환, 소각, 발행)이 표시되어 있는가?
□ 역방향(출금) 흐름이 포함되어 있는가?
□ 긴급 경로(emergencyWithdraw, pause)가 표시되어 있는가?
```

## 8) 자금 흐름 추적 결과를 리스크 시트에 연결하는 방법

추적 결과를 구간으로 분해하여 Fund Flow Risk Sheet의 "흐름 구간" 컬럼 입력값으로 사용한다.

**구간 분해 예시 (Lending 프로토콜):**
```
구간 1: 사용자 → Lending Pool (입금)
구간 2: Lending Pool → Oracle (가격 조회)
구간 3: Lending Pool → 차입자 (대출 실행)
구간 4: Liquidator → Lending Pool (청산)
구간 5: Lending Pool → 공급자 (이자 지급)
구간 6: 공급자 → Lending Pool (출금)
구간 7: Lending Pool → Treasury (수수료 이동)
```

각 구간에서 어떤 리스크가 발생하는지를 ch18에서 학습한다.

## 용어 보충 (Quick Reference)
- **자금 흐름 추적 (Fund Flow Tracing)**: 서비스에 입금된 자금이 보관되고 출금될 때까지의 전체 경로를 단계별로 분석하는 방법.
- **입금 구간 (Inflow Segment)**: 자금이 서비스 컨트랙트로 최초 진입하는 단계.
- **보관 구간 (Storage Segment)**: 자금이 컨트랙트 또는 외부 프로토콜에 보관되는 단계. 이자 발생, 외부 의존성 포함.
- **출금 구간 (Outflow Segment)**: 자금이 사용자에게 반환되는 단계. 조건, 지연, 긴급 경로 포함.
- **흐름 구간 (Flow Segment)**: 전체 자금 흐름을 리스크 평가를 위해 단계별로 분해한 단위. Fund Flow Risk Sheet의 기본 단위.
- **비영구적 손실 (Impermanent Loss)**: AMM에서 유동성을 공급할 때, 단순 보유 대비 손실이 발생하는 현상. 자산 가격 변동에 따른 풀 내 비율 변화로 발생.

## 헷갈리는 개념 보충
- ch11(자산흐름 경로 분석)과 ch17(자금 흐름 추적 방법론)은 같은 흐름을 보지만 목적이 다르다. ch11은 "서비스 구조를 이해"하기 위한 것이고, ch17은 "리스크를 평가하기 위해 구간별로 분해"하는 것이다.
- 외부 의존성은 항상 나쁜 것이 아니다. Chainlink Oracle, Aave, Compound 같은 검증된 외부 프로토콜의 의존성은 관리 가능한 리스크다. 감사 미실시, 익명 팀의 외부 프로토콜이 더 위험하다.
- 자금 흐름 다이어그램은 완벽할 필요가 없다. 중요한 것은 "모든 구간에서 자금이 어디에 있고 누가 통제하는지"를 빠짐없이 표현하는 것이다.
