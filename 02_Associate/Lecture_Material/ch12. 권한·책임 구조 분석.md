# 12강. 권한·책임 구조 분석: 누가 무엇을 할 수 있고, 그 결과를 누가 책임지는가

## 1) 이 강의의 한 문장
**권한·책임 구조 분석은 서비스 안에서 어떤 역할이 어떤 행동을 할 수 있는지를 매핑하고, 그 권한 행사의 결과가 누구에게 귀속되는지를 추적하는 작업이다.**

## 2) 강의 목표 (수강생이 끝나고 할 수 있어야 하는 것)
이 강의를 마치면 수강생은:

1. 권한 구조 분석의 3요소(역할 분류 / 권한 위임 경로 / 책임 경계)를 설명할 수 있다.
2. 서비스 안의 역할을 유형별로 분류할 수 있다.
3. 권한 위임 경로를 추적하는 방법을 말할 수 있다.
4. 책임 경계가 불명확한 구조를 식별할 수 있다.
5. 권한·책임 구조를 Architecture Read Sheet에 기록하는 방법을 알 수 있다.

---

## 3) 왜 권한·책임 구조를 분석해야 하는가

ch10(온보딩)과 ch11(자산흐름)을 통해 우리는 사용자가 서비스에 들어와 자산이 어떻게 움직이는지를 추적했다.

이제 핵심 질문이 남는다:
> **그 자산을 움직일 수 있는 권한은 누가 갖고 있는가?**

서비스의 자산흐름이 정상적으로 작동하는 것처럼 보여도, 권한 구조가 잘못 설계되어 있다면:
- Admin 한 명이 모든 자산을 인출할 수 있다.
- 업그레이드 권한자가 컨트랙트 로직을 바꿔 자산 이동 규칙을 변경할 수 있다.
- 특정 역할이 과도한 권한을 갖고 있지만 아무도 감시하지 않는다.

핵심 문장:
> **자산이 어디 있는지 알아도, 그것을 움직일 수 있는 권한 구조를 모르면 진짜 리스크를 본 것이 아니다.**

## 4) 권한·책임 구조 분석의 3요소

### 1. 역할 분류 (Role Classification)

서비스에 존재하는 역할을 체계적으로 식별한다.

**역할 유형:**

| 역할 유형 | 설명 | 예시 |
|---|---|---|
| 사용자 (User) | 서비스를 이용하는 일반 참여자 | 예치, 스왑, 대출 신청 |
| 운영자 (Operator) | 서비스 일상 운영을 담당하는 역할 | 파라미터 조정, 수수료 설정 |
| 관리자 (Admin) | 권한 부여·회수 및 핵심 기능 제어 | 역할 변경, 긴급 정지 |
| 업그레이드 권한자 (Upgrader) | 컨트랙트 로직을 변경할 수 있는 역할 | Proxy 구현 주소 교체 |
| 거버넌스 (Governance) | DAO 또는 토큰 홀더가 투표로 결정 | 파라미터 변경, 예산 집행 |
| 외부 계약 (External Contract) | 다른 프로토콜 또는 컨트랙트 | 오라클, Bridge, Yield 프로토콜 |

**역할 분류 시 확인 사항:**
- 역할이 명시적으로 정의되어 있는가? (RBAC 구조 여부)
- 한 주소가 여러 역할을 겸하는가?
- 역할 부여·회수 권한은 누가 갖는가?

### 2. 권한 위임 경로 (Authority Delegation Path)

권한이 어디서 시작되어 어디로 위임되는지를 추적한다.

**위임 경로 유형:**

**A. 직접 권한 (Direct Authority)**
```
배포자(Deployer)
    │ 컨트랙트 배포 시 Owner로 설정
    ▼
Admin 주소 (단일 EOA 또는 Multisig)
    │ onlyOwner 함수 직접 실행 가능
    ▼
핵심 기능 (pause, withdraw, upgrade)
```

**B. 역할 위임 (Role Delegation)**
```
Admin
    │ grantRole(OPERATOR_ROLE, 0x...)
    ▼
Operator 주소
    │ 제한된 함수만 실행 가능
    ▼
운영 기능 (파라미터 조정, 수수료 변경)
```

**C. 거버넌스 위임 (Governance Delegation)**
```
토큰 홀더
    │ 투표 제안 및 참여
    ▼
Governor 컨트랙트
    │ 투표 통과 → Timelock 큐 → 실행
    ▼
대상 컨트랙트의 파라미터 변경
```

**권한 위임 경로 추적 시 확인 사항:**
- 권한의 최상위 출발점(Root of Trust)은 어디인가?
- 위임 단계가 몇 단계인가? (단계가 적을수록 집중)
- 위임이 취소 가능한가? (역할 회수 가능 여부)

### 3. 책임 경계 (Responsibility Boundary)

권한 행사의 결과가 누구에게 귀속되는지를 명확히 한다.

**책임 경계 유형:**

**A. 명확한 책임 경계 (Clear Boundary)**
```
Timelock + Multisig 조합:
- 실행 전 공개 기간 → 사용자가 이의 제기 가능
- 다수 서명 요구 → 단일 실패점 제거
- 책임이 서명자 집단에 분산
```

**B. 불명확한 책임 경계 (Unclear Boundary)**
```
단일 Admin EOA:
- 누가 이 키를 관리하는가?
- 행동 결과를 누가 책임지는가?
- 키 분실 시 서비스가 마비되는가?
```

**C. 책임 공백 (Accountability Gap)**
```
외부 프로토콜에 재예치된 자산:
- A 서비스 → B 프로토콜 (자산 이동)
- B 프로토콜이 해킹되면 누가 책임지는가?
- 서비스 약관에 면책 조항이 있는가?
```

## 5) 역할 분류 실습

**분석 대상 구조 예시 (Lending 프로토콜):**

```
역할 목록:
1. Deployer: 0xAAAA (컨트랙트 초기 배포자, 현재 Admin 역할 이전 완료)
2. Admin: 0xBBBB (3-of-5 Multisig)
   - 권한: pause(), unpause(), setOracle(), grantRole()
3. Risk Manager: 0xCCCC (2-of-3 Multisig)
   - 권한: setLTV(), setLiquidationThreshold(), addAsset()
4. Fee Collector: 0xDDDD (단일 EOA)
   - 권한: collectFees() → Treasury로 이동
5. Liquidator: 조건 충족 시 누구나 (Public)
   - 권한: liquidate(address borrower)
6. Oracle: Chainlink 컨트랙트 (외부)
   - 역할: 가격 데이터 제공
```

**역할 분류 결과:**
```
[거버넌스 레이어]
  없음 (팀 직접 운영)

[Admin 레이어]
  Admin Multisig (3-of-5) → 핵심 권한 보유

[운영 레이어]
  Risk Manager (2-of-3) → 리스크 파라미터
  Fee Collector (단일 EOA) → 수수료 수취 ← 취약점 후보

[사용자 레이어]
  Liquidator (Public) → 청산 가능
  일반 사용자 → deposit, borrow, repay, withdraw

[외부 의존]
  Chainlink Oracle → 가격 데이터
```

## 6) 권한 위임 경로 추적 실습

**위임 경로 다이어그램:**
```
[Deployer: 0xAAAA]
    │ 배포 후 transferOwnership(Admin Multisig)
    ▼
[Admin Multisig: 0xBBBB (3-of-5)]
    ├─ grantRole(RISK_MANAGER_ROLE, 0xCCCC)
    ├─ grantRole(FEE_COLLECTOR_ROLE, 0xDDDD)
    ├─ setOracle(Chainlink 주소)
    ├─ pause() / unpause()
    └─ upgrade() ← Proxy 업그레이드 권한 포함

[Risk Manager: 0xCCCC (2-of-3)]
    ├─ setLTV(asset, newLTV)
    ├─ setLiquidationThreshold(asset, threshold)
    └─ addAsset(newAsset)

[Fee Collector: 0xDDDD (단일 EOA)]
    └─ collectFees() → Treasury

[Oracle: Chainlink]
    └─ 가격 데이터 제공 (자동, 외부 의존)
```

**추적 결과 분석:**
- 권한의 루트(Root of Trust): Admin Multisig (3-of-5) → 비교적 안전
- 단일 실패점 후보: Fee Collector (단일 EOA) → 수수료 흐름의 단일 통제점
- 업그레이드 권한: Admin Multisig가 보유 → 핵심 로직 변경 가능
- Oracle 외부 의존: Chainlink 장애/조작 시 가격 데이터 오류 가능

## 7) 책임 경계 식별

**책임이 명확한 경우:**
```
3-of-5 Multisig가 pause() 실행:
- 최소 3명의 서명자가 동의한 행동
- 서명자 목록이 공개되어 있다면 책임 귀속 가능
- Timelock이 추가되면 사전 공개 후 실행 → 사용자 대응 가능
```

**책임이 불명확한 경우:**
```
단일 EOA가 setOracle() 실행:
- 잘못된 Oracle 주소 설정 → 전체 프로토콜 가격 오류
- 이 주소의 실제 운영자가 누구인지 불명확
- 실수인지 악의인지 구분 불가
```

**책임 공백 사례:**
```
외부 Yield 프로토콜 재예치:
- A 서비스 → B 프로토콜 (Compound, Aave 등)
- B 프로토콜 해킹 시 손실은 A 서비스 사용자가 부담
- A 서비스 약관: "외부 프로토콜 리스크 책임 없음"
- 사용자는 이 구조를 알고 있었는가?
```

**책임 경계 식별 시 확인 사항:**
- 권한 행사 내역이 온체인에 기록되는가?
- 서명자/역할 보유자가 공개되어 있는가?
- 중요한 파라미터 변경 전 공지 절차가 있는가?
- 손실 발생 시 보상 메커니즘이 있는가? (보험, 준비금)

## 8) 권한·책임 구조에서 드러나는 취약점 유형

**유형 1: 역할 집중 (Role Concentration)**
```
한 주소(또는 팀)가 Admin + Upgrader + Fee Collector 역할을 동시에 보유.
단일 주소 탈취 시 서비스 전체를 통제 가능.
```

**유형 2: 역할 미분리 (Role Conflation)**
```
RBAC 구조가 형식적으로 존재하지만,
Admin이 모든 역할을 부여/회수할 수 있어
실질적으로 단일 통제와 동일.
```

**유형 3: 책임 없는 권한 (Unaccountable Authority)**
```
강력한 권한을 갖는 역할이지만:
- 다중 서명 없음
- Timelock 없음
- 공개 감시 없음
```

**유형 4: 숨겨진 업그레이드 권한 (Hidden Upgrade Authority)**
```
Proxy 구조에서 업그레이드 권한이 별도 계정에 있지 않고
Admin 권한과 동일하게 묶여 있어 사용자가 인지 못함.
```

**유형 5: 권한 회수 불가 (Irrevocable Grant)**
```
한번 부여된 역할이 기술적으로 회수 불가능한 구조.
또는 회수 권한 자체를 Admin이 잃어버린 경우.
```

**유형 6: 거버넌스 우회 (Governance Bypass)**
```
거버넌스 투표 없이 Admin이 직접 파라미터를 변경할 수 있는 경로 존재.
"비상 권한"이 실질적으로 일상적으로 사용되는 경우.
```

## 10) ch1~ch4와의 연결

이 강의에서 분석하는 권한·책임 구조는 앞서 배운 메커니즘들로 구성된다.

| 발견 내용 | 관련 ch | 의미 |
|---|---|---|
| Admin이 단일 EOA | ch1 Multisig | Multisig로 전환 필요 |
| 파라미터 변경에 지연 없음 | ch2 Timelock | Timelock 도입 필요 |
| 역할 구분 없이 Admin 일원화 | ch3 RBAC | 역할 분리 필요 |
| 업그레이드 권한이 Admin과 동일 | ch4 Proxy | 업그레이드 권한 분리 필요 |

**설계자 포인트:**
- 권한·책임 구조 분석은 ch1~ch4의 지식을 실제 서비스에 적용하는 단계다.
- 이론적으로 알고 있는 취약점이 실제 서비스에서 어떻게 나타나는지를 찾는 작업이다.

## 용어 보충 (Quick Reference)
- **역할 (Role)**: 서비스 내에서 특정 권한을 가진 주체의 분류. Admin, Operator, User 등.
- **권한 위임 (Delegation)**: 상위 역할이 하위 역할에게 특정 권한을 부여하는 행위.
- **루트 오브 트러스트 (Root of Trust)**: 권한 위임 체계의 최상위 출발점. 이 지점이 침해되면 전체 권한 체계가 위험해진다.
- **책임 경계 (Responsibility Boundary)**: 어떤 역할의 행동 결과가 누구에게 귀속되는지를 구분하는 기준.
- **책임 공백 (Accountability Gap)**: 권한 행사의 결과에 대해 어떤 주체도 명확히 책임지지 않는 구조.
- **거버넌스 우회 (Governance Bypass)**: 공식 거버넌스 절차를 거치지 않고 Admin이 직접 핵심 파라미터를 변경하는 경로.

## 헷갈리는 개념 보충
- 역할이 많다고 권한이 분산된 것이 아니다. 모든 역할을 Admin이 부여·회수할 수 있다면 실질적으로 단일 통제다.
- Multisig가 있다고 책임이 명확한 것은 아니다. 서명자 목록이 비공개라면 책임 귀속이 어렵다.
- "비상 권한"은 위기 시 필요하지만, 그 권한이 일상적으로 사용될 수 있는 구조라면 비상이 아닌 일상 리스크가 된다.
